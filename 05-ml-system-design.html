ML Production Pipelines: Offline Training and Online Serving
Key Insight: Decouple offline (scheduled, batch) and online (real-time, per-request) paths for independent latency/accuracy optimization; feedback loops enable continuous improvement.
Offline path:Hourly/daily: aggregate training data from logs, engineer features, train models, validate on holdout sets; prioritizes correctness over speed
Online path:Real-time: load trained models, compute features from live requests, return predictions within SLA (50-300ms)
Feedback loop:Online predictions generate logs (clicks, conversions) → inform next offline training run; enables continuous learning
Critical challenge:Training/serving skew—models trained on historic data encounter distribution shifts at serving time
Mitigation:Feature engineering (ensure computable at serving time), validation (distributions match), monitoring (drift detection), circuit breakers (fallback)
Data quality is foundational: bad labels (late feedback, label noise) propagate through retraining cycles, degrading all downstream models. Implement data quality checks early.
Feedback loops can amplify biases: popular items get more exposure → more training data → stronger model → more exposure (rich-get-richer problem). Monitor for feedback loop amplification.
Offline = preparation (take time to validate); Online = execution (must be fast); feedback loop = learning (continuously improve).

A/B Testing and Statistical Rigor in ML
Key Insight: A/B tests validate ML improvements with statistical rigor; sample size and test duration determine power to detect real effects while controlling false positives.
SE = √(p(1-p)(2/n)), where SE is standard error of difference in click rates; larger n → narrower CI → higher power
SE:Standard error of the difference in proportions (click rates) between control and treatment; measures variability of observed effect
p:Baseline click-through rate (CTR) for control group; expressed as proportion ∈ [0,1]; typically 0.01-0.10 for web
1-p:Complement probability (non-click rate); variance p(1-p) is maximized at p=0.5 and minimized at extremes
n:Sample size per group (control or treatment); total observations = 2n if balanced; larger n → smaller SE → higher statistical power
2/n factor:Factor of 2 accounts for two independent groups being compared; variance pools across both groups
√(...):Square root of variance; converts variance to standard deviation units; enables confidence interval construction
Type I error (α):False positive rate; α=0.05 means 5% chance of rejecting true null hypothesis; controls false discoveries
Type II error (β):False negative rate; power=1-β typically 80% industry standard; ability to detect real effects
MDE (Minimum Detectable Effect):Smallest effect size worth detecting given traffic and target power. Example: 1M users, α=0.05, power=80% → detect 0.2% CTR lift, not 0.01%
Test duration:1-2 weeks minimum; captures daily/weekly/seasonal cycles; shorter tests inflate Type I error due to temporal variations
Production gotchas:Multiple comparisons (inflate false positive rate—use Bonferroni); peeking (early stopping biases estimates); network effects (users not independent)
Practical vs statistical significance: 0.01% CTR lift may be statistically significant with millions of users but business impact negligible. Always compute MDE and business impact before running.
Why not run tests until significance? That's p-hacking. You must pre-register sample size and duration; stopping early when results look good biases estimates toward false positives.
α = false positive risk; β = false negative risk; power = ability to detect truth; larger n = smaller errors.

Feature Stores: Centralized Feature Management
Key Insight: Feature stores eliminate train/serve skew by centralizing feature definitions, ensuring offline training and online serving compute identical features.
Problem solved:Without feature store, engineers duplicate feature logic in batch (training) and real-time (serving), causing divergence → model degradation in production
Feature definition:Logical units (e.g., "user_7day_click_rate" = clicks in last 7 days / 7); single source of truth across pipelines
Offline computation:Batch (hourly, daily) on historical data; stores results in data warehouses/feature tables; correct and thorough
Online computation:Real-time services (compute incrementally or fetch pre-computed); must match offline logic exactly
Versioning:Critical: if feature definition changes (7 days → 14 days), offline and online must sync simultaneously to avoid skew
Architecture decisions:Which features compute offline (expensive joins) vs online (lightweight aggregations); late data handling; in-memory caching (Redis)
Train/serve skew example: Training uses "user_past_10_days" but online serves "user_past_7_days"; model trained on 10 days behavior performs poorly on 7-day patterns.
Tools: Feast (open-source, K8s-native), Tecton (enterprise, seamless offline/online), Hopsworks (distributed, MLOps-focused). Choose based on latency/scale requirements.
Feature store = single source of truth: one definition, two implementations (offline + online), no skew.

Online vs Batch Serving Trade-offs
Key Insight: Online (real-time) serves fresh predictions but expensive; batch (offline) is cheap but stale; hybrid systems batch expensive computations, online lightweight ranking.
Online (Real-time)
Features computed per-request + inference within SLA (50-500ms); fresh predictions with real-time signals; expensive infrastructure (GPUs, caches)
Batch (Offline)
Pre-compute all predictions, store offline (hourly/daily); serving is simple lookup; cheap compute (Spark, Presto) but stale (up to batch interval old), high storage
Latency trade-off:Online = feature computation + inference (slow); Batch = storage lookup (fast)
Freshness trade-off:Online = real-time user context (fresh); Batch = up to batch interval stale (irrelevant if preferences change daily)
Cost trade-off:Online = low-latency infrastructure (expensive); Batch = cheap offline compute but high storage (scale with user-item pairs)
Model complexity:Heavy models (BERT, XGBoost) exceed online budgets; light models (logistic regression) fit; Batch has no latency constraints
Hybrid approach:Batch expensive computations (CF candidates, long-sequence encoding); online lightweight ranking on batch results (fresh + fast)
Choose based on freshness requirements: hourly-changing user preferences → online; daily-changing → batch; stable preferences → batch with weekly refresh.
Hybrid sweet spot: batch pre-compute embeddings (expensive, static); online compute lightweight scores (fresh, fast). Example: batch BERT embeddings, online dot-product ranking.
Online = fresh but expensive; Batch = cheap but stale; Hybrid = batch for expensive, online for fresh.

Model Monitoring and Data Drift Detection
Key Insight: Data drift (distribution shift) causes silent model degradation; detect via KL-divergence or PSI; respond by retraining or model adjustment.
Data drift
Input feature distribution P(X) changes (e.g., user demographics shift); model assumes training distribution
Label drift (Concept drift)
Target distribution P(Y) changes (e.g., user preferences evolve); ground truth definition shifts
Prediction drift:Model outputs diverge from baseline; indirect signal of upstream changes (data or label drift)
KL-divergence:D_KL(P||Q) = Σₓ P(x) log(P(x)/Q(x)); asymmetric, measures divergence between distributions
PSI (Population Stability Index):Symmetric variant; <0.1 = little change; 0.1-0.25 = moderate (investigate); >0.25 = significant (action required)
Kolmogorov-Smirnov test:Statistical hypothesis test for distribution differences; p-value indicates significance
Response actions:Retrain on recent data (capture new distribution); data augmentation (diversify); model adjustment (re-weight drifted regions)
Critical gotcha: drift ≠ causation. Feature drift might correlate with accuracy loss, but root cause could be missing features or label quality decay. Investigate root cause before responding.
When to retrain vs accept: Seasonal variations (expected, monitor but don't panic); unexpected drifts (investigate and retrain). Set baselines for normal drift ranges.
Drift detection = sentinel; if drift exceeds threshold, investigate root cause before applying fixes.
