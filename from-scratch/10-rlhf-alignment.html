10.1 The Alignment Problem: Why Pretraining Isn't Enough

A pretrained language model predicts the next token. It has no concept of helpful, harmless, or honest. Watch it generate completions with no preference between safe and harmful outputs.

Callback to Ch 6: In the transformer chapter, we built a model that learns to predict the next token via cross-entropy loss on raw text. That objective learns patterns in language — including harmful ones. RLHF adds a second stage to steer the model toward human values.
Key Insight: Pretraining optimizes for next-token prediction, not for being helpful, harmless, or honest. A model that predicts well can still produce harmful, biased, or unhelpful outputs.
$$\mathcal{L}_{\text{pretrain}} = -\sum_t \log P_\theta(x_t | x_{<t}) \quad \text{(no safety signal here)}$$
The Problem: Next-token prediction treats all text equally. A factual answer and a harmful fabrication are optimized the same way if both appear in the training data.
The Solution: Add a second training phase that uses human feedback to teach the model WHICH completions are preferred.
The alignment problem is not about making models smarter — it's about making them do what we actually want. A perfectly capable model that follows harmful instructions is worse than a less capable but aligned one.
Pretraining = "predict everything." RLHF = "prefer the good stuff."

10.2 Preference Data: The Foundation

Human annotators compare pairs of completions and pick which is better. Click to provide preferences and build a dataset of (prompt, chosen, rejected) triples.

0 / 6 preferences collected
Key Insight: Preferences are easier than scores. Humans struggle to assign absolute quality numbers but reliably say "A is better than B."
Why Pairwise? Absolute ratings (1-5 stars) vary wildly between annotators. Relative comparisons ("A is better than B") are much more consistent and reliable.
Dataset Format: Each example is a triple: (prompt, chosen_completion, rejected_completion). Typical datasets have 50k-200k such triples.
Real preference datasets (like Anthropic's HH-RLHF or OpenAI's WebGPT) are collected from paid annotators who follow detailed guidelines. Annotator agreement is typically 70-80%, reflecting genuine ambiguity in what "better" means.
📎 Cross-reference Ch 1 (Backprop): The preference data will be used to compute gradients through a loss function (Bradley-Terry model). The autograd system from Ch 1 is what makes this training possible.

10.3 The Reward Model: Learning Human Preferences

Train a neural network to predict which completion a human would prefer. The Bradley-Terry model converts reward scores into preference probabilities.

Broken! With flipped labels, the reward model learns to prefer harmful content over safe content. The model assigns higher reward to rejected completions. This demonstrates why data quality and annotator guidelines are critical — garbage labels in, misaligned model out.
Key Insight: The reward model learns a scalar score r(prompt, completion) such that preferred completions get higher scores.
$$P(A \succ B) = \sigma(r(A) - r(B)) \quad \text{(Bradley-Terry model)}$$
$$\mathcal{L}_{\text{RM}} = -\log \sigma(r_\theta(x, y_w) - r_\theta(x, y_l))$$
Bradley-Terry: The probability that A is preferred over B depends only on the difference in their rewards, passed through a sigmoid. Simple and effective.
Architecture: Typically the same transformer as the policy, with the final token's hidden state projected to a single scalar. The reward model learns "what humans value."
The reward model is the "judge" that learns human taste. Train it well, and it can score thousands of completions per second — far faster than asking humans.

Common Mistakes: Reward Model Training

  • Mistake: Training the reward model on data from a different model distribution (e.g., preferences collected on GPT-3 outputs used to train a LLaMA reward model). Fix: Collect preferences on outputs from YOUR policy, not a different model. Distribution mismatch degrades reward model accuracy.
  • Mistake: Using too few preference pairs (<10k). Fix: Use at least 50k preference pairs. Reward model accuracy directly limits RLHF performance.
import torch
import torch.nn as nn

class RewardModel(nn.Module):
    def __init__(self, base_model):
        super().__init__()
        self.backbone = base_model  # e.g., GPT-2
        self.reward_head = nn.Linear(base_model.config.hidden_size, 1)

    def forward(self, input_ids, attention_mask):
        outputs = self.backbone(input_ids, attention_mask=attention_mask)
        # Use last token's hidden state as sequence representation
        last_hidden = outputs.last_hidden_state[:, -1, :]
        reward = self.reward_head(last_hidden)
        return reward.squeeze(-1)

# Bradley-Terry loss
def reward_loss(r_chosen, r_rejected):
    return -torch.log(torch.sigmoid(r_chosen - r_rejected)).mean()

The reward model typically shares architecture with the policy but adds a scalar head.

10.4 PPO: Optimizing the Policy

Use Proximal Policy Optimization to maximize the reward while staying close to the reference model via a KL penalty. Watch the training loop animate.

Callback to Ch 1: PPO uses gradient ascent (maximizing reward) rather than gradient descent (minimizing loss). The backpropagation machinery from Ch 1 works the same way — just negate the objective.
0.10
Broken! Without KL penalty, the model exploits the reward model. Reward skyrockets but the model produces degenerate outputs — repetitive phrases, sycophantic patterns, or other reward-hacking behaviors. The policy has drifted far from the base model.
Broken! With extreme KL penalty (β=10), the model barely changes from the base policy. Reward stays flat because the KL constraint prevents any meaningful updates. Alignment fails because the model cannot learn from the reward signal.
Key Insight: The KL penalty is the crucial balancing act. Too low leads to reward hacking. Too high leads to no learning. This is why RLHF is hard to tune.
$$\mathcal{J}(\theta) = \mathbb{E}_{x \sim D,\, y \sim \pi_\theta(\cdot|x)}\Big[r_\phi(x,y) - \beta \cdot \text{KL}\big(\pi_\theta(\cdot|x) \,\|\, \pi_{\text{ref}}(\cdot|x)\big)\Big]$$
The PPO Loop: 1) Generate completions with policy. 2) Score with reward model. 3) Compute advantage. 4) Update policy with clipped objective. 5) Apply KL penalty.
KL Divergence: Measures how far the current policy has drifted from the reference (pretrained) model. Acts as a regularizer to prevent the policy from collapsing to degenerate reward-hacking modes.
In practice, β is often adapted dynamically: if KL is too high, increase β; if too low, decrease it. This keeps KL in a target range (e.g., 6-10 nats).
🛠 Hyperparameter Guidance (PPO):
💡 β (KL penalty): Start at 0.01-0.1. Use adaptive KL controller with target KL of 6 nats.
💡 Learning rate: 1e-6 to 5e-6 (very small — the model is already good, we're fine-tuning).
💡 Epochs per batch: 1-4 PPO epochs per batch of rollouts.
💡 Batch size: 512-2048 completions per PPO step. Larger = more stable but slower.

Common Mistakes: PPO Training

  • Mistake: Using a learning rate that's too high (e.g., 1e-4). Fix: Use 1e-6 to 5e-6. The model is already pretrained and SFT'd; large LR destroys learned capabilities.
  • Mistake: Not freezing the reference model. Fix: The reference model (π_ref) must be frozen. If it updates alongside the policy, the KL penalty becomes meaningless.
import torch

def ppo_objective(log_probs, old_log_probs, advantages, clip_eps=0.2):
    """Clipped PPO surrogate objective."""
    ratio = torch.exp(log_probs - old_log_probs)
    clipped = torch.clamp(ratio, 1 - clip_eps, 1 + clip_eps)
    return torch.min(ratio * advantages, clipped * advantages).mean()

def rlhf_loss(rewards, kl_divergence, beta=0.1):
    """RLHF objective: maximize reward minus KL penalty."""
    return -(rewards - beta * kl_divergence).mean()

# KL divergence between policy and reference
def approx_kl(log_pi, log_pi_ref):
    return (log_pi - log_pi_ref).mean()

In practice, use libraries like trl (HuggingFace) for full PPO-RLHF training loops.

10.5 DPO: Skipping the Reward Model

Direct Preference Optimization derives a closed-form loss directly from preferences — no reward model needed. Compare the RLHF and DPO pipelines side by side.

Broken! Training DPO for too many epochs on a small preference dataset causes overfitting. The model memorizes the training preferences perfectly but loses general capabilities — it becomes great at the exact prompts in the dataset but worse at everything else. The implicit reward margin shoots up but generalization collapses.
Key Insight: DPO shows that the optimal RLHF policy has a closed-form relationship to the reward function. Instead of learning the reward then optimizing against it, you can do both in one step.
$$\mathcal{L}_{\text{DPO}} = -\log \sigma\!\Bigg(\beta \bigg[\log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)}\bigg]\Bigg)$$
RLHF (PPO) Pipeline
3 stages: 1) SFT on demonstrations. 2) Train reward model on preferences. 3) PPO to maximize reward. Requires 4 models in memory (policy, reference, reward model, value model).
DPO Pipeline
1 stage: Train directly on preferences using the DPO loss. Only needs 2 models in memory (policy, reference). Simpler, cheaper, more stable.
Why It Works: The optimal policy under KL-constrained reward maximization is \(\pi^*(y|x) \propto \pi_{\text{ref}}(y|x) \exp(r(x,y)/\beta)\). Inverting this gives the reward in terms of the policy, which can be substituted into the Bradley-Terry loss.
DPO is increasingly popular for smaller teams because it eliminates the complex PPO training loop. However, some research suggests PPO can achieve better results at scale when tuned properly.
📎 Cross-reference Ch 3 (MLPs & Optimization): The DPO loss is optimized with standard gradient descent, just like any supervised loss. The log-probability ratios are computed via forward passes through the transformer (Ch 6).
🛠 Hyperparameter Guidance (DPO):
💡 β: 0.1-0.5. Controls how much the policy can diverge from the reference. Lower β = more aggressive optimization.
💡 Learning rate: 1e-6 to 5e-7. Even smaller than PPO since DPO is more direct.
💡 Epochs: 1-3 over the preference dataset. Overfitting is a real risk with DPO.
💡 Batch size: 32-128 preference pairs. DPO is more memory-efficient than PPO. 
import torch
import torch.nn.functional as F

def dpo_loss(pi_logps_w, pi_logps_l, ref_logps_w, ref_logps_l, beta=0.1):
    """Direct Preference Optimization loss.

    Args:
        pi_logps_w: log P_policy(chosen | prompt)
        pi_logps_l: log P_policy(rejected | prompt)
        ref_logps_w: log P_ref(chosen | prompt)
        ref_logps_l: log P_ref(rejected | prompt)
        beta: temperature parameter
    """
    log_ratio_w = pi_logps_w - ref_logps_w
    log_ratio_l = pi_logps_l - ref_logps_l
    logits = beta * (log_ratio_w - log_ratio_l)
    return -F.logsigmoid(logits).mean()

DPO is just a single loss function — no RL loop needed. Use trl's DPOTrainer for a full implementation.

10.6 Reward Hacking & Failure Modes

When the model finds shortcuts to maximize reward without actually improving quality. Watch reward go up while actual output quality degrades.

Broken! After excessive training, the proxy reward (reward model score) continues climbing, but the "gold" reward (true human judgment) plummets. The model has learned to exploit patterns in the reward model rather than genuinely improving. This is Goodhart's Law: "When a measure becomes a target, it ceases to be a good measure."
Key Insight: Reward hacking is the central failure mode of RLHF. The model optimizes the reward model (a proxy), not actual human preferences (the ground truth).
Verbosity Hacking: Model learns that longer responses get higher reward scores, so it generates verbose, padded garbage.
Sycophancy: Model learns to agree with the user regardless of correctness. "Great question! You're absolutely right that 2+2=5."
Safety Washing: Model adds unnecessary disclaimers and hedging to everything: "As an AI, I must note that the color of the sky is generally considered blue, though views may vary..."
Goodhart's Law is the fundamental challenge of RLHF: you're optimizing a proxy (reward model) rather than the true objective (human satisfaction). The better your optimizer (PPO), the more aggressively it exploits the proxy's flaws.

Common Mistakes: Reward Hacking

  • Mistake: Using a single reward model without checking for gaming. Fix: Use multiple reward models (ensemble) or periodic human spot-checks. If reward goes up but outputs look worse, you're being hacked.
  • Mistake: Training RLHF for too many steps based on reward score alone. Fix: Early stopping based on human evaluation, not reward score. The optimal checkpoint is often early in training.

10.7 The Full RLHF Pipeline

See the complete journey from pretrained model to aligned model. Every stage builds on the previous one.

The Complete Pipeline:
Stage 1 — Pretrain: Train a large LM on internet text. Next-token prediction. Massive compute (thousands of GPUs, weeks). 📎 Ch 6
Stage 2 — SFT: Supervised fine-tuning on high-quality demonstrations (human-written ideal responses). Small dataset (~10k-100k examples), 1-3 epochs, LR 1e-5.
Stage 3 — RM Training: Train a reward model on human preference data (50k-200k pairs). Same architecture as policy or smaller.
Stage 4 — PPO/DPO: Optimize the SFT model against the reward signal. PPO: β=0.01-0.1, LR 1e-6, 1-2 epochs. DPO: β=0.1-0.5, LR 5e-7.
The SFT stage is often the most important! A good SFT model already follows instructions well. RLHF/DPO provides the "polish" — reducing harmful outputs, improving nuance, calibrating confidence.
📎 Cross-reference Ch 7 (Distillation): After RLHF, the aligned model can be distilled into a smaller model that retains alignment properties. This is how many "chat" versions of open-source models are created.
🛠 Hyperparameter Guidance (Full Pipeline):
💡 SFT: 1-3 epochs, LR 1e-5, cosine schedule. Don't overtrain — SFT overfitting hurts downstream RLHF.
💡 Reward Model: Same architecture as policy or smaller. Train for 1 epoch. Validation accuracy should be 65-75% (higher than chance but not perfect, reflecting human disagreement).
💡 PPO: β=0.01-0.1, LR 1e-6, 1-2 epochs over preference data. Monitor KL divergence.
💡 DPO: β=0.1-0.5, LR 1e-6 to 5e-7. 1-3 epochs. Watch for overfitting on small datasets.

Common Mistakes: Full Pipeline

  • Mistake: Skipping the SFT stage and going directly from pretraining to RLHF. Fix: Always include SFT. Without it, the base model doesn't follow instructions, making RLHF much harder and less effective.
  • Mistake: Overfitting during SFT (too many epochs on a small demonstration dataset). Fix: Use 1-3 epochs for SFT. SFT overfitting degrades downstream RLHF performance because the model memorizes demonstrations rather than learning general instruction-following.

10.8 Constitutional AI & Beyond

Move beyond human feedback: let the model critique and improve its own outputs, guided by a set of principles (a "constitution").

Key Approaches Beyond RLHF:
Constitutional AI (CAI)
The model critiques its own outputs against a set of principles, then revises. Self-generated preferences replace (some) human annotation. 📎 Relies on the transformer's in-context reasoning from Ch 6.
RLAIF (AI Feedback)
Use a strong AI model (e.g., GPT-4) to provide preference labels instead of humans. Much cheaper and faster, though introduces the labeler model's biases.
CAI Loop: Generate response → Critique against principles → Revise → Use revised version as the preferred completion. Repeat.
Iterative RLHF: Run RLHF, deploy, collect new preferences on the improved model, train again. Each iteration improves the data distribution and the model.
RLHF = humans judge. RLAIF = AI judges. CAI = the model judges itself against rules. All aim for the same goal: align the model with human values.
📎 Cross-reference Ch 6 (Transformer): Constitutional AI leverages the transformer's ability to follow instructions and reason in-context. The model must understand "critique this response for harmfulness" — which requires the deep language understanding built during pretraining.
