5.1 The Vanilla RNN Cell

An RNN processes sequences one token at a time, maintaining a hidden state that carries information from previous steps. At each timestep, the hidden state is updated using both the current input and the previous hidden state.

4
4
Key Insight: An RNN reuses the same weights at every timestep. The hidden state \(h_t\) is a compressed summary of everything the network has seen so far.
$$h_t = \tanh(W_{hh} \cdot h_{t-1} + W_{xh} \cdot x_t + b_h)$$
Weight Sharing: The same \(W_{hh}\) and \(W_{xh}\) are used at every step. This is what makes RNNs handle variable-length sequences.
Sequential Processing: Unlike attention (see Ch 04 where all positions are processed in parallel), RNNs must process tokens one at a time — a fundamental speed limitation.
When asked "RNN vs Transformer?": RNNs process sequentially (O(n) serial steps), while transformers process all positions in parallel. But RNNs use O(1) memory per step, whereas attention uses O(n²).
import torch
import torch.nn as nn

class VanillaRNNCell(nn.Module):
    def __init__(self, input_size, hidden_size):
        super().__init__()
        self.W_xh = nn.Linear(input_size, hidden_size)
        self.W_hh = nn.Linear(hidden_size, hidden_size, bias=False)

    def forward(self, x_t, h_prev):
        # x_t: (batch, input_size), h_prev: (batch, hidden_size)
        h_t = torch.tanh(self.W_xh(x_t) + self.W_hh(h_prev))
        return h_t

# Process a sequence
cell = VanillaRNNCell(input_size=32, hidden_size=64)
h = torch.zeros(1, 64)  # initial hidden state = zeros
for t in range(10):
    x_t = torch.randn(1, 32)
    h = cell(x_t, h)
print(f"Final hidden state: {h.shape}")  # (1, 64)

The same cell is called at every timestep with the previous hidden state. Weight sharing is automatic.

⚠️ Common Mistakes

  • Mistake: Forgetting to initialize hidden state to zeros before each new sequence. Fix: Always call h = torch.zeros(batch, hidden_size) at the start of each sequence. Leftover hidden states from previous sequences leak information and cause erratic training.
  • Mistake: Using vanilla RNN for long sequences (length > 30). Fix: Use LSTM or GRU instead. Vanilla RNNs cannot maintain gradients over long distances due to vanishing/exploding gradients (see Section 5.3 below).

5.2 Hidden State as Memory

The hidden state is the RNN's memory. It must compress the entire history of the sequence into a fixed-size vector. This bottleneck is exactly what Ch 04 (attention) replaces with explicit lookup over all past positions.

Key Insight: The hidden state is a lossy compression of the past. Recent tokens dominate; early tokens fade. This is the fundamental limitation that motivates LSTMs and attention.
$$h_t = f(h_{t-1}, x_t) \quad \Rightarrow \quad h_t = f(f(f(\ldots, x_1), x_2), \ldots, x_t)$$
Information Bottleneck: A hidden state of size 256 must encode ALL information from potentially thousands of tokens. Information is inevitably lost.
Recency Bias: Each application of tanh squashes values into [-1,1]. After many steps, the contribution of early tokens approaches zero.
Think of the hidden state as a game of telephone: the message (early tokens) gets more distorted with each retelling (timestep). LSTMs add a "written note" (cell state) that preserves the original message.

⚠️ Common Mistakes

  • Mistake: Confusing bidirectional hidden size — the output of a bidirectional RNN is 2 * hidden_size, not hidden_size. Fix: When feeding bidirectional output to a linear layer, use nn.Linear(2 * hidden_size, ...). This is one of the most common shape-mismatch bugs.
  • Mistake: Not packing variable-length sequences before feeding to the RNN. Fix: Use pack_padded_sequence and pad_packed_sequence. Without packing, the RNN wastes compute processing padding tokens and the hidden state gets corrupted by padding values.

5.3 Backpropagation Through Time & Vanishing Gradients

When we unroll the RNN to compute gradients, we multiply the same weight matrix repeatedly. This causes gradients to either vanish (most common) or explode. This is the same gradient flow problem from Ch 01 (backprop), but amplified by the recurrent structure.

10
off
0.9
Callback to Ch 01: Remember the chain rule from backpropagation? In an RNN, the chain rule produces a product of Jacobians across all timesteps. If each Jacobian has norm < 1, the product vanishes exponentially. If > 1, it explodes.
Key Insight: Gradients flow backward through time via repeated multiplication by \(\frac{\partial h_t}{\partial h_{t-1}}\). If the spectral radius of \(W_{hh}\) is less than 1, gradients vanish. If greater than 1, they explode.
$$\frac{\partial L}{\partial h_0} = \frac{\partial L}{\partial h_T} \prod_{t=1}^{T} \frac{\partial h_t}{\partial h_{t-1}} = \frac{\partial L}{\partial h_T} \prod_{t=1}^{T} \text{diag}(1 - h_t^2) \cdot W_{hh}$$
Vanishing (spectral radius < 1)
Gradient magnitude decreases exponentially with sequence length. The RNN cannot learn long-range dependencies — early tokens become invisible to the loss.
Exploding (spectral radius > 1)
Gradient magnitude increases exponentially. Training diverges with NaN losses. Gradient clipping is a band-aid; LSTMs are the real fix.
The classic interview question: "Why can't vanilla RNNs learn long-range dependencies?" Answer: repeated multiplication by W_hh causes vanishing gradients. The gradient from step T to step 0 scales as (spectral_radius)^T, which goes to 0 exponentially fast.
import torch
import torch.nn as nn

model = nn.RNN(input_size=32, hidden_size=64, num_layers=1, batch_first=True)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

# Training loop with gradient clipping
for batch in dataloader:
    optimizer.zero_grad()
    output, h_n = model(batch)
    loss = criterion(output, targets)
    loss.backward()

    # Clip gradients to prevent explosion
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

    optimizer.step()

Gradient clipping rescales the gradient vector if its norm exceeds max_norm. Essential for any RNN training.

⚠️ Common Mistakes

  • Mistake: Not using gradient clipping with RNNs (exploding gradients crash training with NaN losses). Fix: Always add clip_grad_norm_(model.parameters(), max_norm=1.0) after loss.backward(). Start with clip value 1.0; increase to 5.0 if training is too slow.
  • Mistake: Not detaching hidden state between sequences during truncated BPTT. The computation graph grows unbounded, consuming all GPU memory. Fix: Call h = h.detach() at the boundary of each truncated segment. This cuts the gradient chain while keeping the hidden state values.
🛠 Hyperparameter Guidance: Gradient clip norm: start at 1.0, increase to 5.0 if training is too slow. Truncated BPTT length: 35–256 tokens. Longer = better gradient signal but more memory. Learning rate: 1e-3 for Adam, 1.0 for SGD with clipping (old-school but works).

5.4 LSTM: Long Short-Term Memory

LSTMs add a separate cell state \(c_t\) that flows through time with minimal transformation. Three gates (forget, input, output) control what information to discard, store, and expose. The sigmoid gates act as soft switches — the same sigmoid activation from Ch 03 that squashes values to [0,1] to control information flow.

0.8
0.5
1
Key Insight: The cell state \(c_t\) is the LSTM's "highway" for gradients. It flows through time via element-wise multiply and add — no matrix multiplication by \(W_{hh}\), so gradients don't vanish.
$$f_t = \sigma(W_f \cdot [h_{t-1}, x_t] + b_f) \quad \text{(forget gate)}$$
$$i_t = \sigma(W_i \cdot [h_{t-1}, x_t] + b_i) \quad \text{(input gate)}$$
$$c_t = f_t \odot c_{t-1} + i_t \odot \tanh(W_c \cdot [h_{t-1}, x_t] + b_c)$$
$$h_t = o_t \odot \tanh(c_t) \quad \text{where } o_t = \sigma(W_o \cdot [h_{t-1}, x_t] + b_o)$$
Forget Gate \(f_t\): Controls what to discard from cell state. Initialized with large bias (~1.0) so the gate starts "open" (remembering by default).
Input Gate \(i_t\): Controls what new information to write to cell state.
Output Gate \(o_t\): Controls what to expose from cell state to the hidden state (and the outside world).
Key LSTM insight: the cell state update is c_t = f * c_{t-1} + i * new. The gradient through the cell state is just f_t (the forget gate), which can stay near 1 for many timesteps. This is why LSTMs solve the vanishing gradient problem.
LSTM gates are like a note-taking system: the forget gate erases old notes, the input gate writes new notes, and the output gate decides which notes to share. The note itself (cell state) persists across time.
import torch
import torch.nn as nn

# Single-layer LSTM
lstm = nn.LSTM(input_size=32, hidden_size=128, num_layers=2,
               batch_first=True, dropout=0.1)

# Input: (batch, seq_len, input_size)
x = torch.randn(16, 50, 32)
h0 = torch.zeros(2, 16, 128)  # (num_layers, batch, hidden)
c0 = torch.zeros(2, 16, 128)  # cell state also zeros

output, (h_n, c_n) = lstm(x, (h0, c0))
print(f"Output: {output.shape}")  # (16, 50, 128)
print(f"Final hidden: {h_n.shape}")  # (2, 16, 128)

LSTM returns both output (all timesteps) and final states. Note: h0 and c0 must both be initialized.

⚠️ Common Mistakes

  • Mistake: Not initializing the forget gate bias to a positive value (~1.0). With default zero bias, the forget gate starts near 0.5, meaning the LSTM forgets half the cell state at every step from the start. Fix: Manually set the forget gate bias to 1.0 after initialization: for name, p in lstm.named_parameters(): if 'bias' in name: p.data[hidden_size:2*hidden_size].fill_(1.0)
  • Mistake: Stacking 3+ LSTM layers without residual connections. Deep LSTMs still suffer from depth-wise vanishing gradients even though they handle temporal gradients well. Fix: For 3+ layers, add residual connections between layers or use 1-2 layers (sufficient for most tasks).
🛠 Hyperparameter Guidance: Hidden size: 128–512 for most tasks. Number of layers: 1–2 (3+ rarely helps without residual connections). Dropout between layers: 0.2–0.5. Forget gate bias: initialize to 1.0. Use Adam with lr=1e-3 and gradient clipping at 1.0.

5.5 GRU: Gated Recurrent Unit

A GRU simplifies the LSTM by merging the cell state and hidden state into one, using two gates instead of three. Fewer parameters, often similar performance. For most tasks where transformers (Ch 06) are overkill, GRUs are a strong default.

0.5
Key Insight: The GRU uses the update gate \(z_t\) to interpolate between the old hidden state and a candidate. When \(z_t \approx 1\), the old state is kept (like an LSTM forget gate at 1). When \(z_t \approx 0\), the new candidate replaces it.
$$z_t = \sigma(W_z \cdot [h_{t-1}, x_t]) \quad \text{(update gate)}$$
$$r_t = \sigma(W_r \cdot [h_{t-1}, x_t]) \quad \text{(reset gate)}$$
$$h_t = (1 - z_t) \odot h_{t-1} + z_t \odot \tanh(W \cdot [r_t \odot h_{t-1}, x_t])$$
LSTM (3 gates, 2 states)
Separate cell state + hidden state. 4x parameters vs vanilla RNN. More expressive for very long sequences.
GRU (2 gates, 1 state)
Merged cell/hidden state. 3x parameters vs vanilla RNN. Faster training, often competitive with LSTM.
When to choose GRU vs LSTM: GRU is the default for smaller datasets and faster experiments. LSTM is better when you need the extra expressiveness (e.g., very long sequences or complex gating patterns). In 2024+, consider transformers first for most tasks.
import torch
import torch.nn as nn

# GRU: simpler, fewer parameters
gru = nn.GRU(input_size=32, hidden_size=128, batch_first=True)
lstm = nn.LSTM(input_size=32, hidden_size=128, batch_first=True)

print(f"GRU params:  {sum(p.numel() for p in gru.parameters()):,}")
print(f"LSTM params: {sum(p.numel() for p in lstm.parameters()):,}")
# GRU: ~62K, LSTM: ~83K (LSTM has ~33% more params)

x = torch.randn(1, 20, 32)
gru_out, gru_h = gru(x)          # returns (output, h_n)
lstm_out, (lstm_h, lstm_c) = lstm(x)  # returns (output, (h_n, c_n))

GRU has 3 weight matrices (z, r, n) vs LSTM's 4 (f, i, c, o). Same API except LSTM also returns cell state.

⚠️ Common Mistakes

  • Mistake: Assuming GRU and LSTM are interchangeable in all cases. GRU can struggle with very long sequences (200+) where LSTM's separate cell state provides better gradient flow. Fix: Benchmark both on your task. If GRU works well at length 20 but fails at length 200, try LSTM or add attention on top.

5.6 Stacked & Bidirectional RNNs

Stacking multiple RNN layers increases model depth. Bidirectional RNNs process the sequence in both directions, giving each position context from the full sequence. This idea of "looking in both directions" also appears in Ch 06 (transformers) where attention naturally sees all positions.

2
Looking Ahead: While RNNs were the dominant sequence architecture for years, Ch 06 (transformers) shows how attention replaces RNNs for most tasks by processing all positions in parallel with O(1) serial depth. However, RNNs still matter: recent state-space models (Mamba, RWKV) revive the recurrent computation pattern with O(n) memory instead of attention's O(n²). The sequential processing flavor of RNNs also appears in Ch 08 (gradient-boosted trees), where each tree is built sequentially to correct the previous ensemble's errors.
Key Insight: Stacking adds representational depth (each layer builds higher-level features). Bidirectionality adds context width (each position sees the full sequence). Use both for tasks like NER, POS tagging, and machine translation encoders.
$$h_t^{(l)} = \text{RNN}(h_t^{(l-1)}, h_{t-1}^{(l)}) \quad \text{(layer } l \text{ sees output of layer } l-1 \text{)}$$
$$h_t^{\text{bidir}} = [\overrightarrow{h_t}; \overleftarrow{h_t}] \quad \text{(concatenate forward and backward hidden states)}$$
Bidirectional Output Size: The output dimension doubles: \(2 \times \text{hidden\_size}\). Downstream layers must account for this.
When Not to Use Bidir: Autoregressive tasks (language generation) cannot use bidirectional RNNs — you can't look at future tokens you haven't generated yet.
Stacking 1-2 layers is standard. 3+ layers rarely help without residual connections (same issue as deep MLPs in Ch 03). Bidirectional is free accuracy for classification/tagging but impossible for generation.
import torch
import torch.nn as nn

# 2-layer bidirectional LSTM
bilstm = nn.LSTM(input_size=32, hidden_size=128,
                 num_layers=2, batch_first=True,
                 bidirectional=True, dropout=0.2)

x = torch.randn(8, 30, 32)  # (batch, seq, features)
output, (h_n, c_n) = bilstm(x)

print(f"Output: {output.shape}")  # (8, 30, 256) -- 2*128!
print(f"h_n: {h_n.shape}")        # (4, 8, 128) -- 2*layers * batch * hidden

# Common pattern: classification head on final hidden states
# Concatenate last forward + first backward hidden states
h_forward = h_n[-2]   # last layer, forward direction
h_backward = h_n[-1]  # last layer, backward direction
final = torch.cat([h_forward, h_backward], dim=-1)  # (8, 256)
classifier = nn.Linear(256, 10)  # 256 = 2 * hidden_size!

Bidirectional output is 2*hidden_size. h_n has shape (num_layers*2, batch, hidden) for bidirectional models.

⚠️ Common Mistakes

  • Mistake: Confusing bidirectional hidden size — using nn.Linear(hidden_size, ...) after a bidirectional RNN instead of nn.Linear(2 * hidden_size, ...). Fix: Always check: if bidirectional=True, output features = 2 * hidden_size.
  • Mistake: Using bidirectional RNNs for autoregressive generation (text generation, speech synthesis). Fix: Use unidirectional for generation. Bidirectional is only valid when the full input sequence is available (e.g., classification, NER, encoding for seq2seq).
