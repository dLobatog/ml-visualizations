6.1 Tokenization: From Text to Numbers

Before a transformer sees any text, it must be broken into tokens — subword units with integer IDs. Type any text and watch BPE tokenization in action.

Callback to Ch 2: In makemore, our tokens were single characters (a-z). Real LLMs use subword tokens — same idea, bigger vocabulary (~50k tokens), much shorter sequences.

Key Insight: BPE (Byte-Pair Encoding) learns a vocabulary by repeatedly merging the most frequent character pairs.
$$\text{"unbelievably"} \rightarrow [\text{"un"}, \text{"believ"}, \text{"ably"}] \rightarrow [348, 17023, 2901]$$
Why Subwords? Character-level is too slow (sequences too long). Word-level can't handle new words. Subwords balance both: common words are single tokens, rare words split into known pieces.
Vocabulary Size: GPT-2: 50,257 tokens. LLaMA: 32,000. Larger vocab = shorter sequences but bigger embedding table.
Tokenization happens BEFORE the model sees anything. A "token" is the fundamental unit the transformer processes. Different tokenizers can produce different results for the same text.
Rare words split, common words stay whole. "the" = 1 token. "cryptocurrency" = "crypto" + "currency" = 2 tokens.
💡 Real-world: Tokenization directly affects cost and quality. GPT-4 charges per token (~$0.03/1K tokens). A better tokenizer that uses fewer tokens for the same text literally saves millions of dollars at scale. That's why every major lab builds custom tokenizers.
import tiktoken

enc = tiktoken.get_encoding('cl100k_base')  # GPT-4 tokenizer
tokens = enc.encode('unbelievably')
print(tokens)          # e.g. [348, 17023, 2901]
print([enc.decode([t]) for t in tokens])  # ['un','believ','ably']
print(len(enc.encode('the')))  # 1 token

6.2 Token Embeddings + Positional Encoding

Token IDs become dense vectors. Add position information so the model knows word order. Three rows: token embedding, positional encoding, and their sum.

32
5
Broken! Without positional encoding, "dog bites man" and "man bites dog" produce identical embeddings. The model cannot distinguish word order — it treats the input as a bag of words.
Key Insight: The input to a transformer is: token_embedding[token_id] + positional_encoding[position].
$$\mathbf{x}_i = \mathbf{E}[\text{token}_i] + \mathbf{PE}[i]$$
Token Embedding: A learnable lookup table of shape (vocab_size, d_model). Each token ID indexes one row.
Positional Encoding: Sinusoidal waves at different frequencies. \(PE(pos, 2i) = \sin(pos / 10000^{2i/d})\), \(PE(pos, 2i+1) = \cos(pos / 10000^{2i/d})\).
Why positions matter: Attention treats inputs as a SET — 'cat sat mat' and 'mat cat sat' produce identical attention scores without positions. The position encoding breaks this symmetry: each position gets a unique fingerprint added to its embedding.
Unlike RNNs which process sequentially (inherently knowing position), transformers process all tokens in parallel. Positional encoding is the explicit signal that tells the model WHERE each token is. As explored in Ch 04, attention is permutation-invariant without this signal — the order of tokens would be invisible.
Token embedding = WHAT the token is. Positional encoding = WHERE it is. Sum them = the model knows both.
📎 See Ch 4 where we show attention is permutation-invariant — positional encoding is the fix that gives the model awareness of token order.
💡 Real-world: RoPE (Rotary Position Embeddings) replaced sinusoidal encoding in LLaMA, Mistral, and most modern models. It enables context length extension — going from 4K to 128K tokens — which is why Claude and GPT-4 can read entire codebases.
import torch, math

def positional_encoding(seq_len, d_model):
    pe = torch.zeros(seq_len, d_model)
    pos = torch.arange(seq_len).unsqueeze(1).float()
    div = torch.exp(torch.arange(0, d_model, 2).float()
                    * -(math.log(10000.0) / d_model))
    pe[:, 0::2] = torch.sin(pos * div)
    pe[:, 1::2] = torch.cos(pos * div)
    return pe  # shape: (seq_len, d_model)

pe = positional_encoding(128, 512)

Add pe[:seq_len] to token embeddings before the first transformer block.

6.3 One Transformer Block — Full Data Flow

Trace data through a single transformer block. Every component from previous chapters assembled together: LayerNorm, Multi-Head Attention, FFN, Residuals.

Why normalize? Without LayerNorm, activations grow or shrink through each layer. After 12 layers, values can be 1000x too large or 0.001x too small, making gradients unstable. LayerNorm rescales each layer's output to mean=0, variance=1.

Modern Practice (GPT-2+)

Key Insight: One transformer block = Attention + FFN, each wrapped with LayerNorm and a residual connection.
$$\mathbf{h} = \mathbf{x} + \text{MHA}(\text{LN}(\mathbf{x})) \quad;\quad \mathbf{out} = \mathbf{h} + \text{FFN}(\text{LN}(\mathbf{h}))$$
LayerNorm [Ch 3.2]: Normalizes each token's embedding to zero mean, unit variance. Stabilizes training.
Multi-Head Attention [Ch 4]: Each token attends to all previous tokens (causal mask). Multiple heads capture different relationships.
FFN (GELU) [Ch 3]: Two linear layers with GELU activation. Expands to 4x dimension, then projects back. This is where "thinking" happens per-token.
Residual [Ch 3.3]: Add the input back to the output. Enables gradient flow and lets the block learn "corrections" rather than full transformations.
The pre-norm variant (LayerNorm BEFORE attention/FFN) is used in GPT-2+ and is more stable than post-norm. Know both variants.
import torch.nn as nn

class TransformerBlock(nn.Module):
    def __init__(self, d_model=512, nhead=8, d_ff=2048):
        super().__init__()
        self.ln1 = nn.LayerNorm(d_model)
        self.attn = nn.MultiheadAttention(d_model, nhead, batch_first=True)
        self.ln2 = nn.LayerNorm(d_model)
        self.ffn = nn.Sequential(
            nn.Linear(d_model, d_ff), nn.GELU(), nn.Linear(d_ff, d_model))
    def forward(self, x, mask=None):
        x = x + self.attn(self.ln1(x), self.ln1(x), self.ln1(x), attn_mask=mask)[0]
        x = x + self.ffn(self.ln2(x))
        return x

Pre-norm applies LayerNorm before attention/FFN, improving training stability.

Pre-Norm vs Post-Norm

Original Transformer (2017): Post-norm — LayerNorm after residual addition. Requires careful learning rate warmup to avoid training instability.

Modern practice (GPT-2+): Pre-norm — LayerNorm before attention/FFN. More stable training, no warmup needed. Gradients flow cleanly through the residual path (see Ch 03 on gradient flow through residual connections).

This visualization uses pre-norm (the modern default).

📎 Ch 3 covers BatchNorm and LayerNorm in depth. Transformers use LayerNorm because it doesn't depend on batch size — each token is normalized independently across its embedding dimensions.

Common Mistakes: Transformer Blocks

  • Mistake: Training with causal mask but evaluating without it (or vice versa). Fix: Always match train and eval masking. Decoder-only models need causal masks at all times.
  • Mistake: Not applying the final LayerNorm before the output head. Fix: Pre-norm architectures require an extra LayerNorm after the last block, before projecting to vocab. Without it, the output logits are unnormalized.
  • Mistake: A 12-layer transformer without residual connections performs like a ~2 layer network. Fix: Always include residuals. Without them, gradients vanish through deep stacks and only the last few layers learn effectively.
  • Mistake: Forgetting that attention is O(n²) — doubling context length quadruples memory usage. Fix: Use efficient attention implementations (FlashAttention) or limit context length. For very long sequences, consider sparse attention or sliding-window approaches.
🛠 Architecture Hyperparameters:
💡 d_model: 256 for toy models, 768 (GPT-2), 4096 (GPT-3). Powers of 2 preferred for hardware efficiency.
💡 n_layers: 6–12 for small, 24–96 for large. More layers = more capacity but diminishing returns per layer.
💡 n_heads: d_model/64 is typical. GPT-2: 768/64 = 12 heads. Each head gets 64 dims.
💡 FFN expansion: 4x d_model is standard (768 → 3072). Some use 8/3x for SwiGLU activation.
💡 Context length: 512–2048 typical. 8k–128k for modern long-context models. Memory grows quadratically with context.

6.4 Stacking Blocks: Depth & Representation Evolution

What changes as we stack more blocks? Embedding similarity matrices evolve from local patterns (early layers) to semantic structure (deep layers).

1
Layer 1: Local patterns
Layer 6: Dependencies
Layer 12: Semantics
Key Insight: Each layer refines the representation. Early layers capture syntax, deep layers capture semantics.
Layer 1-3: Tokens attend to nearby tokens. Similarity is high for adjacent words. Local syntactic patterns (noun-verb agreement) emerge.
Layer 4-8: Long-range dependencies form. Subject connects to its verb across clauses. Coreference resolution begins.
Layer 9-12: High-level semantic structure. Tokens with similar meaning cluster. Task-relevant features crystallize.
More layers = better performance, but with diminishing returns. GPT-2 Small (12 layers) vs GPT-2 Medium (24 layers): 2x layers gave ~15% perplexity improvement, not 100%.
Shallow = syntax. Deep = semantics. The residual stream accumulates information layer by layer.
import torch.nn as nn

    batch_first=True, norm_first=True  # pre-norm

encoder = nn.TransformerEncoder(encoder_layer, num_layers=12)

x = torch.randn(2, 128, 512)  # (batch, seq_len, d_model)

⚠️ Common Mistakes

  • Mistake: Forgetting that attention is O(n²) — doubling context quadruples memory. Fix: Use efficient attention (FlashAttention) or limit context length.
  • Mistake: Not applying final LayerNorm before the output projection head. Fix: Always include the final norm — it stabilizes the last layer's representations before projecting to vocabulary logits.

6.5 The Output Head: Logits → Probabilities → Token

The final hidden state projects back to vocabulary space. Softmax gives probabilities. Different sampling strategies yield different outputs.

1.0
5
Sampling Hyperparameters:
💡 Temperature: 0.0 = greedy (argmax, deterministic), 0.7 = balanced, 1.0 = full diversity, >1.5 = chaos.
💡 Top-p (nucleus): 0.9-0.95 is standard. Lower = more focused output. Top-p adapts the number of candidate tokens dynamically.
Key Insight: The output head converts hidden states to a probability distribution over the entire vocabulary.
$$\text{logits} = \mathbf{h}_{\text{last}} \cdot \mathbf{W}_{\text{vocab}}^T \quad;\quad P(\text{token}) = \text{softmax}(\text{logits} / T)$$
Greedy (T=0)
Always pick the highest probability token. Deterministic but repetitive.
Temperature
T<1 sharpens distribution (more confident). T>1 flattens (more random).
Top-k Sampling
Sample only from the k most likely tokens. Prevents sampling very unlikely tokens.
Top-p (Nucleus)
Sample from the smallest set of tokens whose cumulative probability exceeds p. Adaptive k.
Temperature, top-k, and top-p are applied AFTER the model computes logits. They don't change what the model "thinks" — they change how we select from its predictions. This is the same temperature concept from Ch 02 (language models), now applied at scale.
📎 Temperature sampling is the same concept as in Ch 2 (Makemore) — controlling randomness in next-token prediction. The math is identical, just applied to a much larger vocabulary.
import torch
import torch.nn.functional as F

def sample(logits, temperature=1.0, top_k=50):
    logits = logits / temperature
    # Keep only top-k logits
    top_vals, top_idx = logits.topk(top_k)
    probs = F.softmax(top_vals, dim=-1)
    token = top_idx[torch.multinomial(probs, 1)]
    return token

logits = torch.randn(50257)  # vocab-size logits
next_token = sample(logits, temperature=0.8, top_k=40)

Temperature < 1 sharpens; top-k restricts to most likely tokens.

Common Mistakes: Sampling & Generation

  • Mistake: Using greedy decoding when diversity is needed. Fix: Use temperature + top-p sampling. Greedy is deterministic and leads to repetitive text.
  • Mistake: Setting temperature=0 and expecting creative output. Fix: Temperature=0 is greedy (argmax). For creative tasks, use T=0.7-1.0 with top-p=0.9-0.95.
  • Mistake: Not using KV-cache during inference (recomputing attention for all previous tokens at every step). Fix: Always enable KV-cache for autoregressive generation. Without it, generating n tokens costs O(n^3) instead of O(n^2). See Ch 07 for more inference optimization techniques.
📎 Temperature sampling is the same concept as in Ch 2 (Makemore) — controlling randomness in next-token prediction. The only difference is what model produces the logits: a bigram table (Ch 2) vs a full transformer (here).

6.6 Autoregressive Generation: One Token at a Time

Key Insight: Autoregressive generation: each new token depends on ALL previous tokens. The KV-cache avoids recomputing past tokens.
$$\text{token}_{t+1} = \text{sample}(P(\cdot | \text{token}_1, \text{token}_2, \ldots, \text{token}_t))$$
KV-Cache: Store the Key and Value matrices from previous tokens. Only compute Q, K, V for the NEW token, then concatenate with cached K, V. Saves O(n) per step instead of O(n^2).
Why Autoregressive? The causal mask means each token can only see the past. Generation naturally follows: predict one, append, repeat.
Without KV-cache, generating n tokens costs O(n^3) total. With KV-cache, it costs O(n^2). This is why inference optimization matters so much for long sequences. See Ch 07 (distillation & compression) for further KV-cache optimizations like quantized KV, GQA, and MQA.
Generate = greedy loop: forward pass on full sequence, sample last position, append, repeat. KV-cache = "don't recompute what you already know."
import torch

def generate(model, prompt_ids, max_new=50, temperature=0.8):
    ids = prompt_ids.clone()
    for _ in range(max_new):
        logits = model(ids)[:, -1, :]  # last position
        logits = logits / temperature
        probs = torch.softmax(logits, dim=-1)
        next_id = torch.multinomial(probs, 1)
        ids = torch.cat([ids, next_id], dim=1)
        if next_id.item() == eos_token_id:
            break
    return ids

Production code uses KV-cache; here we recompute for clarity.

⚠️ Common Mistakes

  • Mistake: Not using KV-cache during autoregressive inference (recomputing all tokens every step). Fix: Cache K,V from previous steps, only compute new token. This reduces per-step cost from O(n²) to O(n).
  • Mistake: Training with causal mask but evaluating without it (data leakage). Fix: Always use the same masking at train and eval — decoder-only models need causal masks at all times.

6.7 Training: Cross-Entropy on Next-Token Prediction

Every position in a sequence provides a training signal. One sequence of length N = N-1 training examples for free.

Key Insight: The training objective is simple: predict the next token at every position. Cross-entropy measures how well the model's distribution matches the true next token.
$$\mathcal{L} = -\frac{1}{N-1}\sum_{t=1}^{N-1} \log P_\theta(\text{token}_{t+1} | \text{token}_{1:t})$$
Free Training Data: "The cat sat on the mat" (6 tokens) gives 5 training examples: predict "cat" from "The", predict "sat" from "The cat", etc.
Cross-Entropy: -log(probability assigned to correct token). Low = model is confident and correct. High = model is surprised.
This is why LLMs are so data-efficient: one document of N tokens provides N-1 supervised training examples. No labeling required — the text IS the label (shifted by one position).
Self-supervised learning: the input IS the label. Position t predicts position t+1. Slide a window across all text ever written = unlimited training data. The backpropagation that makes this work is the same engine you built in Ch 01 — gradients flow back through every component.
📎 All these gradients flow through the computation graph from Ch 1 (Backprop). The chain rule applies through every LayerNorm, attention head, and FFN — the same derivative mechanics, just many more parameters.
💡 Real-world: Next-token prediction on internet text is the recipe behind GPT, LLaMA, and Claude. The stunning fact: this simple objective — just predict the next word — produces models that can write code, do math, and reason. No explicit programming of these skills.
import torch
import torch.nn.functional as F

# input_ids: (batch, seq_len) of token IDs
logits = model(input_ids[:, :-1])  # predict from pos 0..N-2
targets = input_ids[:, 1:]          # targets are pos 1..N-1

loss = F.cross_entropy(
    logits.reshape(-1, vocab_size),
    targets.reshape(-1)
)
loss.backward()
optimizer.step()

Each sequence of N tokens yields N-1 training examples for free.

import torch.nn as nn

class MiniGPT(nn.Module):
    def __init__(self, vocab=50257, d=512, heads=8, layers=6):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab, d)
        self.pos_emb = nn.Embedding(1024, d)  # learned positions
        enc_layer = nn.TransformerEncoderLayer(
            d, heads, 4*d, batch_first=True, norm_first=True)
        self.transformer = nn.TransformerEncoder(enc_layer, layers)
        self.head = nn.Linear(d, vocab, bias=False)
    def forward(self, idx):
        B, T = idx.shape
        x = self.tok_emb(idx) + self.pos_emb(torch.arange(T, device=idx.device))
        x = self.transformer(x)
        return self.head(x)  # (B, T, vocab)
