2.1 The Dataset — Character Frequencies

See the raw data. A language model must learn: given what I have seen so far, what character comes next? That is the whole game.

Training Names (50 names)

The special token . marks start and end of each name.
Key Insight: A character-level language model learns to predict the next character given the previous ones.
$$P(c_{t+1} \mid c_1, c_2, \ldots, c_t) \quad \text{for each position } t$$
Special Token ".": Marks both start and end of a name. The model learns which characters tend to begin or end names.
Character Distribution: Not all characters are equally common. 'a', 'e', 'i' dominate in English names. The model must capture these frequencies.
Character-level models are the simplest language models. They are a great pedagogical tool because you can see every detail, but the same principles apply to word-level and subword (BPE) models.
Language modeling = next token prediction. That is it. GPT, BERT, everything reduces to predicting the next token from context.
import torch

words = ["emma", "olivia", "ava"]
chars = sorted(set(''.join(words)))
stoi = {c: i+1 for i, c in enumerate(chars)}
stoi['.'] = 0
itos = {i: c for c, i in stoi.items()}

# Encode a word as tensor
word = ".emma."
ix = torch.tensor([stoi[c] for c in word])
print(ix)  # tensor([0, 5, 9, 9, 1])

The '.' token marks start/end. stoi and itos map between characters and integer indices.

2.2 Bigram Frequency Matrix

Count character pairs. See the simplest possible language model: P(next | current) = count(pair) / count(current).

Key Insight: A bigram model estimates the probability of the next character using only the current character.
$$P(c_{i+1} \mid c_i) = \frac{\text{count}(c_i, c_{i+1})}{\sum_{c} \text{count}(c_i, c)}$$
27 x 27 Matrix: 26 letters + "." special token. Each row sums to 1 after normalization (a valid probability distribution).
Normalization: Raw counts become probabilities by dividing each count by the row total. This is maximum likelihood estimation.
Bigram models are bag-of-pairs: they have no memory beyond the immediately preceding character. This is a first-order Markov assumption.
Each row of the normalized matrix is a categorical distribution. Sampling from it gives the next character.
import torch

words = ["emma", "olivia", "ava"]
N = torch.zeros(27, 27, dtype=torch.int32)

for w in words:
    chs = ['.'] + list(w) + ['.']
    for c1, c2 in zip(chs, chs[1:]):
        ix1, ix2 = ord(c1) - 96 if c1 != '.' else 0, ord(c2) - 96 if c2 != '.' else 0
        N[ix1, ix2] += 1

P = (N + 1).float()  # +1 smoothing
P = P / P.sum(dim=1, keepdim=True)  # normalize rows

Each row of P is a valid probability distribution. The +1 smoothing avoids zero probabilities.

Key Insight: Sampling from a language model means repeatedly drawing from \(P(\text{next} \mid \text{current})\) until the stop token appears.
$$P_{\text{temp}}(c) = \frac{\exp(\log P(c) / \tau)}{\sum_{c'} \exp(\log P(c') / \tau)}$$
Temperature = 1.0: Sample from the learned distribution exactly as-is.
Low temperature: Sharpen the distribution — always pick the most likely next character. Repetitive but "safe".
High temperature: Flatten the distribution — sample more uniformly. More random and diverse, but less coherent.
Temperature is the most common sampling hyperparameter in LLMs. It appears in GPT, LLaMA, and every production system. Explain it as "softmax sharpness control."
import torch
import torch.nn.functional as F

logits = torch.tensor([2.0, 1.0, 0.5, -1.0, 0.1])

for temp in [0.5, 1.0, 2.0]:
    scaled = logits / temp
    probs = F.softmax(scaled, dim=0)
    sample = torch.multinomial(probs, num_samples=5, replacement=True)
    print(f"temp={temp}: probs={probs.tolist()}")
    print(f"         samples={sample.tolist()}")

Low temperature sharpens the distribution (greedy). High temperature flattens it (more random).

2.4 Measuring Quality: Negative Log-Likelihood

How do we know our model is any good? Step through a name and watch the NLL accumulate.

Key Insight: Negative log-likelihood measures how "surprised" the model is by the training data. Lower = better.
$$\text{NLL} = -\frac{1}{N}\sum_{i=1}^{N} \log P(c_{i+1} \mid c_i)$$
High probability: \(\log P \approx 0\), so \(-\log P \approx 0\). The model expected this transition. Low cost.
Low probability: \(\log P \ll 0\), so \(-\log P \gg 0\). The model was surprised. High cost.
"anna" (common patterns)
Common bigrams like "an", "nn", "na" have high probability. Low NLL.
"xzqw" (rare patterns)
Rare bigrams like "xz", "zq", "qw" have near-zero probability. Very high NLL.
NLL and cross-entropy are the same thing for language models. Perplexity = exp(NLL). These three metrics appear in every LLM paper.
NLL = surprise. A perfect model that knows the data perfectly has NLL = 0. A uniform-random model over 27 chars has NLL = log(27) = 3.30.
import torch
import torch.nn.functional as F

# Model outputs logits for 27 characters
logits = torch.randn(5, 27)  # 5 predictions, 27 classes
targets = torch.tensor([1, 14, 14, 1, 0])  # "anna."

loss = F.cross_entropy(logits, targets)
print(f"NLL (cross-entropy): {loss.item():.4f}")

# Manual calculation:
probs = F.softmax(logits, dim=1)
manual = -probs[range(5), targets].log().mean()
print(f"Manual NLL: {manual.item():.4f}")

F.cross_entropy = log_softmax + NLL combined. Numerically stable and faster than doing it manually.

2.5 Beyond Bigrams: N-gram Context Window

More context = better predictions. But the table grows exponentially. This tension motivates neural approaches.

1 (bigram)
Generated names will appear here...
Key Insight: N-gram models trade off expressiveness vs. data sparsity. More context means exponentially more parameters.
$$\text{Table size} = 27^n \quad \Rightarrow \quad n=1: 27, \quad n=2: 729, \quad n=3: 19{,}683, \quad n=5: 14{,}348{,}907$$
The Sparsity Problem: With n=5, most of the 14M entries are zero because we never observed that specific 5-character context in our tiny dataset.
The Key Tension: Quality improves with more context, but counting cannot generalize to unseen contexts. We need a neural network.
This is the exact motivation for neural language models. Bengio et al. (2003) introduced neural LMs precisely to solve this curse-of-dimensionality problem in n-gram models.
Counting can memorize but cannot generalize. Neural networks can generalize to unseen contexts via learned representations.
import torch

word = ".emma."
context_len = 3
stoi = {'.': 0, 'a': 1, 'e': 5, 'm': 13}

X, Y = [], []
for i in range(len(word) - 1):
    start = max(0, i - context_len + 1)
    ctx = [stoi.get(word[j], 0) for j in range(start, i + 1)]
    ctx = [0] * (context_len - len(ctx)) + ctx  # pad left
    X.append(ctx)
    Y.append(stoi.get(word[i + 1], 0))

X, Y = torch.tensor(X), torch.tensor(Y)
print(f"X shape: {X.shape}, Y shape: {Y.shape}")

Context windows slide over the text. Padding with '.' handles the start of sequences.

Key Insight: Embeddings map discrete tokens into continuous vector space where similar tokens are nearby.
$$\text{one-hot: } \mathbf{e}_a = [1,0,0,\ldots] \in \mathbb{R}^{27} \quad\Rightarrow\quad \text{embedding: } \mathbf{e}_a = [0.3, -0.7] \in \mathbb{R}^d$$
Why Embeddings? One-hot vectors are orthogonal — every character is equally "different" from every other. Embeddings learn that 'a' and 'e' are more similar than 'a' and 'z'.
Emergent Structure: Without being told, the network learns to cluster vowels together and consonants together, because they appear in similar contexts.
Word2Vec, GloVe, and the embedding layers in transformers all work on this same principle: tokens that appear in similar contexts get similar embeddings. This is the distributional hypothesis.
import torch
import torch.nn as nn

emb = nn.Embedding(27, 2)  # 27 chars, 2D embeddings

# Lookup: integer index -> dense vector
idx = torch.tensor([1, 5, 13])  # 'a', 'e', 'm'
vectors = emb(idx)
print(vectors)  # shape: (3, 2)

# Equivalent to:
one_hot = torch.nn.functional.one_hot(idx, 27).float()
manual = one_hot @ emb.weight
print(torch.allclose(vectors, manual))  # True

nn.Embedding is just a lookup table. It is equivalent to one-hot times weight matrix, but much faster.

2.7 MLP Language Model — The Full Picture

Wire it all together. Context embeddings → concatenate → hidden layer (ReLU) → softmax → next character prediction.

Chapter 2 Payoff: You have built a language model from counting bigrams to a neural network that understands character patterns. But characters are limiting — to model real language, we need to handle sequences of varying length and learn long-range dependencies. Next: the attention mechanism.
Key Insight: The MLP language model generalizes beyond counting by learning continuous representations.
$$\hat{y} = \text{softmax}(W_2 \cdot \text{ReLU}(W_1 \cdot [\mathbf{e}_{c_1}; \mathbf{e}_{c_2}; \ldots; \mathbf{e}_{c_n}] + b_1) + b_2)$$
Embedding Lookup: Each input character is mapped to its embedding vector. These are concatenated into a single input vector.
Hidden Layer: A linear transform + ReLU nonlinearity allows the model to learn complex patterns over the input characters.
Softmax Output: Produces a probability distribution over 27 possible next characters. Trained with cross-entropy (NLL) loss.
Bigram Model
729 parameters (27x27 table). Only 1 char of context. Cannot generalize to unseen bigrams. NLL ~ 2.45
MLP Model
~3500 parameters. 3 chars of context. Generalizes via embeddings. NLL ~ 2.10
Bengio et al. (2003) "A Neural Probabilistic Language Model" is exactly this architecture. It introduced the idea of learned word embeddings and is the foundation for all modern LLMs.
import torch
import torch.nn as nn
import torch.nn.functional as F

class CharLM(nn.Module):
    def __init__(self, vocab=27, emb_dim=10, ctx=3, hidden=200):
        super().__init__()
        self.emb = nn.Embedding(vocab, emb_dim)
        self.mlp = nn.Sequential(
            nn.Linear(ctx * emb_dim, hidden), nn.ReLU(),
            nn.Linear(hidden, vocab))
    def forward(self, x):
        e = self.emb(x).view(x.size(0), -1)  # concat embeddings
        return self.mlp(e)

model = CharLM()
x = torch.randint(0, 27, (32, 3))  # batch of 32, context 3
logits = model(x)  # (32, 27)

This is Bengio et al. 2003 in PyTorch. Embeddings are concatenated, not summed.
