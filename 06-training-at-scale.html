Data Parallelism: Each GPU has a full copy of the model. Data is split across GPUs, and gradients are synchronized via AllReduce. Best for models that fit in GPU memory.

Model Parallelism: Model layers are split across GPUs. Each GPU processes all data. Requires significant communication between layers. Use when model is too large for single GPU.

Pipeline Parallelism: Combines both - different layers on different GPUs, multiple batches in flight for better utilization.
Distributed Training: Data vs Model Parallelism
Key Insight: Data parallelism (model replicas, per-device gradients synced via AllReduce) dominates; model parallelism splits model across devices but has \(O(N)\) communication overhead.
Data Parallelism
Each GPU holds identical model, processes disjoint data, computes gradients locally; sync via AllReduce (\(O(\log N)\)); communication-efficient
Model Parallelism
Different layers on different GPUs; forward/backward require inter-GPU communication (\(O(N)\) per layer); slower unless model huge
Pipeline parallelism:Stages (layer groups) on different GPUs, multiple micro-batches in flight; maximizes utilization but complex scheduling (Megatron, GShard)
AllReduce bottleneck:Communication bandwidth dominates beyond 32-64 GPUs per node; network or PCIe saturation → poor scaling
When to use model parallelism:Only when model exceeds single GPU memory; trade compute-per-device for communication cost (rarely worth it)
Scaling efficiency:N-GPU training should be ~N times faster; if slower, communication or I/O bottlenecked (not compute-bound)
When to Use
Data Parallelism
Default choice when the model fits in a single GPU memory (e.g., ResNet-50 at 100MB, BERT-base at 400MB). Scale to 8-64 GPUs with near-linear speedup. Used by most production training jobs at Google, Meta, etc.
Model Parallelism (Tensor/Pipeline)
Use only when a single model exceeds GPU memory (e.g., GPT-3 175B at ~350GB, Llama-70B at ~140GB). Split layers or tensors across GPUs. Higher communication overhead; requires NVLink or InfiniBand for acceptable throughput.
Hybrid (3D Parallelism)
State-of-the-art for frontier models: combine data parallelism (across nodes), tensor parallelism (within node), and pipeline parallelism (across stages). Used by Megatron-LM, DeepSpeed for 100B+ parameter training runs.
Communication vs computation ratio: If model has X FLOPs and communication transfers Y bytes, ratio = Y/X. High ratio = communication-bound; low ratio = compute-bound.
Reduce synchronization: Gradient accumulation (accumulate over k batches before sync), communication-computation overlap (compute while communicating), reduce precision (FP16 transfers 2x less data).
👉 Try this: Set k=64 and toggle data parallelism ON with 4 GPUs. Observe all-reduce overhead. Increase k to 512 and notice per-GPU gradient communication cost. Then try model parallelism with k=2048 and watch different GPUs become communication bottlenecks.
Data parallelism = many workers, one model copy each; Model parallelism = one worker per model shard (harder to coordinate).

Learning Rate Schedules and Training Dynamics
Key Insight: Warmup (ramp up LR early) + decay (reduce LR late) helps escape sharp minima and converge to flat minima with better generalization.
$$LR(t) = LR_{\max} \cdot 0.5 \cdot \left(1 + \cos\left(\frac{\pi \cdot t}{T}\right)\right)$$cosine annealing decays smoothly from max to near-zero over \(T\) epochs
\(LR(t)\):Learning rate at epoch \(t\); dynamically adjusted during training; controls optimizer step magnitude
\(LR_{\max}\):Maximum learning rate (peak value); typically \(10^{-3}\) to \(10^{-4}\) for LLMs; set via learning rate finder or grid search
\(0.5\):Scale factor; ensures LR oscillates between \(LR_{\max}\) and \(\approx 0\) (not negative); formula: \(LR(0)=LR_{\max}\), \(LR(T) \approx 0\)
\(t\):Current epoch number (\(0 \leq t \leq T\)); training progresses from \(t=0\) to \(t=T\)
\(T\):Total number of epochs (training duration); annealing period; larger \(T\) = longer slow decay phase
\(\pi \cdot t / T\):Phase angle in radians; as \(t\) goes \(0 \rightarrow T\), angle goes \(0 \rightarrow \pi\); controls cosine curve shape
\(\cos(\pi \cdot t / T)\):Cosine function; starts at \(\cos(0)=1\) and decays to \(\cos(\pi)=-1\); smooth monotonic decay
\(1 + \cos(\pi \cdot t / T)\):Shifts cosine to range \([0, 2]\); ensures final LR approaches 0 (not negative)
Constant LR fails:Too large → gradients explode → diverge; too small → slow convergence; schedule adapts dynamically
Warmup:Start from very low LR, ramp up linearly over initial epochs; stabilizes gradient flow when weights random and loss unstable; critical for Transformers
Decay (Annealing):Reduce LR as training progresses for fine-grained updates near convergence; cosine annealing smooth (avoid sudden disruptions)
LR finder:Train for epoch, exponentially increase LR (\(10^{-5}\) to \(10\)), plot loss vs LR, choose LR in steep descent region (not minimum)
Sharp vs flat minima:Sharp (high curvature) = overfits, generalizes poorly; flat (low curvature) = robust. Schedules help escape sharp minima.
When to Use
Cosine Annealing
Default for most Transformer and vision training (BERT, GPT, ViT). Smooth decay avoids sudden LR drops that destabilize training. Use with warmup for LLM pre-training runs of 100K+ steps.
Step Decay
Classic choice for CNNs (ResNet, VGG) and fine-tuning. Drop LR by 10x at fixed milestones (e.g., epochs 30, 60, 90). Simple to tune but causes transient loss spikes at each drop. Good for shorter training runs.
OneCycleLR / Warmup+Linear Decay
Use for fast convergence when compute budget is limited. OneCycleLR works well with SGD+momentum for vision tasks. Linear warmup + linear decay is standard for BERT/GPT fine-tuning (3-5 epochs, warmup 6-10% of steps).
Warmup necessity: Large-batch training destabilizes early iterations (gradient norms vary wildly); warmup smooths initial gradient flow. Without it, loss explodes.
Validate schedule: Plot training loss curve. Does earlier decay help (lower final loss) or hurt (slower convergence)? Validate on final validation metric.
👉 Try this: Start with LR=0.001 and watch smooth training. Increase to 0.01 and see oscillations. Jump to 0.1 and watch divergence. Then select "Cosine" schedule and watch LR decay smoothly—better convergence than constant. Try "Exponential" with different decay rates.
Warmup = preparation (stabilize); cosine = smooth descent (avoid shocks); flat minima = robustness (generalize well).

FP32: 32-bit floating point. 1 sign bit, 8 exponent bits, 23 mantissa bits. Full precision, slow on modern hardware.
FP16: 16-bit floating point. 1 sign, 5 exponent, 10 mantissa. 2x faster, 2x memory savings, but underflow issues with small gradients.
BF16: 16-bit "brain float". 1 sign, 8 exponent (same as FP32!), 7 mantissa. Better range, less precision, preferred in modern ML.
Loss Scaling: Multiply loss by scale factor before backward pass to prevent gradient underflow. Divide scaled gradients by scale after backward.
Mixed Precision Training and Numerical Stability
Key Insight: Mix FP16/BF16 (fast, low memory) and FP32 (accurate) for \(2\times\) speedup; loss scaling amplifies gradients to avoid FP16 underflow.
FP16
1 sign, 5 exponent, 10 mantissa; \(2\times\) speed, \(2\times\) memory savings; limited range (min \(\sim 6 \times 10^{-5}\))—gradients underflow to zero
BF16
1 sign, 8 exponent (matches FP32!), 7 mantissa; preserves range while saving memory; no underflow risk, more robust
Loss scaling technique:Multiply loss by scale (e.g., \(2^{15}\)) before backward, amplifying gradients; divide after backward; dynamic scaling adjusts for overflow/underflow
Autocast (PyTorch):Automatically use FP16/BF16 for matmuls (stable), FP32 for reductions/losses (require full precision); near-FP32 convergence with \(2\times\) speedup
Gradient underflow:FP16 gradients \(< 6 \times 10^{-5}\) become zero \(\rightarrow\) no gradient update \(\rightarrow\) training stalled. Loss scaling prevents by enlarging gradients.
Production preference:TPUs use BF16 natively (hardware-efficient); GPUs A100+ prefer BF16 (better stability than FP16); older GPUs default to FP16
When to Use
BF16 Mixed Precision
Preferred on A100/H100 GPUs and TPUs. Same dynamic range as FP32 so no loss scaling needed. Use for LLM pre-training (GPT, Llama), large vision models (ViT-L), and any training where gradient underflow has been problematic with FP16.
FP16 Mixed Precision + Loss Scaling
Use on older GPUs (V100, T4) that lack BF16 support. Requires dynamic loss scaling to prevent gradient underflow. Standard for most vision training (ResNet, EfficientNet) and NLP fine-tuning where gradients are well-behaved.
Full FP32 (No Mixed Precision)
Use only when numerical precision is critical and memory is not a constraint (e.g., scientific simulations, reinforcement learning with very small reward signals, debugging NaN issues). 2x slower and 2x more memory than mixed precision.
Debug numerical instabilities: Monitor loss and gradient norms across iterations. Exploding norms → reduce LR or increase loss scaling. NaN/Inf → investigate data or loss function.
When to use BF16 vs FP16: BF16 has range advantage (prefer unless hardware accelerates FP16). If gradient norms are very small (< 1e-4), loss scaling must be more aggressive.
👉 Try this: Train with FP32 and watch memory usage and speed. Switch to FP16 and see 2x speedup, half memory. Try BF16 (faster overflow protection) and compare stability. Set loss scaling to 1.0 with FP16 and watch gradients underflow; increase scaling to 1024 and recover.
FP16 = fast but fragile (underflow risk); BF16 = fast and robust (range-safe); loss scaling = safety net (prevent underflow).
Gradient Accumulation

Drag the Accumulation Steps slider from 1 to 16. Watch how effective batch size grows without increasing GPU memory. Observe the gradient norm—when does it become too small?

32
Effective Batch: 128

Gradient Accumulation: Process multiple micro-batches without updating weights, accumulating gradients in place. After N micro-batches, run optimizer step. Effective batch size = micro-batch size × accumulation steps.

Use Case: When you need large batch size (for stability/convergence) but GPU memory is limited. Alternative to distributed training for single GPU.

Implementation: (1) Forward pass on micro-batch, (2) Backward without optimizer step (use scaler.scale for mixed precision), (3) Repeat N times, (4) Optimizer step, (5) Zero gradients.
Gradient Accumulation for Effective Batch Size Control
Key Insight: Simulate large batch size B by accumulating gradients over N micro-batches (size B/N each), then apply optimizer step on aggregated gradients.
$$\text{Accumulated gradient} \approx \text{average}(N \text{ gradient steps}) \approx \text{one step on batch size } B$$even if \(B\) doesn't fit in GPU memory
Accumulated gradient:Sum of gradients from \(N\) micro-batches; stored in parameter .grad attributes; approximates gradient on full batch \(B\)
average(\(N\) gradient steps):Average of \(N\) individual backward passes; scaled by \(1/N\) when computing optimizer step to match single batch
\(N\):Number of accumulation steps (micro-batches); typical range 2-8; larger \(N\) trades compute time for memory savings
\(B\) (effective batch size):Target total batch size; \(B = N \times B_{\text{micro}}\) where \(B_{\text{micro}}\) is GPU-resident micro-batch size
\(B_{\text{micro}}\) (micro-batch size):Batch size per forward-backward pass; fits in GPU memory; typically 32-256
\(B/N\):Micro-batch size; size of each gradient computation before accumulation
Execution flow:(1) forward on micro-batch i, (2) backward (gradients accumulate), (3) repeat i=2..N, (4) optimizer step, (5) zero gradients
Trade-off:Memory reduced by \(N\) (effective batch \(B\) with micro-batch \(B/N\)); wall-clock time increased \(\sim N \times\) (\(N\) forward-backward iterations)
Convergence benefit:Larger effective batch size improves generalization and gradient stability; empirically often converges better than single micro-batch
Critical implementation:Do NOT zero between accumulation steps (loses accumulated gradients); with autocast, scale/unscale correctly; gradient clipping after full accumulation
Batch normalization:Computed per micro-batch (not full batch); causes biased estimates; synchronized batch norm across accumulation complex (usually ignored)
When to Use
Gradient Accumulation (Single GPU)
Use when you need large effective batch size but only have 1 GPU (e.g., fine-tuning BERT-large on a single A100, want batch 256 but only 64 fits in memory). Set accumulation steps = 4. Training is 4x slower but equivalent to batch 256.
Distributed Data Parallelism (Multi-GPU)
Prefer when multiple GPUs are available and batch fits per-GPU. 4 GPUs with batch 64 each = effective batch 256, with near-linear speedup. Faster wall-clock time than accumulation. Use DDP (PyTorch) or tf.distribute.
Accumulation + DDP Combined
Use for very large effective batch sizes in LLM pre-training (e.g., batch 2048 across 8 GPUs, each accumulating 32 steps with micro-batch 8). Common in GPT/Llama training where large batches improve throughput and convergence stability.
Batch size for stable training: Typical rule is 32-256 for vision, 64-512 for NLP. If loss noisy, increase batch size; if gradient updates too conservative, decrease.
Choose accumulation steps: If effective batch B needed but GPU memory only holds B/N, use N accumulation steps. Example: want batch 512, GPU holds 128 → N=4 steps.
👉 Try this: Set batch size=16, accumulation=1 (total 16). Then keep batch=8, accumulation=2 (same total, less memory). Compare training curves—should match. Increase accumulation to 8 with batch=2 and observe when accumulated gradient becomes stale.
Gradient accumulation = financial analogy: collect payments (gradients) over time, apply one large transaction (optimizer step) periodically.

ZeRO Stage 1: Partition optimizer states (momentum, variance) across GPUs. 4x memory reduction for optimizer states. Each GPU computes gradients for all parameters.

ZeRO Stage 2: Also partition gradients across GPUs. 8x total memory reduction. Gradients are local to partition - gathered only when needed.

ZeRO Optimization: Partitioning for Massive Scale
Key Insight: ZeRO partitions optimizer state/gradients/parameters across GPUs, enabling training of models \(10\text{-}100\times\) larger by trading memory for communication.
$$\text{Per-param memory: } 4\text{ bytes (weight)} + 4\text{ bytes (grad)} + 8\text{ bytes (Adam state)} = 16\text{ bytes}$$7B model = 112GB per GPU \(\rightarrow\) impossible
Per-param memory:Total memory required per parameter across all training components
4 bytes (weight):FP32 model parameter; 1 parameter = 4 bytes; \(7B \times 4 = 28\text{GB}\)
4 bytes (grad):FP32 gradient accumulator; 1 parameter = 4 bytes; \(7B \times 4 = 28\text{GB}\)
8 bytes (Adam state):Two FP32 buffers (momentum \(m\), variance \(v\)) per parameter; \(2 \times 4 = 8\) bytes; \(7B \times 8 = 56\text{GB}\)
16 bytes/param total:Sum of weight + grad + optimizer state; \(7B \times 16 = 112\text{GB}\) per single GPU (impossible for typical 40-80GB VRAM)
\(N\):Number of GPUs in distributed training group; ZeRO partitions across \(N\) devices
Stage 1 (Optimizer states):Each GPU stores \(1/N\) of Adam momentum/variance; memory \(\rightarrow\) 12 bytes/param; 7B + 8 GPUs = 14GB/GPU (\(11\times\) reduction)
Stage 2 (Gradients + Stage 1):Each GPU stores \(1/N\) of gradient buffer; memory \(\rightarrow\) 8 bytes/param; 7B + 8 GPUs = 7GB/GPU (\(16\times\) reduction)
Stage 3 (Parameters + Stage 1-2):Parameters gathered on-the-fly during forward/backward; memory \(\rightarrow\) 4 bytes/param; 7B + 8 GPUs = 3.5GB/GPU (\(32\times\) reduction); \(2\text{-}3\times\) communication overhead
CPU offloading:Store partitions in pinned CPU RAM, asynchronously transfer to GPU; further GPU memory reduction; requires high PCIe bandwidth
Trade-off tiers:Stage 1-2: cost-effective (10-20% overhead); Stage 3: necessary for extreme scale but requires high-bandwidth interconnects (NVLink, RoCE)
When to Use
ZeRO Stage 1-2
Default for models that almost fit in GPU memory (e.g., 7B model on 8x A100-40GB). Low communication overhead (10-20%). Stage 1 alone often sufficient for fine-tuning; Stage 2 adds gradient partitioning for pre-training larger models.
ZeRO Stage 3 (+ CPU Offload)
Use for models that cannot fit even optimizer states in GPU memory (e.g., 70B model on 8x A100-80GB). Partitions everything including parameters. CPU offload extends further but requires high PCIe bandwidth. Essential for training Llama-65B, Falcon-180B.
FSDP (PyTorch Native)
PyTorch's built-in alternative to ZeRO Stage 3. Prefer when already in PyTorch ecosystem without DeepSpeed dependency. Equivalent memory savings, integrated with torch.compile. Use for models 13B-70B parameters on moderate GPU clusters.
Communication overhead analysis: If bandwidth is B GB/s and gather 1TB parameters, overhead = 1TB / B seconds per iteration. Example: 100 GB/s × 1TB = 10s overhead—significant.
When to offload to CPU: GPU memory < 10GB available; trade GPU-CPU PCIe bandwidth for GPU VRAM. Typical: CPU offloading worthwhile only if PCIe bandwidth > 50 GB/s.
👉 Try this: Stage 0 (no sharding) baseline—see full optimizer/gradient/param memory. Stage 1: optimizer state splits (25% memory). Stage 2: gradients split (50% total). Stage 3: parameters split (75% total, all-gather on backward). Watch memory drop and communication increase.
ZeRO = sharing is caring: partition parameters/gradients across GPUs, gather only what's needed, reduce per-GPU memory drastically.

Activation Checkpointing: Trading Compute for Memory
Key Insight: Selectively checkpoint activations instead of storing all; recompute during backward; memory reduces \(O(L) \rightarrow O(\sqrt{L})\) with modest compute overhead.
Standard (All activations)
Store all \(h_1, \ldots, h_L\) during forward; memory \(O(L)\); backward uses stored activations instantly
Selective checkpoint
Store every \(k\)-th layer (\(k=\sqrt{L}\)); recompute missing on-demand during backward; memory \(O(\sqrt{L})\), time \(T_f \times 1.2\text{-}1.5\)
Full recomputation:Store only inputs, recompute all activations during backward; memory \(O(1)\) (maximum savings), \(\sim\)33% slowdown (extra forward passes)
Checkpoint frequency:Trade-off: more frequent checkpoints → less recomputation (faster) but more memory; less frequent → more recomputation (slower) but less memory
Math:If forward = \(T_f\), memory = \(M\); selective checkpoint \(\approx T_f + \frac{1}{\sqrt{L}} \cdot T_f \approx 1.2\text{-}1.5 \cdot T_f\) time, \(M/\sqrt{L}\) memory
When worthwhile:Deep networks (\(L > 100\), e.g., ResNet-200, GPT-3); memory-constrained hardware; compute cheaper than memory bandwidth
Production use:Checkpointing + ZeRO-3 enables training 100B+ parameter models on 8-16 GPUs with 20-30% slowdown (acceptable)
When to Use
Selective Checkpointing (sqrt(L))
Best trade-off for most deep networks. Checkpoint every sqrt(L) layers (e.g., every 7th layer in a 48-layer Transformer). Reduces activation memory by 7x with only 20-30% compute overhead. Standard for GPT/Llama pre-training.
Full Recomputation
Use when memory is extremely constrained and you need maximum batch size (e.g., training 70B model on limited GPUs). ~33% slower but activation memory drops to O(1). Combine with ZeRO-3 for training models that otherwise would not fit.
No Checkpointing
Use for small models (ResNet-18, DistilBERT) or when GPU memory is abundant relative to model size. Avoids recomputation overhead entirely. Fine-tuning small models on A100-80GB rarely needs checkpointing.
Debug checkpointing: Verify gradients match between checkpointed and non-checkpointed via numerical differentiation (finite differences); catches recomputation errors.
Choose k (checkpoint frequency): Start k=√L; if still OOM, increase k (less checkpoints, more recompute). If mostly memory-bound, decrease k (more checkpoints, less recompute).
👉 Try this: Batch=32 without checkpointing—set as baseline memory. Enable checkpointing and memory drops 50%. Now increase batch to 64 with checkpointing (same peak memory as original batch 32). Observe speed slowdown (recomputation overhead) vs memory savings.
Checkpointing = bookmarks: mark certain pages (layers), skip middle pages during reading (backward), flip back to marked pages to reread (recompute).

When to Use
Gradient Clipping (max_norm)

EMA Checkpointing (Save Best Model)
Use for any training run longer than a few hours. Save model weights at best validation metric, not just the last checkpoint. EMA (exponential moving average) of weights often generalizes better than final weights. Standard in LLM and diffusion model training.
Automated Alerting (W&B, MLflow)
Essential for multi-day training runs (LLM pre-training, large-scale distributed jobs). Set alerts on: loss spike > 2x EMA, gradient norm > 100, throughput drop > 20%. Saves thousands of GPU-hours by catching failures early instead of discovering them days later.
Leading indicators of failure: Gradient norms explode 1-2 iterations before NaN loss; catch early via norm thresholds (alert if >100).
Monitoring architecture: Central logging (W&B, TensorBoard, MLflow) with alerting thresholds; saves best validation model (not last); enables post-hoc analysis.

Hardware Awareness and the Roofline Model
Key Insight: Roofline model predicts performance: workloads are compute-bound or memory-bound; understand arithmetic intensity to optimize kernels.
$$\text{Peak Performance} = \min(\text{Peak FLOPS},\; \text{Arithmetic Intensity} \times \text{Bandwidth})$$
Peak_Performance:Maximum achievable throughput (FLOPS); limited by either compute or memory, whichever is tighter
min(·,·):Takes minimum of two terms; bottleneck is whichever resource saturates first (compute or memory)
Peak_FLOPS:Theoretical peak floating-point operations per second; GPU spec (e.g., A100: 312 TFLOPS FP32); hardware limit
Arithmetic Intensity (AI):Ratio of FLOPs to bytes transferred (FLOPS/byte); measures compute-to-communication ratio; higher AI \(\rightarrow\) more compute-bound
Bandwidth:Maximum memory bandwidth (bytes/sec); GPU-memory throughput (e.g., A100 HBM: 2TB/s); hardware limit
AI × Bandwidth:FLOPS achievable if memory bandwidth is saturated; determines memory-bound performance ceiling
Compute-bound condition:\(\text{AI} \times \text{Bandwidth} \geq \text{Peak FLOPS}\); workload compute requirement exceeds bandwidth capacity; optimize compute
Memory-bound condition:\(\text{AI} \times \text{Bandwidth} < \text{Peak FLOPS}\); workload limited by memory bandwidth, not compute; optimize memory access
Memory hierarchy:SRAM (1 cycle), L2 (10 cycles), HBM (100 cycles)—100× latency difference; data movement dominates latency
Arithmetic intensity (AI):FLOPs per byte transferred; matmul high AI (\(O(n^3)\) FLOPs, \(O(n^2)\) memory \(\rightarrow\) compute-bound); ReLU low AI (\(O(n)\), memory-bound)
Compute-bound:AI high enough that computation saturates GPU compute (achieves near-peak FLOPS); examples: dense matmuls, convolutions
Memory-bound:Memory bandwidth saturates first; increasing compute doesn't help (still limited by BW); examples: element-wise ops, normalization
Flash Attention:Standard attention memory-bound (softmax \(O(n^2)\) HBM passes); Flash computes in SRAM (tiling, recomputing) \(\rightarrow O(n)\) passes \(\rightarrow\) compute-bound
Kernel fusion:Combine operations to avoid intermediate HBM writes/reads; 2-10× speedup vs naive implementation
When to Use
Flash Attention / Kernel Fusion
Use for any Transformer model with long sequences (>512 tokens). Flash Attention reduces attention from O(n^2) HBM access to O(n) via SRAM tiling. 2-4x speedup on GPT, Llama, and diffusion model training. Enable via xformers or PyTorch 2.0 SDPA.
Operator Fusion (torch.compile)
Use when your model has many element-wise operations (layer norm, GELU, dropout) that are individually memory-bound. torch.compile fuses them into single GPU kernels, reducing HBM round-trips. 10-30% speedup on typical Transformer training with zero code changes.
Hardware-Aware Batch Size Tuning
Use when GPU utilization is below 70%. Increase batch size until compute-bound (matmul saturates GPU). For memory-bound workloads, profile with nsight/nvprof to find the bottleneck. Example: doubling batch size on A100 may not increase wall-clock time if memory-bound.
Calculate AI for your model: Count FLOPs (matrix multiplications, convolutions) and memory transfers (gradient updates, batch norm). High AI = compute-bound (optimize compute); low AI = memory-bound (optimize memory access).
Quantization effect: Lower precision reduces memory traffic (good for memory-bound ops) but reduces compute (hurts compute-bound ops). Analyze AI before and after quantization.

Roofline = ceiling for performance: either compute or memory bandwidth caps you; find which and optimize that bottleneck.
