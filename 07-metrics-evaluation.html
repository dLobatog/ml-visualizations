Interview Tips: ROC Curve and AUC

What it is: The ROC (Receiver Operating Characteristic) curve visualizes the trade-off between True Positive Rate (sensitivity) and False Positive Rate (1 - specificity) across decision thresholds. AUC quantifies discrimination ability: 0.5 = random, 1.0 = perfect.

Formulas: TPR = TP/(TP+FN), FPR = FP/(FP+TN)
AUC = P(scorepositive > scorenegative)
scorepositive:Model's predicted score/probability for a randomly chosen positive example from the dataset
scorenegative:Model's predicted score/probability for a randomly chosen negative example from the dataset
P():Probability function; AUC = the probability that the model ranks a random positive higher than a random negative
AUC Range:∈ [0, 1]. AUC = 0.5 means random classifier, AUC = 1.0 means perfect discrimination
ROC = Receiver Operating Characteristic. Imagine moving a threshold dial from 0 to 1; the curve traces all possible (FPR, TPR) trade-offs.
When is AUC misleading? With highly imbalanced data (99% negatives), AUC can be high (0.8+) yet model fails on minorities. Use Precision-Recall instead.
Why report AUC instead of picking one threshold? Different applications have different false positive vs. false negative costs. AUC lets stakeholders choose their operating point.
👉 Try this: Set threshold to 0.1—TPR soars near 1.0, FPR near 1.0. Drag to 0.9—TPR drops to 0.1, FPR to 0.01. Watch the red dot trace the curve. At 0.5, is it above or below diagonal? Drag to find where AUC=0.75 line would intersect.

Key insight: AUC = probability that model ranks a random positive higher than a random negative. For AUC=0.85, 85% of positive-negative pairs are ranked correctly.

Production tip: Use AUC for model comparison, but always select explicit threshold based on business costs. Tune on validation set only, test on held-out data.
Pitfall: High AUC does NOT guarantee well-calibrated probabilities. Monitor calibration separately.

Interview Tips: Precision-Recall Curve

What it is: PR curve plots Precision vs Recall across thresholds. Superior to ROC for imbalanced data because it focuses on minority class performance only.

Precision = TP/(TP+FP) | Recall = TP/(TP+FN) | F1 = 2·Precision·Recall/(Precision+Recall)
TP (True Positives):Model correctly predicted positive class
FP (False Positives):Model incorrectly predicted positive (ground truth was negative)
FN (False Negatives):Model incorrectly predicted negative (ground truth was positive)
Precision Range:∈ [0, 1]; high precision = few false alarms, low precision = many false positives
Recall Range:∈ [0, 1]; also called sensitivity or true positive rate; high recall = few positives missed
F1 Range:∈ [0, 1]; harmonic mean of precision and recall; balanced metric, penalizes extreme skew
Precision = "of my positive predictions, how many are correct?" (quality of positives) | Recall = "of true positives in data, how many did I catch?" (coverage)
Critical: PR baseline = prior probability. For 1% positives, baseline = 0.01. For ROC, baseline = 0.5. This makes PR-AUC more honest for imbalanced problems.
Why PR over ROC for imbalanced data? ROC's FPR dominated by huge TN count—model can achieve high AUC just by predicting all negatives. PR focuses on minority only.
Can a model have high precision but low recall? Yes! Predict positive only on high-confidence examples. E.g., fraud detection at 95% precision but 5% recall (catches few frauds).
👉 Try this: Set confidence threshold to 0.1 and watch recall spike (catch all positives) but precision tank. Drag to 0.9 and precision rises but recall crashes. Find the 'sweet spot' at 0.5. Does this curve show class imbalance?
F1 interpretation: F1 picks one point on PR curve (threshold maximizing F1). Full curve shows all trade-offs across thresholds.
Production rule: Monitor precision and recall separately, not collapsed into F1. Different stakeholders have different error cost tolerances.

Interview Tips: Confusion Matrix and Derived Metrics

What it is: 2x2 matrix (binary) or NxN (multiclass) showing joint distribution of predicted and true labels. Foundation for all threshold-dependent metrics and reveals systematic classifier errors.

Precision = TP/(TP+FP) | Recall = TP/(TP+FN) | Specificity = TN/(TN+FP)
TN (True Negatives):Model correctly predicted negative class
Specificity Range:∈ [0, 1]; also called true negative rate; high specificity = few false positives
Accuracy = (TP+TN)/(TP+FP+FN+TN) | F1 = 2·P·R/(P+R)
Accuracy Range:∈ [0, 1]; overall correctness across all predictions; misleading on imbalanced datasets
P (Precision):See Precision-Recall section; abbreviation for Precision
R (Recall):See Precision-Recall section; abbreviation for Recall
Remember the four cells: TP (correct positive), FP (wrong positive), FN (wrong negative), TN (correct negative).
Why is accuracy misleading on imbalanced data? With 99% negatives, predicting all negatives = 99% accuracy but zero recall. Ignores class imbalance entirely.
Macro vs Micro: Micro: sum TP, FP, FN across classes then compute metric (weighted by support). Macro: compute metric per class, average equally (ignores support).
When do false positives cost more than false negatives? When alerting users with false positives is expensive/annoying (spam), or triggers costly human review.
👉 Try this: Toggle threshold from 0.1 to 0.9 and watch TP+TN peak at different thresholds than FP+FN. Notice when Accuracy=95% but Recall=10%—model predicts too many negatives. Find the threshold matching your business cost (FP vs FN).
Production rule: Never report accuracy alone. Always show precision, recall, F1 per class. Confusion matrix reveals distribution shift over time.
Fairness check: Stratify confusion matrices by demographic group, geography, segments. High global accuracy can hide poor performance on minorities.

ECE: Average absolute difference between predicted confidence and actual accuracy in probability bins.

Interview Tips: Calibration Plot and Confidence

What it is: Calibration = whether predicted confidence matches empirical correctness. A 70% confidence prediction should be correct 70% of the time. Critical for downstream decision-making (credit limits, thresholds, recommendations).

ECE = Σ|predicted_confidencebin - empirical_accuracybin| × bin_fraction
Σ:Sum over all K probability bins (e.g., K=10 for deciles: [0-0.1], [0.1-0.2], ..., [0.9-1.0])
predicted_confidencebin:Average predicted probability for all samples falling into this bin
empirical_accuracybin:Fraction of samples in this bin that have correct predictions (actual positive labels)
bin_fraction:Proportion of total samples that fall into this bin; ∑ bin_fraction = 1
ECE Range:∈ [0, 1]; lower is better; 0 = perfect calibration, measures average mismatch across bins
Brier Score = mean((predicted_prob - true_label)2) | Perfect calibration = diagonal line y=x
predicted_prob:Model's predicted probability ∈ [0, 1] for positive class
true_label:Ground truth label, ∈ {0, 1} (0 for negative, 1 for positive)
Brier Score Range:∈ [0, 1]; lower is better; 0 = perfect predictions, 0.25 = predicting always 0.5, 1 = always wrong
Perfect calibration: if model says 70%, it's correct 70% of the time. Overconfident: says 90% but only correct 60%. Underconfident: says 60% but correct 90%.
Model patterns: DNNs = overconfident (Adam + weight decay encourages it). Logistic Regression = naturally well-calibrated. Tree ensembles = underconfident.
Why does my neural network output 99% confidence on wrong predictions? DNNs optimize for accuracy, not calibration. Use dropout, weight decay, label smoothing, or post-hoc calibration.
👉 Try this: Compare the blue diagonal (perfect) to red model curve. If red is above = overconfident (says 70%, actually 50%). Below = underconfident. Drag temperature slider left (T=0.5) and watch red bend toward diagonal—calibration improves.
Post-hoc fixes: Temperature scaling: p_cal = softmax(logits/T), T>1 reduces overconfidence. Isotonic regression: fit monotonic mapping, preserves ranking.
How is calibration different from AUC? AUC = discrimination (ranking). Calibration = trustworthiness of probabilities. High AUC ≠ good calibration.
Production pitfall: Never tune temperature scaling on training data—overfits to training distribution. Use separate held-out calibration set. Monitor ECE drift in production quarterly.

Interview Tips: Ranking Metrics (NDCG and MAP)

What it is: NDCG evaluates ranking quality by measuring how well relevant items are placed, penalizing low positions. Users consume top-k results predominantly, so position matters exponentially.

DCG = Σ(relevancei / log2(i+1)) | NDCG = DCG / iDCG
Σ:Sum over all ranked positions i from 1 to k (or total results)
relevancei:Relevance score of item at position i; typically 0 (not relevant), 1 (relevant), or 2-3 (highly relevant)
i:Position in ranking, starting from 1 (top)
log2(i+1):Logarithmic discount factor; position 1: 1.0, position 2: 0.63, position 4: 0.43, position 8: 0.33
iDCG (Ideal DCG):DCG of perfect ranking (all relevant items at top, sorted by relevance descending)
NDCG Range:∈ [0, 1]; normalized to [0, 1] where 1 = perfect ranking, 0 = worst ranking
MAP = Σ(precision@k) / (# relevant items), where k = positions of relevant items
precision@k:Precision computed at each position k where a relevant item appears; 0 if no relevant items at/before position k
# relevant items:Total number of relevant items in the entire ground truth result set
k:Positions where relevant items appear; MAP averages precision only at these positions, ignoring non-relevant positions
MAP Range:∈ [0, 1]; 1 = all relevant items ranked first, 0 = no relevant items ranked first
NDCG uses log discount: position 1 = 1.0, position 2 = 0.63, position 4 = 0.43. Reflects that users rarely scroll past 5.
NDCG@k cutoffs: @5 = mobile/quick searches. @10 = exploratory search. @100 = backend ranking. Report multiple cutoffs.
Why log discount instead of linear? Log matches user behavior: position 2 vs 3 is bigger drop than 100 vs 101. Exponential decay in attention.
How does NDCG handle missing relevant items? Does NOT penalize missing items. iDCG includes all 10 items even if you only ranked top 5. Differs from Recall.
MAP vs NDCG: MAP = no position discount, just average precision at relevant positions. NDCG = position-weighted. NDCG more sensitive to ranking order.
Position bias trap: Users rarely scroll past position 5. Clicks NOT reliable relevance signals. Models learn to exploit this. Use inverse propensity weighting.
👉 Try this: Move wrong item to position 1 and watch NDCG crash to ~0.5 (huge log penalty). Move same item to position 5 and NDCG rises to ~0.85 (smaller penalty). Compare to MAP—it penalizes position 1 and 5 equally. At position 10, which metric cares more, NDCG or MAP?
Production rule: Offline NDCG for development speed, but ALWAYS A/B test with online metrics (CTR, conversion, dwell time). Offline metrics can be misleading.
