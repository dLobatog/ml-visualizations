1. Feature Scaling & Normalization

Different scaling methods have significant impacts on model training, especially for distance-based algorithms and neural networks.

Interview Tips: Feature Standardization and Scaling

What it is: Rescales numeric features to common scale. Critical for linear models, distance-based algorithms, regularization. Different methods for different data distributions.

$$z = \frac{x - \mu}{\sigma} \quad \rightarrow \quad \mu=0,\; \sigma=1$$
\(x\):Original feature value
\(\mu\):Sample mean of the feature; computed on training data only
\(\sigma\):Sample standard deviation; computed on training data only
\(z\):Standardized value with \(\mu=0, \sigma=1\); allows all features comparable
$$x_{\text{scaled}} = \frac{x - x_{\min}}{x_{\max} - x_{\min}} \quad \rightarrow \quad \text{range } [0,1]$$
\(x\):Original feature value
\(x_{\min}\):Minimum value in training data
\(x_{\max}\):Maximum value in training data
\(x_{\text{scaled}}\):Scaled value, always \(\in [0, 1]\); sensitive to outliers if min/max based on outliers
$$x_{\text{scaled}} = \frac{x - \text{median}}{IQR} \quad \rightarrow \quad \text{outlier-resistant}$$
\(x\):Original feature value
median:50th percentile of training data; unaffected by extreme outliers
IQR (Interquartile Range):\(Q_3 - Q_1\) (75th percentile - 25th percentile); insensitive to outliers
\(x_{\text{scaled}}\):Scaled value; robust to outliers; different range than \([0,1]\) (depends on data distribution)
Z-Score puts all features on common scale. Salary (high variance) won't dominate age (low variance) in L2 regularization.
Best for: Z-Score = linear models, DNNs, distance-based (KNN, K-means). Min-Max = bounded features, probabilities. Robust = outliers present.
Do tree-based models need scaling? No. Decision trees invariant to scaling. Standardization unnecessary and wastes compute.
👉 Try this: Click the Z-Score button and watch the distribution center at 0. Then try Robust scaling with the log-transformed data—see how it handles outliers differently. Compare Min-Max for bounded features vs Z-Score for unbounded.
Should I standardize before or after train/test split? Always AFTER. Fit mean/std on train only, apply to test. Prevents data leakage.
👉 Try this: Use Log Transform on income-like data (right-skewed). Watch how it stabilizes variance compared to raw scaling. Then increase kurtosis—see which scaling handles heavy tails best.
When to Use
Z-Score (StandardScaler)
Gaussian-ish features with no hard bounds. Linear regression, logistic regression, SVMs, neural networks. E.g., user age, temperature readings, stock returns. Required when using L1/L2 regularization.
Min-Max Scaling
Features with known bounds or when you need [0,1] range. Image pixel values (0-255), probabilities, percentage features. Sensitive to outliers -- one extreme value distorts entire scale.
Robust Scaler
Data with significant outliers that are real (not errors). Income data, insurance claims, web traffic spikes. Uses median/IQR instead of mean/std, so outliers do not distort the scaling.
Log transform: For right-skewed distributions (income, traffic). Use \(\log(x+1)\) to handle zeros. Normalizes variance.
Production rule: Store scaler params (mean/std) versioned with model. At inference, use EXACT same params. Drift >5% in stats = retrain scaler.
Non-normal data? Use robust scaling (median/IQR) or power transform (Yeo-Johnson, Box-Cox) before standardization.
👉 Try this: Toggle outlier magnitude slider to 10x and rewatch Z-Score vs Robust. Notice how Robust stays stable while Z-Score gets pulled. This is why Robust is best when outliers are real (not errors).

                import numpy as np
                
                X_train = np.array([[1, 100], [2, 200], [3, 50000]])
                X_test = np.array([[1.5, 150]])
                
                scaler = StandardScaler()  # or MinMaxScaler(), RobustScaler()
                X_train_scaled = scaler.fit_transform(X_train)
                X_test_scaled = scaler.transform(X_test)  # use train params
                print(f"Mean: {X_train_scaled.mean(axis=0)}")  # [0, 0]
                print(f"Std:  {X_train_scaled.std(axis=0)}")   # [1, 1]

Always fit on train, transform on test. RobustScaler uses median/IQR for outlier resistance.

2. Feature Encoding

Categorical variables require encoding before model training. Different techniques suit different scenarios.

Interview Tips: Feature Encoding for Categorical Variables

What it is: Converts categorical variables to numeric for ML. Three main methods: one-hot (binary indicators), label (integers), target (mean encoding).

$$\text{One-Hot: } d \text{ categories} \rightarrow d-1 \text{ binary columns (reference category dropped)}$$
\(d\):Number of unique categories/classes in the categorical variable
\(d-1\):Number of binary columns created; one category chosen as reference (usually most frequent or first alphabetically)
Binary columns:Each column \(\in \{0, 1\}\); value=1 if sample belongs to that category, 0 otherwise; reference category = all zeros
$$\text{Target Encoding: category} \rightarrow \mathbb{E}[y \mid \text{category}] = \text{proportion of positives}$$
category:A unique value in the categorical feature (e.g., "red" in color feature)
\(y\):Target variable; for binary classification, \(y \in \{0, 1\}\)
\(\mathbb{E}[y \mid \text{category}]\):Average of target values for all samples with this category; represents probability of positive class
Target Encoding Range:\(\in [0, 1]\) for binary classification; captures class imbalance per category
One-hot: [red, blue, green] → columns [red, blue] (green=reference=[0,0]). Prevents dummy variable trap (multicollinearity).
One-Hot
Binary features, [0,1] range, no ordering. Best for: linear models, DNNs. Bad for: high cardinality (explosion).
Label
Integers 1,2,3… Introduces artificial ordering. Only for tree models. Avoid elsewhere.
Target Encoding
Category → mean target. Powerful but risks overfitting. Needs cross-validation. Leaks target info.
Embeddings
Learn low-rank dense vectors. For very high cardinality (millions of values). Replaces one-hot.
When to Use
One-Hot Encoding
Low cardinality (< 20 categories), no natural ordering. E.g., color (red/blue/green), country, browser type. Use with linear models, DNNs. Drop one column to avoid multicollinearity.
Label / Ordinal Encoding
Natural ordering exists OR tree-based models only. E.g., education level (HS < BS < MS < PhD), satisfaction (low/med/high). Trees handle arbitrary integers; linear models will misinterpret ordering.
Target / Embedding Encoding
High cardinality (100+ categories). ZIP codes, user IDs, product SKUs. Target encoding: fast but needs CV to prevent leakage. Embeddings: learn dense representations in DNNs, best for millions of categories.
Why drop one column in one-hot? All d columns sum to 1 (perfect multicollinearity). Drop one → preserves info + maintains rank.
👉 Try this: Switch Encoding Method to One-Hot and observe the dummy variable trap prevention. Toggle to Label encoding to see artificial ordering introduced. Then use Target Encoding on a high-cardinality feature—watch how it smoothly compresses categories.
High-cardinality features (millions of values)? Don't one-hot—causes sparsity + memory bloat. Use target encoding or embeddings instead.
👉 Try this: Click through all three encoding methods. For One-Hot, count output dimensions; for Target, notice each category becomes a single feature. Target Encoding is your friend for postal codes, ZIP codes, or user IDs.
Unseen categories: Map to "unknown" column, or use smoothed target encoding (blend with global mean). Trees: map to most frequent training category.
Production rule: Store encoder versioned with model. Fit on train only (no leakage). Monitor cardinality for new categories. Switching encoding types mid-deployment breaks consistency. 
from sklearn.preprocessing import OneHotEncoder, LabelEncoder
                import numpy as np

                X_encoded = ohe.fit_transform(categories)
                print(X_encoded)  # drops 'blue' as reference
                # [[0. 1.]    green=0, red=1
                #  [0. 0.]    reference (blue)
                #  [1. 0.]    green=1, red=0
                #  [0. 1.]]
                
                le = LabelEncoder()
                y_encoded = le.fit_transform(['cat', 'dog', 'cat', 'bird'])
                print(y_encoded)  # [1, 2, 1, 0]

Use drop='first' in OneHotEncoder to avoid the dummy variable trap.

3. Handling Missing Data

Missing values must be handled strategically. Different imputation methods suit different data patterns.

Interview Tips: Handling Missing Data

What it is: Choice of imputation depends critically on missingness mechanism. Three types: MCAR (random), MAR (depends on observed), MNAR (depends on unobserved value).

MCAR = no info in absence | MAR = missingness correlated with observed features | MNAR = missingness reveals info about missing value (e.g., high earners hide income)
Mean/Median
Fast. Unbiased only for MCAR. Underestimates variance. Use for MCAR, low % missing.
KNN Imputation
Borrows from k nearest neighbors. Handles MAR. Slow in high dimensions (curse of dimensionality).
Model-based
Fit regression \(X_{\text{obs}} \rightarrow X_{\text{miss}}\). Powerful for MAR. Assumes model captures true relationship.
Multiple Imputation
Create m imputed datasets, pool results. Preserves uncertainty. Expensive compute.
Can I just delete missing rows? Only if MCAR AND <5% missing. Deleting MAR/MNAR rows biases estimates.
👉 Try this: Select KNN imputation and watch it use neighbors to fill gaps. Now increase missing percentage to 30%—KNN struggles in high-dimensional space (curse of dimensionality). Compare Mean vs Median on skewed data to see variance underestimation.
What if data is MNAR? Create binary missing indicator feature (is_missing=1/0). Then impute. Lets model learn missingness is predictive.
👉 Try this: Toggle between original (has missing) and Forward Fill methods. For MNAR data (e.g., high-income hiding income), the missing pattern itself is predictive—create an is_missing feature alongside imputation.
When to Use
Mean/Median Imputation
MCAR mechanism, < 5% missing, quick baseline. E.g., a few sensor readings dropped randomly. Median for skewed data (income), mean for symmetric (height). Fast, but underestimates variance.
KNN / Model-Based
MAR mechanism, features are correlated. E.g., missing blood pressure predicted from age + weight. KNN for < 50 features; regression-based for structured relationships. Expensive but preserves feature correlations.
Indicator + Impute (MNAR)
Missingness itself is informative. E.g., high earners hide income, sick patients skip follow-ups. Add binary is_missing column, then impute with any method. Lets model learn that absence = signal.
Impute when: After train/test split. Fit imputation on train only. Never fit on full dataset (leakage of test info).
Time-series trap: Forward fill creates dependence (violates i.i.d.). Fill before train/test split, never cross boundary. Use interpolation instead.
Production rule: Document strategy & % missing per feature. Monitor missingness drift. Features >50% missing = unreliable imputation, consider dropping. 
from sklearn.impute import SimpleImputer, KNNImputer
                import numpy as np
                
                X = np.array([[1, 2], [np.nan, 3], [7, np.nan], [4, 5]])
                
                # Mean imputation
                imp_mean = SimpleImputer(strategy='mean')
                print(imp_mean.fit_transform(X))
                
                # KNN imputation (uses k nearest neighbors)
                imp_knn = KNNImputer(n_neighbors=2)
                print(imp_knn.fit_transform(X))

SimpleImputer supports mean, median, most_frequent. KNNImputer preserves feature correlations.

4. Class Imbalance

Imbalanced classes distort model training and evaluation. Multiple strategies exist to handle this.

Interview Tips: Class Imbalance Handling

What it is: Minority class rare (e.g., 1% fraud, 0.1% disease). Algorithms optimize accuracy, which is trivial: "always predict negative" = 99% accuracy but zero recall on minority. Need special strategies: resampling, weighting, thresholding, different metrics.

$$x_{\text{synth}} = x_i + \lambda \cdot (x_j - x_i), \quad \lambda \in [0,1] \text{ random}$$
\(x_i\):A randomly selected minority class sample (the one to interpolate from)
\(x_j\):One of the \(k\) nearest neighbors of \(x_i\) in feature space (typically \(k=5\)), also minority class
\(\lambda\):Random interpolation weight, \(\lambda \in [0, 1]\) uniform; different value sampled for each synthetic sample
\(x_{\text{synth}}\):New artificial minority sample created by linear interpolation; lies between \(x_i\) and \(x_j\)
$$\mathcal{L} = w_{\text{min}} \cdot L(\hat{y}_{\text{neg}} \mid y_{\text{pos}}) + w_{\text{maj}} \cdot L(\hat{y}_{\text{pos}} \mid y_{\text{neg}})$$
\(w_{\text{min}}\):Class weight for minority class; typically \(> 1.0\) to penalize minority errors more; often inversely proportional to class frequency
\(w_{\text{maj}}\):Class weight for majority class; typically \(< 1.0\); often set to balance total weighted samples across classes
\(L(\hat{y}_{\text{neg}} \mid y_{\text{pos}})\):Loss for predicting negative when true label is positive (false negative); penalized by \(w_{\text{min}}\)
\(L(\hat{y}_{\text{pos}} \mid y_{\text{neg}})\):Loss for predicting positive when true label is negative (false positive); penalized by \(w_{\text{maj}}\)
Never use accuracy for imbalanced data. Always F1 (harmonic mean), AUC-PR (focuses on minority), or recall (catch positives).
Oversampling
Duplicate minority samples. Simple, risks overfitting. Use only if data is small.
SMOTE
Synthesize new samples (interpolate neighbors). Better than duplicates. Less effective in high dimensions.
Undersampling
Remove majority samples. Fast, loses data. For large datasets only.
Cost-sensitive
Higher penalty for minority errors. Efficient, no data loss. Equivalent to oversampling mathematically.
When to Use
SMOTE / Oversampling
Small datasets where losing majority samples hurts. E.g., rare disease detection with 500 total samples. SMOTE creates synthetic minority examples via interpolation. Avoid in high dimensions (> 50 features).
Undersampling
Large datasets where majority class is abundant. E.g., click prediction with 10M negatives, 100K positives. Randomly remove majority samples. Fast training, but discards potentially useful data.
Class Weights / Focal Loss
Any dataset size, preferred for DNNs. E.g., fraud detection (1:1000), set class_weight="balanced" in sklearn or focal loss in PyTorch. No data modification, no synthetic artifacts. Mathematically equivalent to oversampling.
Oversample before or after train/test split? ALWAYS AFTER. Before = leakage (duplicate in both sets inflates metrics).
👉 Try this: Drag the Imbalance Ratio slider to 1:50. Watch what happens with "None"—majority class dominates. Switch to SMOTE and observe synthetic samples between minority neighbors. Now try Class Weight to see the penalty trade-off without data duplication.
SMOTE in high dimensions? Curse of dimensionality—distances meaningless, synthetic samples off manifold. Use feature selection first.
👉 Try this: Set ratio to 1:100 (extreme imbalance). SMOTE at high dimensions creates samples far from true manifold. Compare with Class Weight (no synthetic samples needed, just penalize minority errors more).
Threshold tuning: Instead of \(P(y=1) > 0.5\), use \(P(y=1) > 0.3\) to raise recall. Trade precision for recall based on business cost.
Production rule: Monitor minority class recall separately from accuracy. Use stratified split/K-fold. Document imbalance ratio. Choose class weights based on BUSINESS cost, not frequency.

                from sklearn.datasets import make_classification
                import numpy as np
                
                # Imbalanced dataset: 950 vs 50
                X, y = make_classification(n_samples=1000, weights=[0.95],
                                           n_features=10, random_state=42)
                print(f"Class counts: {np.bincount(y)}")  # [950, 50]
                
                # Class weights (no data modification needed)
                clf = RandomForestClassifier(class_weight='balanced', random_state=42)
                clf.fit(X, y)
                # 'balanced' sets w_i = n_samples / (n_classes * n_i)

class_weight='balanced' auto-adjusts weights inversely proportional to class frequency.

5. Train/Validation/Test Split Strategies

Different split strategies suit different data types and problems.

Interview Tips: Train/Validation/Test Split Strategies

What it is: Train = fit params. Validation = tune hyperparameters. Test = final eval, touch only once. Split strategy depends on data type (i.i.d. vs time-series vs imbalanced).

Random Split
70/15/15 or 60/20/20. For i.i.d. data. Assumes independent samples. Unbiased but high variance on small data.
Time-Based Split
Train before t, val [t, t+Δ], test after t+Δ. For time-series. No temporal leakage. Realistic deployment sim.
Stratified Split
Maintain class proportions in each fold. For imbalanced data. Avoids high-variance estimates.
K-Fold CV
k=5 or 10. Partition into k subsets, train k-1, test 1, average. Reduces variance, expensive compute.
When to Use
Random Split (70/15/15)
i.i.d. tabular data, sufficient samples (> 10K). E.g., customer churn prediction, house price regression. Fast, simple. High variance on small data -- use K-Fold instead if < 5K samples.
Time-Based Split
Any temporal / sequential data. Stock prediction, demand forecasting, user behavior over time. Train on past, validate on near-future, test on far-future. Prevents temporal leakage. Mandatory for production realism.
Stratified K-Fold CV
Small datasets (< 5K) or imbalanced classes. E.g., rare disease classification with 200 positive samples. Preserves class ratios in each fold. k=5 for speed, k=10 for lower variance. Use nested CV if also tuning hyperparameters.
Report validation or test performance? Test only. Validation is for dev; test is for stakeholders. Reporting validation = optimistic bias (you picked best on val).
👉 Try this: Click K-Fold CV button and watch fold boundaries. Observe how each fold becomes validation once while others train. Now try Time-Based for temporal data—notice past/future separation preventing leakage. Stratified keeps class balance in each fold.
Why not reuse same validation split? You implicitly overfit to it via hyperparameter selection. Use multiple splits or nested CV.
👉 Try this: Random Split is fast but high variance on small data. K-Fold (k=5 or 10) uses data efficiently. Time-Based mandates past→future for realistic deployment simulation. Which do you need depends on data type and size.
Why time-based split for time-series? Adjacent points correlated (non-i.i.d.). Random split mixes past/future = temporal leakage. Time-based respects causality.
👉 Try this: Click Random Split on the canvas—data shuffles arbitrarily. Now switch to Time-Based—clean chronological order. See the difference? For stock prices or sensor data, random split is a fantasy that doesn't match production.
Critical rule: Fit preprocessing (scaler, encoder) on TRAIN only. Apply to val/test using train params. Prevents leakage.
Production rule: Never touch test set during dev. Document which samples in which set. For limited data: use nested CV (outer=eval, inner=tuning). 
from sklearn.model_selection import StratifiedKFold, cross_val_score
                from sklearn.linear_model import LogisticRegression
                import numpy as np
                
                X = np.random.randn(200, 5)
                y = np.array([0]*180 + [1]*20)  # imbalanced
                
                skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
                clf = LogisticRegression()
                scores = cross_val_score(clf, X, y, cv=skf, scoring='f1')
                print(f"F1 per fold: {scores.round(3)}")
                print(f"Mean F1: {scores.mean():.3f} +/- {scores.std():.3f}")

StratifiedKFold preserves class proportions in each fold; critical for imbalanced data.

6. Data Leakage

Leakage Type

Interview Tips: Data Leakage

What it is: Test info used in training → optimistic offline metrics, terrible online. Model has 95% test accuracy but 40% production (massive gap). Leakage breaks independence assumption.

Classic signs: 95% offline but 40% online. Large train-test gap. Feature works offline but not online. Check preprocessing, timestamps, feature definitions.
Train-Test Contamination
Fit scaler on FULL data (before split), then apply to train/val/test. Test data influenced scaler fit = leakage. Fix: fit on train only.
Temporal Leakage
Use future data to predict past (e.g., next month sales). Random split mixes past/future. Use time-based split to detect.
Feature Leakage
Feature depends on target or only available at train time. E.g., "is_fraud_detected" to predict fraud, or customer ID encoding behavior.
Domain Leakage
Proxy variables (e.g., collection date) only available at training. Correlates with outcome due to distribution shift.
When to Use
Leakage Audit Checklist
Run before every model deployment. Check: (1) preprocessing fit on train only? (2) No future data in features? (3) No target-derived features? (4) Offline-online gap < 5%? If any fail, investigate before shipping.
Suspect Leakage When...
Test AUC > 0.95 on first try (too good to be true). Offline metrics far exceed online. One feature dominates importance (may encode target). Performance drops when retrained on new data window.
Prevention Strategy
Time-based splits for temporal data. Pipeline unit tests (no timestamp overlap). Feature availability audit (is this feature available at inference time?). Shadow mode deployment before launch to compare offline vs online.
Using month M to predict month M+1 = valid. Using month M to predict month M where M ∈ test = temporal leakage.
👉 Try this: Click the Temporal button to visualize temporal leakage. Notice how random mixing of past/future breaks causality. Switch to Clean—future info is unavailable at prediction time. This is why time-based splits prevent leakage.
Can I detect leakage from train/test gap? No, unreliable. Large gaps suggest it, but can be subtle. Audit pipeline, check timestamps, verify features.
👉 Try this: Cycle through Clean, Train-Test, Feature Leakage. Each looks different on the canvas. Clean: independent sets. Feature Leakage: a single feature carries target info (post-hoc label). Always audit your pipeline, not just check metrics.
Leakage sources: Customer ID (encodes history). Aggregation including sample itself. Derived features including target. Feature selection on full data.
Prevention: Clear data pipeline with explicit train/val/test. Version control. Unit tests: verify no timestamp overlap, preprocessing on train only, test features inference-only.
Offline-online gap >5% = likely leakage. For temporal data: ALWAYS time-based split, verify timestamps ascending order.
👉 Try this: Set split to Time-Based, then look at the canvas: data flows left-to-right (past to future). Verify: does your preprocessing fit on train-only data? Are timestamps monotonically increasing? Check your code—leakage hides in these details.
from sklearn.pipeline import Pipeline
                from sklearn.preprocessing import StandardScaler
                from sklearn.linear_model import LogisticRegression
                from sklearn.model_selection import train_test_split
                import numpy as np
                
                X, y = np.random.randn(500, 4), (np.random.rand(500) > 0.5).astype(int)
                X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)
                
                # Pipeline prevents leakage: scaler fits on train only
                pipe = Pipeline([('scaler', StandardScaler()),
                                 ('clf', LogisticRegression())])
                pipe.fit(X_train, y_train)
                print(f"Test acc: {pipe.score(X_test, y_test):.3f}")

Pipeline ensures preprocessing fits only on training data during cross-validation.

7. Data Augmentation

Augmentation Type
Intensity
50%

Interview Tips: Data Augmentation

What it is: Generate new training samples from existing via transformations preserving label. Cheap way to increase training size. Only for training, not test/inference.

$$\tilde{x} = \lambda \cdot x_i + (1-\lambda) \cdot x_j, \quad \tilde{y} = \lambda \cdot y_i + (1-\lambda) \cdot y_j, \quad \lambda \sim \text{Beta}(\alpha, \alpha)$$
\(x_i, x_j\):Two randomly sampled training examples (e.g., images or feature vectors); can have same or different classes
\(\lambda\):Mixing weight sampled from \(\text{Beta}(\alpha, \alpha)\); \(\alpha\) typically 1.0; \(\lambda \in [0, 1]\) controls the blend ratio
\(\tilde{x}\):Augmented feature; linear combination of two samples; new synthetic example
\(y_i, y_j\):Target labels (0 or 1 for binary, or class indices for multiclass)
\(\tilde{y}\):Mixed target label; soft target \(\in [0, 1]\); represents uncertain/interpolated label between two classes
\(\text{Beta}(\alpha, \alpha)\):Symmetric Beta distribution; \(\alpha=1\) gives uniform \([0,1]\), \(\alpha>1\) concentrates near 0.5 (balanced mixing)
$$\text{CutMix: patch one image with region from another; aggressive + label-preserving}$$
CutMix mechanism:Cuts a random rectangular patch from one image, pastes it into another image at a random location
Label handling:Label proportional to patch area ratio; if 30% of image from class A, label = \(0.3 \cdot y_A + 0.7 \cdot y_B\) (soft label)
Images: rotate, flip, crop, color jitter. Text: back-translate, synonym replace. Tabular: SMOTE better. Augment train only, never test.
Geometric (Images)
Rotate, translate, scale, shear, flip, crop. Preserves label if realistic (horiz flip yes, vert flip no for faces).
Color (Images)
Brightness, contrast, saturation, hue jitter in HSV. RandAugment auto-selects random ops.
Text
Back-translation (lang→lang), synonym replace, deletion, insertion. Creates paraphrases.
Tabular
Hard to design, risks violating constraints. Use SMOTE instead for imbalance.
When to Use
Geometric / Color (Images)
Image classification, object detection, segmentation. Flip, rotate, crop, color jitter. E.g., medical imaging (rotate X-rays), autonomous driving (brightness changes). Use RandAugment for auto-selected ops.
MixUp / CutMix
When you need smoother decision boundaries and better calibration. Image classification (CIFAR, ImageNet). Creates soft labels that act as regularization. CutMix better for localization tasks (forces model to use all regions).
Text Augmentation
NLP with limited labeled data. Back-translation for paraphrasing, synonym replacement for diversity, random deletion for robustness. E.g., sentiment analysis with 1K labeled reviews. Validate augmented text preserves original label.
Augment test set? No—test must be fixed, representative of deployment. Augmenting test inflates estimates.
👉 Try this: Drag Intensity slider from 10% to 100% and watch augmentations strengthen. At 10%, subtle rotations; at 100%, aggressive warping. Find the sweet spot where augmentations remain label-preserving (flip yes, invert no). Only augment training data.
Augmentation strength = hyperparameter. Too weak = no benefit. Too strong = distort labels (180° rotation changes 6→9). Cross-validate.
👉 Try this: Toggle Image vs Text vs MixUp. Each has different intensity semantics. For images, 50% intensity ≈ mild rotation; for text, same intensity ≈ semantic-preserving paraphrasing. MixUp blends inputs—watch it create soft labels.
Production rule: Augment train only, never inference. Set random seed for reproducibility. Document strength. Changes to augmentation = model change. For medical: validate with domain experts.
Complementary to: Regularization (both reduce overfitting). Transfer learning (augmented pre-training helps). Can target minority class for imbalance. 
import numpy as np
                
                def mixup(X, y, alpha=1.0):
                    lam = np.random.beta(alpha, alpha, size=len(X))
                    idx = np.random.permutation(len(X))
                    X_mix = lam.reshape(-1,1) * X + (1-lam.reshape(-1,1)) * X[idx]
                    y_mix = lam * y + (1-lam) * y[idx]
                    return X_mix, y_mix
                
                X = np.random.randn(100, 5)
                y = np.random.randint(0, 2, 100).astype(float)
                X_aug, y_aug = mixup(X, y, alpha=0.4)
                print(f"Soft labels range: [{y_aug.min():.2f}, {y_aug.max():.2f}]")

MixUp creates soft labels via linear interpolation; alpha controls mixing strength.

8. Temporal Features

Feature Type
Period
24 hours

Interview Tips: Temporal Features and Time-Series Representation

What it is: Encode time-indexed data into ML-friendly features. Adjacent time steps correlated (non-i.i.d.). Need cyclical, lag, rolling features. Avoid lookahead bias.

$$\text{hour\_sin} = \sin\!\left(\frac{2\pi \cdot \text{hour}}{24}\right), \quad \text{hour\_cos} = \cos\!\left(\frac{2\pi \cdot \text{hour}}{24}\right) \;\rightarrow\; \text{circularity preserved}$$
hour:Hour-of-day value, \(\in \{0, 1, \ldots, 23\}\)
\(2\pi\):Full circle in radians; maps period to complete rotation
hour/24:Normalized hour to \([0, 1)\) fraction of day
hour_sin:Sine component \(\in [-1, 1]\); represents y-coordinate on unit circle
hour_cos:Cosine component \(\in [-1, 1]\); represents x-coordinate on unit circle; together preserve circularity (hour 23 \(\approx\) hour 0)
$$\text{Lag features: } X_{t-1}, X_{t-2}, \ldots, X_{t-k} \quad | \quad MA_t = \frac{1}{L}\sum_{i=0}^{L-1} X_{t-i}$$
\(X_t\):Feature value at current time step \(t\)
\(X_{t-i}\):Feature value \(i\) steps in the past (lag \(i\)); \(i \in \{1, 2, \ldots, k\}\)
\(k\):Maximum lag; number of past time steps to use as features; chosen via ACF plots or domain knowledge
\(MA_t\):Moving average at time \(t\) computed over last \(L\) time steps (e.g., \(L=6\)); smooths noise, captures trend
\(L\):Window size for rolling statistic; \(L=24\) for daily seasonality (hourly data), \(L=168\) for weekly (hourly data)
Hour 23 close to hour 0 (one-hot fails). Use sin/cos. Lag-24 = 1 day seasonality, lag-168 = 1 week (for hourly data).
Cyclical Features
Hour, day-of-week, month (periodic). Use sin/cos encoding, NOT one-hot. Preserves circularity (23→0 nearby).
Lag Features
Past values \(X_{t-k}\). Capture autoregressive patterns. Use ACF plots to find significant lags. Too many = noise.
Rolling Statistics
Moving average (smooth), std (volatility), sum (accumulation). Capture trend + seasonality.
Date Features
day_of_week, is_weekend, is_holiday, month, quarter. Domain effects (sales↑ weekends, ↓ holidays).
When to Use
Cyclical Encoding (sin/cos)
Periodic features: hour-of-day, day-of-week, month-of-year, wind direction. E.g., ride-share demand prediction (hour 23 near hour 0). Always use sin+cos pair together -- one alone loses phase information.
Lag Features
Autoregressive patterns where recent past predicts future. E.g., stock price (lag-1), daily sales (lag-7 for weekly pattern), energy consumption (lag-24 for daily cycle). Select lags via ACF/PACF plots.
Rolling Statistics
Capturing trends and volatility. Moving average (smooth noise), rolling std (detect regime changes), rolling sum (accumulation). E.g., 7-day rolling avg for COVID case trends, 30-day rolling volume for stock trading signals.
One-hot hour-of-day? Creates 24 sparse columns, treats 23 far from 0 (wrong). Sin/cos naturally wraps circle.
👉 Try this: Select Cyclical feature type and watch the sin/cos encode hours on a circle. Hour 23 and Hour 0 are neighbors (nearby in Euclidean space). Lag-24 captures daily seasonality (tomorrow looks like today). Rolling average smooths noise—watch standard deviation shrink.
Rolling average lookahead bias? Only if include future samples. Lag-based (past k) = safe. Centered (past+future) = lookahead.
👉 Try this: Toggle between Lag and Rolling on the canvas. Lag uses past data only (safe). Centered rolling (past+future) leaks future info (production breaks). Adjust Period slider to capture different seasonality windows.
Production rule: NEVER shuffle time-series. Use time-based split mandatory. Compute rolling stats on train only (no leakage). At inference: ensure sufficient historical data for lags.
Stationarity: If non-stationary (trend), difference: \(X_t - X_{t-1}\). For seasonality: \(X_t - X_{t-s}\). Many models assume stationarity.
Include target lag-1 as feature? Yes if available at train. Production: use \(X_{t-1}\) predict \(Y_t\) (not \(Y_{t-1}\), which unavailable).
👉 Try this: Build a time-series model: use X(t-1) to predict Y(t) (previous input predicts current target). Never use Y(t-1) as input—it's unavailable at deployment. Lag-1 often most predictive; higher lags add noise.

Interview Tips: Weak Supervision and Programmatic Labeling

What it is: Label data via noisy heuristics (labeling functions) instead of manual annotation. Trade annotation cost (low) for label quality (imperfect). Snorkel learns LF accuracies, downweights bad ones.

Labeling function = heuristic rule mapping sample \(\rightarrow\) label (or abstain). Snorkel combines multiple LFs via label model learning \(P(Y \mid \text{LF outputs})\).
Label model learning: Learns (1) accuracy of each LF, (2) correlations between LFs. Outputs \(P(Y \mid L_1, \ldots, L_k)\). High-confidence samples become train data.
LF requirements: Better than random (>50% accuracy). Independent errors ideal. Abstain on uncertain samples. 3-10 well-designed LFs typical. Diminishing returns >20.
When to Use
Weak Supervision (Snorkel)
Tons of unlabeled data + domain experts who can write heuristics. E.g., labeling 1M medical notes for diagnosis, classifying 500K support tickets. 5-10 labeling functions from domain rules, regex patterns, knowledge bases. Cost: 10x cheaper than manual labeling.
Active Learning Instead
When you have budget for some human labels and want max label efficiency. E.g., 10K images, budget for 500 labels. Model selects most uncertain/informative samples for human review. Complements weak supervision (seed with weak labels, refine with active learning).
Manual Labeling Instead
Small datasets (< 1K), high-stakes domains (legal, medical diagnosis), or when heuristics cannot capture label semantics. E.g., annotating radiology images requires trained physicians. Use when label quality matters more than quantity.
Can weak supervision replace manual labeling? Dramatically reduces burden but not entirely. Need small labeled dev set for LF accuracy estimation. Shines at 1000s-millions unlabeled samples.
👉 Try this: Drag # Labeling Functions slider to 2—simple but coverage limited. Drag to 10—coverage better but correlations complex. Adjust Average Accuracy from 50% to 95%—Snorkel learns to upweight good LFs and ignore bad ones. Design 5-7 good LFs, not 20 poor ones.
Bad LF (40% accuracy)? Snorkel learns to ignore it automatically. LFs can be arbitrary accuracy—learns to upweight good, downweight bad.
👉 Try this: Set Average Accuracy to 50% (barely better than random coin flip). Snorkel automatically downweights these. Now 95%—these dominate. Mix them: some 60%, some 85%. Watch the label model blend them intelligently. This is the power—heterogeneous LF quality accepted.
Label leakage trap: if LF uses info unavailable at inference, downstream model fails in production. E.g., use "fraud" keyword to label but keyword only post-hoc.
👉 Try this: LF using "fraud_detected" keyword at training time seems great (high accuracy). But in production, you don't have this label yet! Every LF must use inference-available features only. Audit: will this LF work in production?
Design process: Domain experts → heuristics. Small LF set, measure accuracy on dev set. Combine + estimate label model. Train downstream. Iterate.
Production rule: Document + version control LFs (changing 1 LF = model change). Monitor coverage (if LF labels 1%, contributes little). Retrain when new LFs/data.
Complementary to: Active learning (selective human labeling). Semi-supervised (Snorkel seeds unlabeled). Multi-task (shared robust representations). Pre-training (more robust to noise).
