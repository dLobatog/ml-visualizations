Tokenization: Byte Pair Encoding (BPE)

Click 'Next Merge' repeatedly and watch vocabulary grow from characters to common subwords. Notice how frequent patterns like 'the', 'ing', 'tion' emerge early—that's BPE finding the most common byte pairs. Adjust Target Vocab Size to see how final vocabulary changes.

300
Key Insight: BPE solves the vocabulary problem — how to represent any text with a fixed-size token set, balancing sequence length vs. vocabulary size.
The problem:Character-level: vocab = 256 (tiny), but sequences are very long (slow). Word-level: vocab = 100K+ (huge embedding table), can't handle typos or new words ("ChatGPT" = unknown).
BPE solution:Start with 256 byte-level tokens. Find the most frequent adjacent pair (e.g., "t"+"h" → "th"). Merge it into one new token. Repeat until vocab reaches target size (32K–100K).
Why it works:Common words become single tokens ("the" = 1 token). Rare words decompose into subwords ("unforgettable" = "un" + "forget" + "table"). New words always representable by falling back to bytes.
Vocab size tradeoff:Larger vocab = shorter sequences (faster inference) but bigger embedding table (more memory). GPT-4 uses ~100K tokens. LLaMA uses ~32K.
When to Use
BPE (Byte Pair Encoding)
Best for general-purpose LLMs and multilingual models. Used by GPT-4, LLaMA, and most modern models. Handles unknown words gracefully by decomposing to subwords. Choose when you need a single tokenizer across many languages or domains.
WordPiece
Preferred for encoder models like BERT and electra-style architectures. Uses likelihood-based merging instead of frequency. Choose for classification, NER, or sentence-level tasks where subword consistency matters more than generation quality.
SentencePiece / Unigram
Language-agnostic, works directly on raw text without pre-tokenization. Used by T5, mBART, and XLNet. Choose when building multilingual systems or when you cannot assume whitespace-delimited words (e.g., Chinese, Japanese).
"Why not just use words as tokens?" → Unknown words break the model. "ChatGPT" isn't in any dictionary. BPE decomposes it to known subwords. Also: multilingual models need subword tokenization to share tokens across languages.
BPE = greedy compression. Most frequent pair → merge → repeat. Common things get short codes, rare things decompose. Like zip compression for language.

Self-Attention: Q, K, V Computation

1.0
Key Insight: Self-attention is the core mechanism enabling Transformers to learn long-range dependencies.
$$\text{Self-Attention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$
\(Q\) (Query):Query matrix, shape \((\text{seq\_len}, d_k)\). "What am I looking for?" Learned projection of input.
\(K\) (Key):Key matrix, shape \((\text{seq\_len}, d_k)\). "What does each position represent?" Compared against queries to determine relevance.
\(V\) (Value):Value matrix, shape \((\text{seq\_len}, d_v)\). "What information to extract?" Weighted and aggregated based on attention scores.
\(d_k\):Dimension of keys (usually 64). Affects scaling factor and model capacity.
\(d_v\):Dimension of values (usually 64). Output dimension per head. Total output = \(\text{num\_heads} \times d_v\).
\(QK^T\):Attention logits, shape (seq_len, seq_len). Measures similarity between each query and all keys. Unnormalized scores.
\(1/\sqrt{d_k}\):Scaling factor prevents attention weights from becoming too confident. Variance of \(QK^T\) is \(O(d_k)\), scaling keeps gradient \(O(1)\).
softmax(...):Converts scaled scores to probability distribution. Each row sums to 1. Determines how much each position attends to others.
softmax(...) · V:Weighted sum of values. Output shape: (seq_len, d_v). Each position gets weighted combination of all value vectors.
Process: 1. Project input to Q, K, V. 2. Compute attention weights: softmax(QK^T). 3. Weight values: apply weights to V. 4. Concatenate heads.
Multi-head attention: h heads with different projections. Each head learns different attention patterns. Output: concat of all heads.
Scaling by √d_k: Prevents softmax from becoming too peaked. Variance of \(QK^T\) is \(O(d_k)\), so scale gradients stay \(O(1)\).
Masking in decoder: Prevent attending to future tokens. Set attention weights to -∞ for future positions before softmax.
When to Use
Full Self-Attention
Standard for most Transformer models (GPT, BERT, LLaMA). Use when sequence length is manageable (under 8K tokens) and you need maximum quality. Every token attends to every other token. E.g., text generation, summarization, translation.
Sparse / Local Attention
Use for very long sequences (documents, books, genomics) where O(n^2) is prohibitive. Longformer uses local + global patterns, BigBird adds random attention. E.g., processing 16K+ token documents, long-form QA.
Cross-Attention
Use when the model needs to attend to a different sequence (encoder-decoder). Queries come from decoder, keys/values from encoder. E.g., machine translation (attend to source), image captioning (attend to image features), RAG (attend to retrieved docs).

Notice how attention becomes sharp (one token dominates) vs spread (distributed across many). This is temperature controlling softmax concentration.
Why Q, K, V not just use input directly? Learned projections provide flexibility. Different heads can attend differently.
Self-attention = compare all positions with each other. Learn what to pay attention to. Q asks, K answers, V provides info.
import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    d_k = Q.size(-1)
    scores = Q @ K.transpose(-2, -1) / d_k**0.5
    if mask is not None:
        scores = scores.masked_fill(mask == 0, -1e9)
    weights = F.softmax(scores, dim=-1)
    return weights @ V, weights

Core attention computation used in every transformer. Returns both output and attention weights for visualization.

Key Insight: Layer normalization and position-wise feed-forward networks stabilize Transformer training.
$$\text{LayerNorm}(x) = \gamma \cdot \frac{x - \mu}{\sqrt{\sigma^2 + \epsilon}} + \beta$$
\(x\):Input to layer normalization. Shape: (batch_size, seq_len, hidden_dim) or (batch_size, hidden_dim)
\(\mu\):Mean of x computed per example (across the feature dimension). Shape: (batch_size, seq_len) or (batch_size,)
\(\sigma^2\):Variance of x per example (across features). Measures spread of activation values.
\(\epsilon\):Small constant (e.g., 1e-5 or 1e-6) for numerical stability. Prevents division by zero when variance is very small.
\(\frac{x - \mu}{\sqrt{\sigma^2 + \epsilon}}\):Standardization step. Shifts to mean 0, scales to variance 1. Normalizes across features for each example.
\(\gamma\) (gamma):Learnable scale parameter. Shape: (hidden_dim,). Allows network to scale normalized activations. Learned during training.
\(\beta\) (beta):Learnable shift parameter. Shape: (hidden_dim,). Allows network to shift normalized activations. Learned during training.
\(\gamma \cdot (\ldots) + \beta\):Affine transformation. Restores model expressiveness after normalization. If \(\gamma=1, \beta=0\) everywhere, recovers standard normalization.
Why layer norm (not batch norm): Batch norm needs batch dimension. Transformers process variable-length sequences. LayerNorm normalizes per instance.
Position-wise FFN: Applied to each position independently. Two linear layers with ReLU: FFN(x) = max(0, xW₁ + b₁)W₂ + b₂.
FFN dimensions: First layer expands (4x hidden size). Second layer projects back. Adds expressiveness. Most parameters in Transformers.
Residual connections: Add input to output of each sublayer. Enables training of deep Transformers. Stabilizes gradients.
When to Use
Pre-Norm (GPT-2+, LLaMA)
Apply LayerNorm before attention and FFN sublayers. Stabilizes training for very deep models (50+ layers). Easier to train without learning rate warmup. Use for most modern LLM architectures. E.g., GPT-3, LLaMA, PaLM.
Post-Norm (Original Transformer)
Apply LayerNorm after residual addition. Can achieve slightly better final performance but harder to train (needs careful warmup). Use when training stability is not a concern and you want to squeeze out marginal gains. E.g., original BERT, T5.
RMSNorm (LLaMA 2/3)
Simplified LayerNorm that only normalizes by root mean square (no mean centering). 10-15% faster than LayerNorm with comparable quality. Use in large-scale training where compute savings matter. E.g., LLaMA 2, Mistral, Gemma.

Transformer depth comes from stacking. Each layer adds computational power but also parameters and memory.
LayerNorm normalizes each example independently. FFN expands then contracts. Residual connections = skip highways.

Positional Encodings: Sinusoidal & RoPE

Click the 'Sinusoidal (Original)' and 'RoPE (LLaMA/GPT-NeoX)' tabs to compare two approaches. Watch how each encodes position information differently. Slide Max Position to see how encoding patterns extend to longer sequences. RoPE produces tighter rotations—notice the difference in periodicity.

Key Insight: Tokenization and embedding convert discrete text into continuous vectors for neural processing.
Tokenization: Split text into tokens (words, subwords, or characters). Subword tokenization (BPE, WordPiece) balances vocabulary size vs sequence length.
Token embeddings: Each token → learned embedding vector. Embedding matrix: vocab_size × d_model. Randomly initialized, learned during training.
Positional embeddings: Add position information (sinusoidal or learned). Allows network to use word order. Token emb + position emb.
Special tokens: [CLS] = classification (BERT). [SEP] = separator. [PAD] = padding. [UNK] = unknown word. Handle these during tokenization.
When to Use
Sinusoidal (Fixed)
Deterministic sin/cos encodings at different frequencies. No learned parameters. Generalizes to unseen sequence lengths in theory. Use for smaller models or when you want zero extra parameters. E.g., original Transformer, some encoder-only models.
RoPE (Rotary Position Embeddings)
Encodes relative position by rotating Q and K vectors. Extrapolates better to longer sequences than trained on. Industry standard for modern LLMs. Use for any autoregressive model needing long-context support. E.g., LLaMA, Mistral, GPT-NeoX, Qwen.
Learned Absolute Embeddings
Trainable embedding per position. Simple but cannot generalize beyond max training length. Use when sequence length is fixed and known at training time. E.g., GPT-2 (1024 positions), BERT (512 positions), ViT (fixed patch count).

Notice RoPE maintains better distance preservation at long sequences—this is why modern models prefer RoPE. Watch the rotation pattern change.
Subword tokenization is crucial for handling rare words and multilingual text. Understand vocab size trade-offs.
Tokenization + embedding = convert text to dense vectors. Positions matter. Special tokens provide structure.
import torch, math

def sinusoidal_encoding(seq_len, d_model):
    pe = torch.zeros(seq_len, d_model)
    pos = torch.arange(seq_len).unsqueeze(1).float()
    div = torch.exp(torch.arange(0, d_model, 2).float() * -(math.log(10000.0) / d_model))
    pe[:, 0::2] = torch.sin(pos * div)
    pe[:, 1::2] = torch.cos(pos * div)
    return pe  # shape: (seq_len, d_model)

pe = sinusoidal_encoding(128, 512)  # 128 positions, 512 dims

Generates fixed positional encodings as described in 'Attention Is All You Need'. Added to token embeddings before the transformer.

KV-Cache: Efficient Autoregressive Inference

Click 'Generate Next Token' and watch the KV-cache expand. Notice how the cache stores computed keys and values from all previous tokens. Toggle 'Without Cache' to see the full recomputation—watch how much slower it becomes. This shows why caching is critical for inference speed.

Key Insight: Language model pretraining learns general linguistic patterns from large unlabeled text via next-token prediction.
$$P(x_{t+1} | x_1, \ldots, x_t) = \text{softmax}(W \cdot h_t + b)$$
\(P(x_{t+1} | x_1, \ldots, x_t)\):Conditional probability distribution over next token \(x_{t+1}\), given all previous tokens. Output shape: (vocab_size,). Sum to 1.
\(x_{t+1}\):Next token to predict. Element of vocabulary (0 to vocab_size-1). Loss = cross-entropy between true next token and this distribution.
\(x_1, \ldots, x_t\):All previous tokens (context). Model processes entire sequence, attends to all previous positions via self-attention masks.
\(h_t\):Hidden representation after processing sequence up to position t. Shape: (batch_size, hidden_dim). Contains context from all previous tokens.
W:Output projection matrix. Shape: (hidden_dim, vocab_size). Projects hidden state to logit space (unscaled scores for each token).
b:Output bias. Shape: (vocab_size,). Learned scalar offset per token.
W · h_t + b:Logits (unnormalized scores for each vocab token). Shape: (vocab_size,). No probability constraints yet.
softmax(...):Converts logits to valid probability distribution. Exponentiates, normalizes. Output: (vocab_size,) with elements in [0,1], sum=1.
Causal language modeling: Predict next token given previous tokens. Masks future positions. Loss = cross-entropy on next token prediction.
Scale matters: Larger models → better performance. 7B → 13B → 70B. More data helps too. Chinchilla scaling: match model size and data.
Emergent abilities: Chain-of-thought reasoning, few-shot learning appear at scale. Model size threshold matters.
Compute-optimal training: Optimal allocation: train to model convergence (2T tokens, where T = model parameters). LLaMA follows this principle.
When to Use
KV-Cache (Always for Inference)
Essential for any autoregressive generation. Caches computed keys and values from previous tokens so each new token only computes one new Q, K, V instead of recomputing the full sequence. Use always during inference. Without it, generating 1000 tokens is ~500x slower.
Grouped-Query Attention (GQA)
Shares KV heads across multiple query heads, reducing KV-cache memory by 4-8x. Use when deploying large models with long contexts where cache memory is the bottleneck. E.g., LLaMA 2 70B uses 8 KV heads for 64 query heads, cutting cache from 20GB to 2.5GB at 32K context.
Multi-Query Attention (MQA)
Extreme version: all query heads share one KV head. Maximum cache reduction but slight quality loss. Use for latency-critical applications like real-time chat or code completion. E.g., PaLM, Falcon, StarCoder use MQA for faster serving.

# Pseudocode for KV-cache in autoregressive generation
past_key, past_value = None, None

for step in range(max_new_tokens):
    if past_key is not None:
        q = project_q(x[:, -1:, :])          # only last token
        k = torch.cat([past_key, project_k(x[:, -1:, :])], dim=1)
        v = torch.cat([past_value, project_v(x[:, -1:, :])], dim=1)
    else:
        q, k, v = project_q(x), project_k(x), project_v(x)
    past_key, past_value = k, v
    out = scaled_dot_product_attention(q, k, v)

Shows how KV-cache avoids recomputing keys/values for previous tokens, reducing generation from O(n^2) to O(n) per step.

Transformer Block Architecture

Click 'Step Through' to follow data flow through each sub-layer. Watch the signal pass through LayerNorm → Attention → residual add, then LayerNorm → FFN → residual add. Toggle the 'Visualize Residuals' checkbox to highlight skip connections. These shortcuts are why deep Transformers can train effectively.

Key Insight: Instruction tuning and RLHF align models with human preferences and make them helpful, harmless, honest.
Instruction tuning: Fine-tune pretrained LM on (instruction, response) pairs. Teaches model to follow directions. Simple but effective.
RLHF (Reinforcement Learning from Human Feedback): 1. Collect human preference data (which output better?). 2. Train reward model. 3. Use RL to optimize policy (LM) against reward.
Reward model training: Model learns to predict human preferences. Trained on paired outputs (better, worse). Loss = negative log likelihood of preference.
RL loss: PPO or similar policy gradient. Maximize: \(E[\text{reward}] - D_{KL}(\pi_{\text{new}} \| \pi_{\text{old}})\). KL term prevents diverging from pretrained model.
When to Use
Instruction Tuning (SFT)
Simplest alignment method. Fine-tune on curated (instruction, response) pairs. Use as the first step after pretraining, or when you have high-quality demonstration data. Sufficient for many production use cases. E.g., Alpaca, Vicuna, domain-specific chatbots.
RLHF with PPO
Full pipeline: SFT, then train reward model on human preferences, then optimize with PPO. Use when you need nuanced alignment (safety, helpfulness, tone) beyond what SFT alone achieves. Complex but powerful. E.g., ChatGPT, Claude, Gemini.
DPO (Direct Preference Optimization)
Skips the reward model entirely. Learns directly from preference pairs using a classification-like loss. Simpler, faster, more stable training than PPO. Use when you have preference data but want to avoid RL complexity. E.g., Zephyr, many open-source alignment efforts.
👉 Try this: Click "Step Through" to animate data flow. Pause at each stage: after Attention, watch residual add. After FFN, watch residual add again. These skips are critical—they let gradients flow back unobstructed. Toggle "Remove Residuals" to see training collapse. This is why Transformers can be 100+ layers deep.
RLHF is expensive (human feedback needed) but crucial for alignment. Alternative: DPO (direct preference optimization, no reward model).
Instruction tuning = teach format. RLHF = optimize for human preferences. Alignment = make models helpful, safe, honest.
import torch.nn as nn

class TransformerBlock(nn.Module):
    def __init__(self, d_model=512, nhead=8, d_ff=2048, dropout=0.1):
        super().__init__()
        self.attn = nn.MultiheadAttention(d_model, nhead, batch_first=True)
        self.ff = nn.Sequential(nn.Linear(d_model, d_ff), nn.GELU(), nn.Linear(d_ff, d_model))
        self.ln1 = nn.LayerNorm(d_model)
        self.ln2 = nn.LayerNorm(d_model)
        self.drop = nn.Dropout(dropout)

    def forward(self, x):
        x = x + self.drop(self.attn(self.ln1(x), self.ln1(x), self.ln1(x))[0])
        x = x + self.drop(self.ff(self.ln2(x)))
        return x

Modern pre-LayerNorm transformer block with GELU activation. This is the building block stacked N times in GPT-style models.

Key Insight: In-context learning and prompt engineering guide LLMs to perform tasks without parameter updates.
Few-shot prompting: Include examples in context. Model learns from examples + pattern recognition. No gradient updates, just completion generation.
Chain-of-thought prompting: "Let's think step by step". Show intermediate reasoning steps in examples. Model generates steps before final answer.
Prompt structure: System message (role/instructions) + context/examples + query. Temperature (randomness) affects output diversity.
Why it works: Pretraining on diverse data teaches patterns. Transformers learn what comes next. Context updates hidden state toward task.
When to Use
Zero-Shot Prompting
Give the model a task description with no examples. Use when the task is straightforward and the model is large enough (70B+). Fastest to iterate. E.g., "Translate this to French:", simple classification, formatting tasks.
Few-Shot Prompting
Include 3-8 examples in the prompt. Use when the task has a specific format the model needs to learn or when zero-shot quality is insufficient. No training needed. E.g., custom entity extraction, structured output generation, style matching.
Chain-of-Thought (CoT)
Prompt the model to reason step-by-step before answering. Use for math, logic, multi-step reasoning, or any task where intermediate steps improve accuracy. Can combine with few-shot. E.g., word problems, code debugging, complex QA.

Key Insight: Quantization reduces model size and inference latency by using lower-precision numbers instead of float32.
Precision levels: float32 (32 bits). float16 (16 bits, 2x smaller). int8 (8 bits, 4x smaller). int4 (4 bits, 8x smaller).
Quantization methods: Post-training quantization: quantize after training. Quantization-aware training: simulate quantization during training.
Trade-offs: int4: 8x smaller, but quality loss. int8: ~4x smaller, minimal loss. Depends on layer sensitivity.
Mixed precision: Keep critical layers (embeddings, attention output) in higher precision. Quantize feed-forward networks heavily. Optimal quality/speed.
When to Use
FP16 / BF16 (Half Precision)
Standard for training and high-quality inference. 2x memory reduction vs FP32 with negligible quality loss. BF16 preferred for training (larger dynamic range). Use as your default. E.g., GPU training with mixed precision, production serving on A100/H100.
INT8 Quantization
4x memory reduction vs FP32. Minimal quality degradation (less than 1% on most benchmarks). Use for cost-effective production serving. Works well with post-training quantization (no retraining needed). E.g., serving 7B-13B models on consumer GPUs, API endpoints.
INT4 / GPTQ / AWQ
8x memory reduction vs FP32. Enables 70B models on a single 24GB GPU. Some quality loss (1-3%), especially on reasoning tasks. Use for local inference, edge deployment, or when GPU memory is severely constrained. E.g., running LLaMA 70B on RTX 4090, mobile deployment.

Key Insight: Scaling laws govern how model performance improves with more parameters, data, and compute.
$$\text{Loss}(N) \approx \frac{a}{N^{\alpha}} \quad \text{(model size scaling)}$$
\(\text{Loss}(N)\):Test loss when training with N parameters. Lower = better performance. Asymptotically approaches optimal loss.
\(N\):Number of model parameters (weights, biases). Measured in millions (M) or billions (B). Ranges from 1M to 400B+ for LLMs.
\(a\):Scaling constant (empirical fit). Depends on task, data quality, architecture. Fitted from experimental data.
\(\alpha\):Scaling exponent for model size, typically 0.07-0.1. Log-log slope. \(\alpha = 0.1\) means 10x params → ~2x loss reduction.
≈:Approximation sign. Empirical power law, not exact theory. Holds across wide range of N (millions to hundreds of billions).
$$\text{Loss}(D) \approx \frac{b}{D^{\beta}} \quad \text{(data scaling)}$$
\(\text{Loss}(D)\):Test loss when training with D tokens of data. Larger D → lower loss. Follows similar power law to model size.
\(D\):Number of training tokens (data quantity). For 70B parameter model, optimal \(D \approx 1.4\) trillion tokens (Chinchilla scaling).
\(b\):Scaling constant for data dimension. Empirically fitted. Typically larger coefficient than model scaling constant a.
\(\beta\):Scaling exponent for data, typically 0.1-0.2. \(\beta = 0.15\) means 10x more data → ~1.4x loss reduction. Slower than model scaling.
Observed exponents: \(\alpha \approx 0.07\text{-}0.1\) (model), \(\beta \approx 0.1\text{-}0.2\) (data). Both improve loss, but slowly (power law).
Chinchilla/Compute optimal: Optimal allocation: increase model and data together. ~20 tokens per parameter. Avoid: huge model on small data or vice versa.
Implications: Performance improves continuously with scale. No ceiling expected. Bigger models generalize better despite more capacity.
Practical impact: Predicting loss on held-out data. Choosing model size. Budget allocation (compute, parameters, data).
When to Use
Continuous Batching (vLLM, TGI)
Dynamically adds/removes requests from the batch as they complete. Increases GPU utilization from ~60% to 95%+. Use for any production LLM serving with variable-length requests. E.g., chatbot APIs, code completion services, any multi-user deployment.
PagedAttention (vLLM)
Manages KV-cache memory in non-contiguous blocks (like OS virtual memory). Eliminates fragmentation, supports 2-4x more concurrent requests. Use when serving many concurrent users with varying context lengths. E.g., high-throughput API serving, shared GPU clusters.
Speculative Decoding
Small draft model generates candidate tokens, large model verifies in parallel. 2-3x speedup for latency-sensitive applications with no quality loss (mathematically equivalent). Use when latency matters more than throughput. E.g., interactive chat, real-time code suggestions.

Watch GPU utilization jump from 60% to 95%. Then view "PagedAttention"—notice how variable-length blocks prevent memory waste compared to fixed allocations.
Scaling laws are empirical (fit from data) not proven. But remarkably consistent across orders of magnitude. Useful for planning.
Scaling laws = loss improves as power law with model/data size. Chinchilla = balance model and data. Bigger = better.

RAG: Retrieval-Augmented Generation

Key Insight: Knowledge distillation transfers knowledge from large teacher models to smaller student models.
Goal: Make smaller model mimic large model. Smaller = faster inference, deployable on edge devices.
KD training: Loss = \(\alpha \cdot \text{CE}(\text{student}, \text{hard\_labels}) + (1-\alpha) \cdot D_{KL}(\text{student\_logits}, \text{teacher\_logits})\). Soft targets from teacher.
Temperature scaling: Higher temperature softens probability distributions. Makes soft targets more informative. Typical: T=4.
Results: Student can match large part of teacher performance at much smaller size. 10x compression: ~5-10% quality loss.
When to Use
RAG (Retrieval-Augmented Generation)
Retrieve relevant documents at inference time and include them in the prompt. Use when knowledge must be up-to-date, domain-specific, or verifiable with citations. No retraining required. E.g., enterprise QA over internal docs, customer support, legal research, medical knowledge bases.
Fine-Tuning (Full or LoRA)
Update model weights on your domain data. Use when you need the model to learn new behaviors, formats, or domain-specific reasoning patterns that prompting cannot achieve. E.g., code generation in a proprietary language, medical report writing, structured data extraction.
Knowledge Distillation
Train a small student model to mimic a large teacher. Use when you need production deployment on edge devices or strict latency budgets. Trade quality for speed/cost. E.g., on-device autocomplete, mobile assistants, real-time classification where 70B is too slow.

Key Insight: Model merging and mixture of experts enable efficient scaling without increasing individual model size.
Model merging: Average weights of multiple models trained on different tasks. Single inference cost. Combines capabilities. Task vectors help alignment.
Mixture of Experts (MoE): Sparse expert networks. Only few experts active per token. Routes tokens to experts via learned gating. Massive scale with reasonable compute.
MoE benefits: 7B sparse MoE ≈ capability of dense 50B model. Load balancing critical (ensure all experts used).
Trade-offs: MoE: higher memory (all experts loaded), better scaling. Merging: simpler, but capacity limited by individual models.
When to Use
Dense Models
All parameters active for every token. Simpler to train, serve, and debug. Use when you have sufficient compute for your target size and want predictable behavior. Best for smaller deployments. E.g., LLaMA 7B-70B, GPT-4 (rumored dense), Mistral 7B for single-GPU serving.
Mixture of Experts (MoE)
Only 2 of 8 experts active per token. 4-8x more total parameters but same inference cost as a smaller dense model. Use when you need maximum capability at fixed inference budget. Requires more memory (all experts loaded). E.g., Mixtral 8x7B, Gemini, GPT-4 (rumored MoE).
Model Merging
Average or interpolate weights from multiple fine-tuned models. No additional training needed. Use when you want to combine capabilities from specialist models cheaply. E.g., merge a code model + chat model + math model into one generalist, quick prototyping of multi-skill models.
