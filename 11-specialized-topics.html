CLIP: Contrastive Language-Image Pre-training

Core Concept: CLIP learns a joint embedding space where matched image-text pairs are close together, mismatched pairs far apart. Enables zero-shot classification by comparing image embeddings to text class label embeddings.

$$\mathcal{L}_{\text{CLIP}} = -\log\frac{\exp(\text{sim}(I,T)/\tau)}{\sum_j \exp(\text{sim}(I,T_j)/\tau)}$$
L_CLIP:Contrastive loss; negative log-likelihood of matching image-text pair among negatives
\(\text{sim}(I,T)\):Cosine similarity between image embedding \(I\) and text embedding \(T\); \(\in [-1, 1]\)
\(\tau\):Temperature parameter \(> 0\); controls softmax sharpness. Lower \(\tau\) = sharper (harder negs), higher \(\tau\) = softer. Typical: 0.07
Σ_j:Sum over all N text embeddings in batch (1 positive + N-1 negatives)
\(\exp(\text{sim}(\cdot)/\tau)\):Numerator: similarity of positive pair, temperature-scaled
\(\sum_j \exp(\text{sim}(I,T_j)/\tau)\):Denominator: sum of all (positive + negatives), forms softmax partition function
Architecture:Image encoder (CNN/ViT) and text encoder (Transformer) produce same-dim embeddings for cosine similarity
InfoNCE Loss:N image-text pairs; each image's matched text is positive, N-1 others are negatives
Temperature \(\tau\):Controls distribution sharpness; higher \(\tau\) makes hard negatives easier to distinguish
Why does CLIP work zero-shot? Text embeddings naturally encode semantic information about classes—classifier emerges without seeing those classes during training.
👉 Try this: Give CLIP a class it never saw (e.g., "zebra" if trained on birds). It classifies zero-shot because text encoder learns semantic features. Temperature τ controls confidence—low τ = sharp predictions, high τ = uncertain. Typical τ=0.07.
CLIP = two encoders in a joint space. Align image-text pairs; text carries semantic labels. At test time, embed class names, find nearest image.

Key Gotchas: Domain shift hurts performance (fine-tune text encoder on target); Embedding collapse if not careful; Temperature tuning matters.

Production Practices: Pre-compute/cache image embeddings; Use FAISS for fast retrieval; Monitor embedding distribution for collapse; Combine with downstream classifier for specificity.

Vision Transformer: Pure Attention for Vision

Core Intuition: Treat image as sequence of patches; use pure self-attention instead of convolutions. Removes translation equivariance assumption of CNNs; learns all spatial relationships from data.

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right) V$$
\(Q\) (Query):Query matrix \(\in \mathbb{R}^{L \times d_k}\); what to attend to. L = sequence length, d_k = key dimension
\(K\) (Key):Key matrix \(\in \mathbb{R}^{L \times d_k}\); what can be attended to
\(V\) (Value):Value matrix \(\in \mathbb{R}^{L \times d_v}\); what information is aggregated; often d_v = d_k
\(QK^T\):Attention scores \(\in \mathbb{R}^{L \times L}\); (i,j) entry = similarity between query i and key j
\(\sqrt{d_k}\):Scaling factor; prevents attention scores from growing too large (variance stabilization). d_k = head dimension
softmax(...):Normalizes attention weights per query; sums to 1 across keys for each position
Attention(...)·V:Weighted sum of values; each position's output is mixture of all values, weighted by attention
Tokenization:224x224 image → 16x16 patches (196 total) → flatten to 768D via linear projection
[CLS] Token:Prepended learned token; its output used for classification (like NLP)
Positional Embeddings:Learned 1D embeddings added to preserve spatial info; can interpolate for different resolutions
ViT vs CNNs
Better scaling with compute/data; Long-range interactions; Needs more training data
Variable Size Images
Interpolate position embeddings or use relative position bias
ViT needs more data than CNNs because it has no convolutional inductive bias. It must learn spatial relationships purely from data.
👉 Try this: Set image size to 224×224 and patch size P=16 → 196 tokens. P=32 → 49 tokens. ViT is permutation-invariant (no built-in 2D structure), so it needs 4x data compared to ResNet. Pre-train on large datasets (ImageNet-21K, JFT-300M), then fine-tune efficiently.
ViT = image as text. Patches = tokens. [CLS] = special token like NLP. Attention = all patches talk to each other globally.

Key Gotchas: Requires large datasets; Throughput memory-bounded (batch size critical); Position embedding interpolation impacts performance.

Production Best Practices: Use relative position bias for generalization; ImageNet normalization critical; Interpolate pos embeddings for different resolutions; Batch size matters more than compute efficiency.

Core Problem: How to combine image and text representations? Different strategies trade off efficiency, information preservation, and computation.

Q_text:Query from text modality; what text attends to
K_image, V_image:Keys and values from image modality; what text can attend to and get from images
\(\sqrt{d_k}\):Scaling factor for stability; d_k = key dimension (often 64 per head)

Early fusion is efficient for deployment; late fusion maximizes accuracy; cross-attention balances both. Choose based on latency vs accuracy tradeoff.

Early = blend quickly. Late = process separately. Cross = one modality "looks at" the other. Each has pros/cons.

SimCLR: Simple Contrastive Learning

Core Intuition: Maximize agreement between embeddings of different augmented views of same image. "Same image" provides supervision (self-supervised). Simple framework, powerful results.

$$\mathcal{L} = -\log\frac{\exp(\text{sim}(z_i, z_j)/\tau)}{\sum_k \exp(\text{sim}(z_i, z_k)/\tau)}$$
L:NT-Xent (Normalized Temperature-scaled Cross-Entropy) loss
z_i, z_j:Normalized embeddings (after projection) from two augmented views of same image (positive pair)
\(\text{sim}(z_i, z_j)\):Cosine similarity between \(z_i\) and \(z_j\); \(\in [-1, 1]\), higher for similar views
\(\tau\):Temperature \(\in (0, \infty)\); controls softmax sharpness. Typical: 0.07. Lower \(\tau\) = harder negative mining
Σ_k:Sum over all 2N embeddings in batch (z_i paired with z_j positive + 2N-2 negatives from other images)
Algorithm:Image → 2 augmentations → shared encoder → projection head → contrastive loss
Positive Pair:Two views of same image (should be close); batch provides negatives
Key Insight:Batch size essential (2N negatives per N images); aggressive augmentations required
Positive Similarity
Two views same image should be high after training
Negative Similarity
Different images should be low; more negatives better
Projection Head
Training only; discarded at inference (use encoder embeddings)
Temperature \(\tau\)
Lower (≈0.07) makes loss sharper; controls distribution
Why contrastive learning works: augmentations preserve semantics. Training to match embeddings forces encoder to learn semantic features, not visual artifacts.
👉 Try this: Batch size B=256 minimum (needs many negatives). B=4096 (modern GPU). Larger B = more negatives = easier training. Temperature τ=0.07 standard (sharp contrast). Increase to 0.5—softer, easier but weaker signal. Decrease to 0.01—very sharp, training harder but better features.
SimCLR = two cropped views of same image should agree. The bigger the batch, the harder they have to work to be similar (more negatives to beat).

Key Gotchas: Batch size directly impacts performance (8K+ needed); Augmentations matter more than architecture; Projection head is training-only.

Production Best Practices: Aggressive augmentations (cropping, color jitter, blur); Sync batch norm across GPUs; Monitor positive/negative similarities; Linear probe on frozen embeddings to evaluate; Large batches non-negotiable.

Momentum Contrast (MoCo): Memory Bank

Watch the memory bank (queue) accumulate embeddings. Momentum encoder updates slowly (exponential moving average), preventing embedding staleness. Increase momentum coefficient m to see queue stability—high m preserves history, low m = fresh embeddings but stale queue.

0.99

MoCo: Large Queue of Negative Samples

Core Problem: SimCLR needs huge batches for good negatives. MoCo solves by maintaining queue of past negatives—enables training on single GPU with smaller batches.

$$\theta_k \leftarrow m \cdot \theta_k + (1-m) \cdot \theta_q \quad (m \approx 0.999)$$
\(\theta_k\):Key encoder parameters; updated via exponential moving average (EMA)
\(\theta_q\):Query encoder parameters; updated via gradient descent
\(m\):Momentum coefficient \(\in (0, 1)\); controls EMA decay. Higher m = slower tracking, more staleness. Typical: 0.999
1-m:Learning rate for key encoder; (1-0.999) = 0.001 = 0.1% of query update per step

Momentum update: slow key encoder tracks query encoder, ensures queue consistency

Two Encoders:Query (gradient updates) and Key (momentum updates); produces consistent representations
Queue Mechanism:Store K past embeddings; current positive enqueues, oldest dequeues; provides K negatives
Momentum m:Higher m = more stale negatives (stable); lower m = fresher (but risks collapse)
SimCLR
Needs batch_size negatives; requires huge batches
MoCo
Needs queue_size negatives; smaller batches sufficient
Momentum update prevents collapse: if you updated key encoder by gradient, queue would have stale embeddings (from different model state) and collapse.
👉 Try this: Momentum m=0.999 (recommended). Key encoder (K) updates slowly: θ_K ← m·θ_K + (1-m)·θ_Q. This maintains stable queue. m=0.9 = updates faster, queue less stable. Queue size=65536 (large bank of negatives). Older entries decay out, newer ones in.
MoCo = two encoders, one fast (query), one slow (key via momentum). Queue accumulates past keys. Current positive joins queue; oldest leaves.

Key Gotchas: Momentum coefficient m critical (too low → collapse; too high → stale negatives); Queue initialization; Smaller batch sizes need larger queues.

Production Best Practices: Tune m based on dataset; Monitor queue freshness; Use in-batch negatives + queue for stronger signal; Distribute queue across GPUs at scale; Enables single-GPU training.

BYOL & DINO: Bootstrap Without Negatives

BYOL uses momentum to prevent collapse (no explicit negatives). DINO adds centering to maintain feature diversity. Watch how representations stay diverse despite no negative samples. Toggle between BYOL (simpler) and DINO (more robust). Observe: no need for large batch sizes or memory banks.

BYOL & DINO: Negatives-Free Learning

Core Insight: Contrastive learning NOT required! BYOL: student trained to match EMA teacher. No negatives, yet excellent representations—shocking result.

$$\mathcal{L}_{\text{BYOL}} = \|h_{\text{student}} - [h_{\text{teacher}}]_{\text{sg}}\|^2$$
L_BYOL:Mean-squared error (MSE) between student and teacher representations
h_student:Representation from student network; receives gradients during backprop
h_teacher:Representation from teacher network; should match student after training
[·]_sg:Stop-gradient operator; detaches tensor from computational graph. Prevents gradient flow to teacher
||·||^2:Squared Euclidean norm; L2 distance. Loss = 0 when student and teacher match

[ ]_sg = stop-gradient; teacher updated by EMA, not gradients

BYOL Architecture:Student (gradient updates) + Teacher (EMA); stop-gradient through teacher prevents collapse
DINO (ViT):Student learns to match teacher via cross-entropy; centering + sharpening prevent collapse
Asymmetric Temps:DINO: student \(\tau_s=0.1\), teacher \(\tau_t=0.04\) (inverted); sharpens teacher outputs
BYOL
MSE loss; stop-gradient; simpler; emergent semantic segmentation
DINO
Cross-entropy; centering+sharpening; ViT-based; more stable
Standard Distill
Fixed teacher; BYOL updates teacher from student (key difference)
BYOL's big finding: negative samples aren't essential for learning good representations. Paradigm shift from contrastive to self-distillation methods.
👉 Try this: BYOL stops collapse via momentum (no negatives needed). DINO adds center & sharpening to prevent mode collapse. Batch size B=256 fine for both (no memory bank). Key: momentum update on student→teacher. This enables efficient contrastive-free learning.
BYOL = student chases its moving target (EMA teacher). Stop gradients through teacher prevents collapse. No negatives needed—mind-bending!

Key Gotchas: EMA momentum critical; Too-fast updates cause instability; Need to monitor embedding variance for collapse; Architecture-specific (works well with ViT/CNN).

Production Best Practices: More stable than contrastive methods; Less hyperparameter tuning; DINO especially good for ViTs; Monitor embedding collapse metrics; Both more robust than SimCLR.

SHAP: SHapley Additive exPlanations

Core Idea: Use game theory (Shapley values) to assign each feature its contribution to prediction. Shapley value = average marginal contribution across all possible coalitions of features.

$$\phi_i(f, x) = \sum_{S \subseteq F \setminus \{i\}} \frac{|S|!\,(|F|-|S|-1)!}{|F|!} \cdot (f(S \cup \{i\}) - f(S))$$
\(\phi_i(f,x)\):Shapley value for feature i on sample x; contribution of i to prediction f(x)
f:Model/function being explained; outputs scalar prediction
F:Full set of all features; |F| = number of features
S⊆F\{i}:Subset of features excluding i; averages over all possible coalitions
f(S∪{i}) - f(S):Marginal contribution of feature i when added to coalition S
|S|!(|F|-|S|-1)! / |F|!:Coalition weight; ensures each ordering weighted equally. Normalized by factorial terms

Accounts for ALL possible orderings/coalitions; \(\phi_i\) = marginal contribution of feature \(i\)

Interpretation:\(f(x) = \mathbb{E}[f(X)] + \sum \phi_i(x)\); \(\phi_i > 0\) pushes prediction higher, \(\phi_i < 0\) pushes lower
Computational Cost:Exact = \(2^{|F|}\) subsets (exponential); TreeSHAP (poly); KernelSHAP (any model, regression)
KernelSHAP:Generate binary feature masks, fit weighted linear regression with normalized weights
Local Explanation
SHAP values per prediction; why did model predict this?
Global Importance
Average |SHAP| values; feature importance across data
TreeSHAP
Exact for tree models; polynomial time
vs LIME
LIME is local approximation; SHAP is game-theoretically founded
Shapley values are uniquely determined by symmetry, dummy, and linearity axioms. This axiomatic foundation makes SHAP more principled than alternative explainability methods.
👉 Try this: SHAP = game theory (Shapley value). Feature i's contribution = its marginal value across all coalitions. Computationally expensive (2^d), so use approximations (TreeSHAP=O(d·T), KernelSHAP=O(m·d)). Perfect for model debugging, feature importance ranking, fairness audits.
SHAP = each feature gets a "contribution score." Game theory ensures it's fair: sums to difference from baseline. Every possible coalition considered.

Key Gotchas: Computational cost high (mitigate with TreeSHAP or KernelSHAP); Background dataset choice affects explanations; Missing feature handling via conditional expectation.

Production Best Practices: Use TreeSHAP for trees; Representative background data; Compute on sample for speed; Waterfall/force plots for viz; Regulatory-friendly (game theory foundation); Finance/healthcare standard.

LIME: Local Linear Approximations

Core Concept: Explain predictions by fitting simple linear model in neighborhood. Complex model locally linear → linear model provides human-understandable explanation.

$$\min_w \sum_j \text{weight}_j \cdot (f(x_j) - (w_0 + \sum_i w_i \cdot x_j^i))^2 + \lambda\|w\|$$
w:Linear regression coefficients to learn; w_0 = intercept, w_i = feature slopes
weight_j:Proximity weight for sample j; higher for samples near x (example). Computed via exponential kernel
f(x_j):Prediction from complex model on perturbed sample j
w_0 + Σ_i w_i · x_j^i:Prediction from linear approximation; simple interpretable model
\(\lambda\|w\|\):Regularization term (L1 or L2); \(\lambda \geq 0\) controls sparsity. Larger \(\lambda\) = simpler model
Σ_j:Sum over perturbed samples in local neighborhood (typically 100-1000 samples)

Weighted regression with proximity weights; L1/L2 encourages sparse solutions

Algorithm:Perturb features randomly → get predictions → fit weighted linear regression → interpret coefficients
Weights:Exponential kernel: weight_j = \(\exp(-\text{dist}^2/\sigma^2)\); closer instances weighted more
Text/Images:Text = remove words; Images = superpixels (treat segments not pixels as features)
Local Linearity
Complex model may be linear in small neighborhood
Neighborhood Size
Too small = high variance; too large = poor approx
vs SHAP
LIME simpler/faster; SHAP theoretically grounded
Model-Agnostic
Works with any black-box model
Key insight: you don't need to understand the whole model, just the local behavior around one prediction. Linear model works because of local linearity.
👉 Try this: LIME samples m≈1000 perturbed instances. Weights them by distance to original sample (exponential decay). Fits weighted linear model. Feature coefficients = local importance. Good for debugging, bad for global understanding. Fast, intuitive, but unstable (perturbation random).
LIME = local + linear. Perturb around instance, fit simple linear model. Coefficients show which features matter locally.

Key Gotchas: Stability issues (perturb twice, get different explanations); Neighborhood size/bandwidth critical; May miss feature interactions; Slower than simple feature importance.

Production Best Practices: 1000-5000 perturbed samples; Normalize features; Use for debugging model failures; Consider with SHAP for rigor; Superpixels for images; Word removal for text.

Mechanistic Interpretability: Understanding Neural Networks

Goal: Reverse-engineer learned algorithms by understanding individual components and how they compose. Move beyond "black box" to understand mechanisms of computation.

$$\text{Output} = \sum (\text{neuron\_activation} \cdot \text{learned\_weight} \cdot \text{downstream\_importance})$$
Output:Final model prediction or logit
neuron_activation:Activation value of neuron; output of ReLU, tanh, etc. \(\in [0, \infty)\) or \([-1, 1]\)
learned_weight:Weight connecting neuron to downstream layer; trained parameter
downstream_importance:Measure of how much downstream layers use this neuron's output. Can be gradient, attention, or causal
Σ:Sum over all neurons in layer; aggregates contributions to final output
Feature Decomposition:Neurons activate for specific concepts ("ears", "whiskers" in vision; syntactic properties in NLP)
Superposition:One neuron encodes multiple features simultaneously via learned interference patterns
Circuits:Attention heads + MLPs + neurons form "circuits" implementing specific algorithms

Key Analysis Techniques:

Probing (Correlation)
Train classifier on activations; shows what info exists
Intervention (Causation)
Ablate neurons; test actual causal importance
Attribution
Trace backward via gradients/attention weights
Saliency Maps
Visualize input-output importance via LRP
Probing ≠ causation. Classifier on activations shows info exists, but intervention (ablation) proves components matter for output.
👉 Try this: Linear probe: train classifier on neuron activations to predict labels. High accuracy = neuron encodes info. Ablation: set neuron to 0, measure performance drop. Only ablation proves causation. Neurons often multiplex (encode many features at once)—interpretation hard.
Mechanistic = understand the gears. Feature decomposition answers "what do neurons care about." Circuits answer "how do they talk." Intervention answers "what actually matters."

Key Gotchas: Superposition complicates interpretation; Probing misleading without intervention; Hypothesis-driven; Small models > large; Causation hard to establish.

Graph Neural Networks: Relational Learning

Core Idea: Learn on graph-structured data. Nodes = entities, edges = relationships. Aggregate neighbor features.

$$h'_v = \sigma\!\left(\sum_{u \in N(v)} W \cdot h_u\right) \quad N(v) = \text{neighbors}$$
h'_v:Updated representation (embedding) of node v after one layer
\(h_u\):Feature/embedding of neighbor node u ∈ N(v); input to aggregation
\(W\):Weight matrix \(\in \mathbb{R}^{d_{\text{out}} \times d_{\text{in}}}\); shared transformation applied to all neighbors
\(\sum_{u \in N(v)}\):Aggregation function; typically mean or sum over neighbors. Other options: max, attention
\(\sigma(\cdot)\):Activation function; ReLU, tanh, or others. Introduces non-linearity
N(v):Neighborhood of node v; set of adjacent nodes (1-hop neighbors in graph)
Message Passing:Nodes send messages to neighbors; aggregate to update representation
Graph Types:Directed, undirected, heterogeneous (multi-type nodes/edges), dynamic
Locality Bias:Neighbors matter; inductive bias matches many domains naturally
Leverages relational structure. More efficient than fully-connected networks for sparse graphs.
👉 Try this: GNN layer: h_i^{l+1} = σ(W · AGG({h_j^l : j ∈ N(i)})). Message passing = aggregation of neighbor info. Sparse edges = low compute. Deep GNNs (L>6) risk over-smoothing (all nodes become similar). Solution: residual connections, skip connections, or shallow networks.
GNN = aggregate neighbor features. Update nodes. Repeat. Powerful for relational data.

Applications: Drug discovery, social networks, recommendation, knowledge graphs, physics.

Knowledge Distillation: Model Compression

Core Idea: Train small "student" model to match large "teacher" model. Compress knowledge into smaller network via soft targets.

$$\mathcal{L} = \alpha \cdot \mathcal{L}_{\text{CE}}(y_{\text{student}}, y_{\text{true}}) + (1-\alpha) \cdot \mathcal{L}_{\text{KL}}(T_{\text{student}}, T_{\text{teacher}})$$
L:Combined loss for student model
\(\alpha\):Weight balancing hard labels vs soft targets; \(\in [0, 1]\). Typical: 0.5-0.9
L_CE(y_student, y_true):Cross-entropy loss on hard labels; standard classification loss
y_true:Ground-truth one-hot labels
L_KL(T_student, T_teacher):KL divergence between softened student and teacher outputs. T = temperature
\(T\) (Temperature):Softening factor \(\in (1, \infty)\); higher T = softer probability distributions. Typical: 3-5. T=1 → no softening
Temperature T:Controls softness of outputs; higher T = softer targets = easier learning for student
Dark Knowledge:Soft targets encode class relationships teacher learned; more informative than hard labels
Student learns "dark knowledge"—soft targets embed class similarities that hard labels don't convey.
👉 Try this: Loss = α·KL(student, teacher) + (1-α)·CE(student, labels). Temperature τ softens teacher logits: p_soft = softmax(z/τ). High τ (e.g., 10) = soft targets (smooth probability, easier gradient). Low τ (e.g., 1) = sharp targets (hard contrast). Typical τ=4-20.
Distillation = teacher transfers knowledge to student. Soft targets better than hard labels.

Benefits: Smaller models; Fast inference; Knowledge transfer; Often beats training small from scratch.

vs LoRA: Distillation for compression; LoRA for efficient fine-tuning. Orthogonal approaches.

Quantization: Reduce Precision, Maintain Performance

Core Idea: Use lower precision (int8, fp16, binary) instead of fp32 to reduce size and speed up inference.

Post-Training Quantization:Quantize after training; fast but may lose 1-2% accuracy
Quantization-Aware Training:Simulate quantization during training; recover accuracy via gradient updates
Mixed Precision:Different layers at different precisions; balance accuracy and efficiency
Neural networks surprisingly robust to low precision. Often <2% accuracy loss with int8.
👉 Try this: INT8 = 128 levels (vs FP32 = billions). Activation quantization (per-channel) more forgiving than weight quantization. Calibration: collect statistics on representative samples. QAT (quantization-aware training) beats post-training quantization. INT8: mobile-ready, 4x faster. INT4: extreme compression, needs tuning.
Quantization = fewer bits. int8 instead of fp32. 4x smaller, 4x faster on hardware.

Benefits: Smaller models; Faster inference; Lower memory; Hardware efficient (TPU/GPU optimized).

Adjust rank r: small = cheap, large = flexible. Watch: training parameters drop 100x while accuracy stays competitive.

LoRA: Efficient Fine-tuning

Core Idea: Instead of fine-tuning all weights, train low-rank adaptations. Freeze original weights, add trainable low-rank matrices.

$$W' = W + \Delta W = W + A \cdot B^T \quad \text{where } A \in \mathbb{R}^{n \times r},\; B \in \mathbb{R}^{m \times r},\; r \ll \min(n,m)$$
W':Updated weight matrix after LoRA; W' = original + low-rank update
\(W\):Original (frozen) weight matrix \(\in \mathbb{R}^{n \times m}\); does not receive gradients
\(A\):Left low-rank matrix \(\in \mathbb{R}^{n \times r}\); trainable (receives gradients)
\(B\):Right low-rank matrix \(\in \mathbb{R}^{m \times r}\); trainable (receives gradients)
\(r\):Rank \(\in [1, \min(n,m)]\); much smaller than dimensions. Typical: 8-64. Larger r = more expressive
\(A \cdot B^T\):Low-rank update; produces \(\in \mathbb{R}^{n \times m}\), same shape as W
Parameter Efficient:Only \(r \cdot (n+m)\) params vs \(n \cdot m\) full; r ≈ 8-64 typical = 10-100x fewer params
Composable:Multiple LoRAs combine for different tasks; switch without reloading weights
Hypothesis:Fine-tuning changes live in low-dimensional subspace; LoRA exploits this
Low-rank hypothesis: weight updates during fine-tuning don't need full rank. Major efficiency win.
👉 Try this: LoRA rank r=8 or 16 typical (1% params vs full fine-tune). r=4 too constraining, r=64 unnecessary for most tasks. Inference: merge LoRA into base W (no latency cost). Multiple tasks: train separate LoRA modules, share base (switch modules per task). Merging enables efficient deployment.
LoRA = low-rank update. Add small trainable matrices. Freeze original. Efficient and composable.

Benefits: 10-100x fewer params; Fast training; Composable; Memory efficient; Production-friendly.
