Bayes Theorem: Disease Testing

Goal: understand why a positive test can still mean low actual risk when the underlying event is rare.

What This Is

Bayes’ theorem tells you how to update a belief after seeing evidence. You start with a prior belief, then revise it using how compatible the evidence is with the hypothesis.

The classic trap is to confuse test accuracy with posterior probability. A positive test can still mean modest risk if the condition itself is rare.

Core Interview Concept Trap: confusing accuracy with posterior Practice: base-rate reasoning

  2. Say out loud whether the posterior should be close to 99% before you look.
  3. Now raise the prior to 50% and notice how the exact same test suddenly becomes much more convincing.
Current read: with a very rare condition, even a strong test produces many false positives relative to true positives. That is why the posterior stays much lower than most people expect.

Why This Matters

Bayes shows up whenever you update a belief after seeing evidence: disease tests, spam filters, anomaly detection, A/B testing, and model regularization.

Common Misconception

“The test is 99% accurate, so a positive result means 99% chance of disease.” That mixes up test accuracy with the posterior probability after accounting for prevalence.

What Changes At Scale?

In production systems, calibration matters as much as ranking. A model can order risks well but still produce poor probabilities if its outputs are not calibrated to real-world base rates.

Check Yourself

Why can a positive test still imply only modest posterior probability when prevalence is low?

Key Insight: Bayes Theorem is the foundation of probabilistic inference.
$$P(A|B) = \frac{\color{#c9a96e}{P(B|A) \cdot P(A)}}{P(B)}$$
Posterior: \(P(A|B)\) - updated belief after observing evidence
Likelihood: \(P(B|A)\) - probability of data if hypothesis is true
Prior: \(P(A)\) - initial belief before seeing data
Evidence: \(P(B)\) - total probability of observing data (normalizer)
When to Use
Medical / Anomaly Detection
When base rates matter and false positives are costly. Bayes lets you update disease probability after a test result.
Examples: Disease screening (COVID tests), fraud detection with rare events, spam filtering with prior spam rates.
Bayesian Inference
When you have prior domain knowledge and want to incorporate it with observed data. Essential when data is limited.
Examples: A/B testing with informative priors, Bayesian neural networks for uncertainty, clinical trials with historical data.
Naive Bayes Classifier
Fast, simple baseline for classification. Assumes feature independence (often violated but still works well in practice).
Examples: Text classification (spam, sentiment), document categorization, real-time predictions needing low latency.
Base rate fallacy: A 99% accurate test on a 1% prevalence disease gives only ~50% posterior probability. Prior dominates with rare events!
Posterior ∝ Likelihood × Prior. Update beliefs by multiplying how well data fits with what you already believed.
ML Connection: L2 regularization \(\lambda\|w\|^2\) = log Gaussian prior on weights. Regularization = MAP with Gaussian prior.
👉 Try this: Set Prior=1%, Sensitivity=99%, Specificity=99%. The posterior is ~50%, not 99%! Now increase Prior to 50% — the posterior jumps to 99%. Base rate is everything. Then reduce Sensitivity to 70% with Prior=1% — posterior drops below 10%.
import numpy as np

# Bayes: P(disease|positive) = P(pos|disease)*P(disease) / P(pos)
prior = 0.01          # 1% disease prevalence
sensitivity = 0.99    # P(positive | disease)
specificity = 0.99    # P(negative | no disease)

p_pos = sensitivity * prior + (1 - specificity) * (1 - prior)
posterior = (sensitivity * prior) / p_pos
print(f"P(disease|positive) = {posterior:.3f}")  # ~0.500

Even with 99% accuracy, rare diseases yield ~50% posterior due to base rate.

Probability Distributions

Goal: learn to match the kind of random quantity you have to the right probability distribution.

What This Is

A probability distribution describes what values a random variable tends to take and how likely those values are.

The main practical question is not “can I memorize the formula?” It is “what kind of thing am I modeling?” Counts, waiting times, binary events, probabilities, and continuous measurements usually need different families.

Core Modeling Choice Trap: “everything is Gaussian” Practice: map data shape to family

Current read: Bernoulli is for one binary event. Use it when the outcome is yes/no, not when you are counting multiple events or modeling continuous values.

Why This Matters

Choosing the wrong family gives the wrong support, the wrong uncertainty, and often the wrong loss function. Distribution choice is one of the fastest ways to reveal conceptual maturity.

Common Misconception

People often reach for a Gaussian because it is familiar, even when the target is a count, a probability, or a waiting time. The support of the variable should guide the choice first.

What Changes At Scale?

Real systems often mix families: Poisson-like counts, Bernoulli labels, heavy-tailed continuous features, and Beta priors for rates. Good modeling starts by matching each signal to the right assumptions.

Check Yourself

Which pairing is the best match?

Key Insight: Each distribution has specific use cases, parameters, and assumptions.
Bernoulli / Binomial
Binary events. Binomial: \(E[X] = np\), \(\text{Var} = np(1-p)\) for n trials.
Poisson
Rare events with rate \(\lambda\). When n large, p small, approximates Binomial.
Gaussian
Parameterized by \(\mu, \sigma^2\). Everywhere due to Central Limit Theorem.
Exponential
Waiting times, inter-arrival. Memoryless property.
Beta Distribution: Conjugate prior to Binomial. Flexible shapes via \(\alpha, \beta\). Standard for Bayesian A/B tests.
When to Use
Gaussian / Normal
Default for continuous real-valued data. Justified by CLT for aggregates and sums. Use when data is symmetric around a mean.
Examples: Height/weight modeling, measurement errors, sensor noise, regression residuals, financial returns (approximately).
Poisson / Exponential
Poisson for count data (events per time). Exponential for time between events. Both tied to rate parameter lambda.
Examples: Website clicks/hour, customer arrivals, server requests, time between failures, earthquake occurrences.
Beta / Bernoulli
Bernoulli for binary outcomes. Beta as prior for probability parameters in Bayesian inference (conjugate to Binomial).
Examples: Click-through rates (Beta), coin flips (Bernoulli), A/B test conversion rates (Beta posterior), churn prediction (Bernoulli).
👉 Try this: Click Gaussian and drag Parameter to 0.1, then to 0.9 — watch spread change continuously. Then switch to Poisson with \(\lambda=1\), then \(\lambda=10\). The shape becomes less skewed as the expected count grows.
When to use each? Poisson for counts, Gaussian for continuous data, Beta for probability parameters, Exponential for waiting times.
Binary→Bernoulli, Counts→Poisson, Continuous→Gaussian, Waiting times→Exponential, Probability parameters→Beta

MLE (Maximum Likelihood Estimation) vs MAP (Maximum A Posteriori): Coin Flips

Goal: see the difference between trusting only the observed data and combining data with prior belief.

What This Is

MLE asks: “Which parameter makes the observed data most likely?” It uses only the data you saw.

MAP asks the same question but also includes a prior belief about plausible parameter values. That makes it especially useful when data is scarce or noisy.

Why This Matters

MLE vs MAP is the conceptual bridge between classical statistics and practical ML regularization. It is one of the cleanest ways to explain why priors and penalties help.

Common Misconception

People often think MAP is always superior because it “uses more information.” In practice, a bad prior can bias the answer, and with lots of data the prior should matter much less.

What Changes At Scale?

In large deep learning systems, explicit priors often appear as implicit penalties: L2 weight decay, sparsity penalties, and architectural biases all play the role of soft prior assumptions.

Check Yourself

When is MAP most noticeably different from MLE?

Key Insight: MLE maximizes likelihood; MAP incorporates prior beliefs.
$$\text{MLE: } \theta^* = \arg\max_{\theta} \log P(\text{data}|\theta)$$
\(\theta\):Model parameters to optimize (weights, means, variances, etc.). \(\theta^*\) is the optimal value.
\(\arg\max_{\theta}\):Find the parameter value \(\theta\) that maximizes the expression—returns \(\theta\) not the maximum value
\(P(\text{data}|\theta)\):Likelihood—probability of observing the data given parameters \(\theta\). Higher = better fit.
$$\text{MAP: } \theta^* = \arg\max_{\theta} \left[\log P(\text{data}|\theta) + \color{#c9a96e}{\log P(\theta)}\right]$$
\(P(\text{data}|\theta)\):Likelihood—same as MLE, fits data well
\(P(\theta)\):Prior—encodes domain belief about which \(\theta\) values are reasonable before seeing data
Adding \(\log P(\theta)\):Biases optimization toward high-probability \(\theta\) regions. In neural networks, this is regularization (L2 = Gaussian prior, L1 = Laplace prior)
Maximum Likelihood
Ignores prior and trusts the observed data entirely. With enough data, this often becomes the same practical answer MAP would give.
Maximum A Posteriori
Incorporates \(P(\theta)\). Combines likelihood with domain knowledge.
👉 Try this: Set Flips=5, Prior Strength=50. The posterior is visibly pulled by the prior. Increase Flips to 100 — the posterior peak moves toward the data. With enough data, MAP behaves more like MLE.
When to Use
MLE (Maximum Likelihood)
When you have abundant data and no strong prior beliefs. MLE is simpler and converges to truth with enough data. Most deep learning uses MLE (cross-entropy loss).
Examples: Training large neural networks, logistic regression on large datasets, any scenario with millions of samples.
MAP (Maximum A Posteriori)
When data is limited or you have domain knowledge to encode as priors. MAP = regularized MLE. Prevents overfitting on small datasets.
Examples: Small-sample medical studies, L2-regularized models, cold-start recommendations, transfer learning with informative priors.
Regularization = MAP: L2 regularization \(\lambda\|w\|^2\) = log Gaussian prior. Regularization is MAP in disguise.
Use MAP when you have domain knowledge or limited data. Why regularization works? It encodes the belief that weights should be small.
MLE = trust data completely. MAP = balance data with prior beliefs. Regularization = MAP hidden in neural networks.
import numpy as np

# Coin flip: 7 heads out of 10 flips
heads, n = 7, 10

# MLE: maximize likelihood -> just the proportion
mle = heads / n  # 0.7

# MAP with Beta(2,2) prior (slight bias toward 0.5)
alpha_prior, beta_prior = 2, 2
map_estimate = (heads + alpha_prior - 1) / (n + alpha_prior + beta_prior - 2)
print(f"MLE: {mle:.2f}, MAP: {map_estimate:.2f}")  # 0.7 vs 0.667

MAP shrinks estimate toward the prior. With more data, MAP converges to MLE.

CLT (Central Limit Theorem)

Goal: understand that CLT is about averages becoming Gaussian, not raw data points magically changing shape.

What This Is

The Central Limit Theorem says that if you repeatedly average independent samples, the distribution of those averages tends toward a Gaussian as sample size grows.

The subtle but crucial point is that the theorem is about the sample mean, not the original data distribution. Your raw data can stay skewed while the averages become bell-shaped.

Core Statistical Intuition Trap: CLT applies to raw data Practice: means vs observations

  2. Increase sample size to 30 or 50. Now each bar comes from an average of many draws.
  3. Notice that the histogram narrows as n grows: the average becomes more stable.
Current read: CLT is about the sampling distribution of the mean, not about individual points becoming Gaussian.

Why This Matters

CLT is the backbone of confidence intervals, A/B tests, and large-sample inference. It explains why Gaussian approximations show up everywhere in applied statistics and ML.

Common Misconception

Students often say “the data becomes normal.” What actually becomes approximately normal is the distribution of the sample mean across repeated samples.

What Changes At Scale?

In ML, larger batches reduce variance in gradient estimates for the same reason: averaging many noisy terms produces a more stable aggregate.

Check Yourself

What does the CLT approximate with a Gaussian?

Key Insight: Sample means approach a Gaussian distribution as sample size increases, regardless of underlying distribution.
$$\text{If } X_1, X_2, \ldots, X_n \sim \text{any distribution with mean } \mu, \text{ variance } \sigma^2$$$$\text{Then: } \frac{\bar{X} - \mu}{\sigma / \sqrt{n}} \to \color{#c9a96e}{\mathcal{N}(0,1)}$$
Key condition: Finite variance. Distribution shape doesn't matter—only mean and variance.
Why it matters: Justifies t-tests, confidence intervals, Gaussian assumptions in inference on sample means.
Sample size needed: ~30 samples for most distributions. More if distribution is very skewed.
When to Use
Confidence Intervals
CLT justifies using normal-based confidence intervals for sample means, even when the underlying data is not Gaussian, as long as n is large enough (~30+).
Examples: A/B test lift estimates, average revenue per user, mean response time for a service.
Hypothesis Testing
Enables t-tests and z-tests on non-normal data. The test statistic's distribution converges to normal under CLT, making p-values valid.
Examples: Comparing two group means, testing if a new model is significantly better, evaluating treatment effects.
Batch Averaging in ML
Mini-batch gradient estimates are approximately normal by CLT. Larger batches give more stable (less noisy) gradient estimates.
Examples: Choosing batch size in SGD, understanding why batch normalization works, error analysis in ensemble methods.
This is why we can use normal-based tests even when data isn't normally distributed. Always remember: applies to sample MEANS, not individual observations.
Averages of many draws look Gaussian, even when the original distribution does not. CLT is the reason Gaussian approximations appear so often.
👉 Try this: Set Sample Size = 1. Animate 3 times — histogram stays jagged and skewed. Set Sample Size = 100. Animate again — the histogram becomes much smoother and narrower. More averaging means less variance in the mean.
import numpy as np

# Sample means from ANY distribution converge to Gaussian
population = np.random.exponential(scale=2, size=100000)  # skewed!
sample_means = [np.random.choice(population, 50).mean() for _ in range(1000)]

print(f"Population: mean={population.mean():.2f}, std={population.std():.2f}")
print(f"Sample means: mean={np.mean(sample_means):.2f}, "
      f"std={np.std(sample_means):.2f}")  # std ≈ pop_std/sqrt(50)

CLT: sample means are ~Gaussian regardless of population shape, with std = sigma/sqrt(n).

Entropy & Cross-Entropy

Goal: understand entropy as uncertainty, cross-entropy as the cost of predicting with the wrong probabilities, and KL as the extra penalty for that mismatch.

What This Is

Entropy answers: “How uncertain is the true outcome?” If a coin is near 50/50, entropy is high because the next result is hard to predict. If it is near 99/1, entropy is low because the outcome is mostly predictable.

Cross-entropy answers: “How expensive is it to predict the truth using my model’s probabilities?” If your model q matches the real distribution p, that cost is as low as possible. If q is wrong, you pay extra.

KL divergence is exactly that extra payment: the gap between the irreducible uncertainty in the task and the additional cost caused by a mismatched model.

Current read: entropy describes the true task uncertainty; cross-entropy adds the cost of using an imperfect model.

Why This Matters

Cross-entropy is the default loss for classification and language modeling because training is really asking the model to match the true distribution as closely as it can.

Common Misconception

Entropy is about the task itself. Cross-entropy is about the task plus your model. They become equal only when the model distribution matches the truth.

What Changes At Scale?

In real neural networks, p is often the label distribution and q is the softmax output. So minimizing cross-entropy is just distribution matching in disguise.

Check Yourself

If the model distribution q matches the true distribution p exactly, what happens?

Key Insight: Entropy measures information content and fundamental limits of compression.
$$H(p) = -\sum p(x) \log p(x)$$
\(p(x)\):True probability of outcome x (the real data distribution)
\(\sum\):Sum over all possible outcomes x in the distribution
H(p):Minimum average bits needed to encode outcomes from distribution p (lossless compression limit). Maximum = log₂(N) for N equally likely outcomes.
$$\text{Cross-entropy: } H(p, q) = -\sum p(x) \log q(x)$$
p(x):True distribution — how outcomes actually occur (e.g., true labels)
q(x):Model's predicted distribution — your approximation (e.g., softmax outputs)
H(p, q):Average bits needed when encoding with wrong distribution q. Always ≥ H(p). The gap is the KL divergence.
$$D_{KL}(p \| q) = H(p, q) - H(p) = \sum p(x) \log \color{#c9a96e}{\frac{p(x)}{q(x)}}$$
\(\frac{p(x)}{q(x)}\):Likelihood ratio — how much the true distribution disagrees with the model at each point x
KL = 0:q perfectly matches p everywhere. KL ≥ 0 always (Gibbs' inequality).
Asymmetric:KL(p‖q) ≠ KL(q‖p) — this matters hugely for VAEs vs. expectation propagation
When to Use
Cross-Entropy Loss
Standard loss for classification tasks. Measures how well predicted probabilities match true labels. Used in virtually all neural network classifiers.
Examples: Image classification (softmax + CE), sentiment analysis, language model next-token prediction, multi-label classification.
KL Divergence
Measures distribution mismatch. Used as a regularizer or objective when you want one distribution to approximate another.
Examples: VAE latent space regularization, knowledge distillation (teacher-student), policy optimization in reinforcement learning (PPO/TRPO).
Entropy as a Metric
Measures uncertainty or information content. High entropy = uncertain predictions. Used for active learning and decision tree splits.
Examples: Decision tree split criterion, active learning (query most uncertain samples), measuring model calibration, data compression.
"What loss does classification optimize?" → Cross-entropy H(p, q) where p = one-hot true labels, q = softmax predictions. Minimizing cross-entropy = minimizing KL(p‖q) because H(p) is fixed.
p = truth, q = your model. H(p) = irreducible uncertainty. H(p,q) = what you pay with your model. KL = the extra tax for being wrong.
👉 Try this: Set p=0.5 and q=0.5. Now KL is near zero because the model matches the truth. Move q to 0.9 and watch cross-entropy rise above entropy. The gap is exactly the mismatch penalty.
import numpy as np

def entropy(p):
    p = np.array(p)
    return -np.sum(p * np.log2(p + 1e-10))

def cross_entropy(p, q):
    return -np.sum(np.array(p) * np.log2(np.array(q) + 1e-10))

p = [0.7, 0.2, 0.1]  # true distribution
q = [0.33, 0.33, 0.34]  # model prediction (uniform-ish)

print(f"H(p) = {entropy(p):.3f} bits")
print(f"H(p,q) = {cross_entropy(p, q):.3f} bits")
print(f"KL(p||q) = {cross_entropy(p, q) - entropy(p):.3f} bits")

Cross-entropy = entropy + KL divergence. Minimizing cross-entropy loss = minimizing KL from true distribution.

Key Insight: KL divergence is asymmetric; forward and reverse KL have opposite behavior.
Forward KL: KL(p||q)
Penalizes missing modes where p has mass. q must have modes everywhere p has mass. Mode-covering, covers all modes.
👉 Try this: Set p to a narrow peak (width=5), q to same location. KL(p||q) ≈ 0 (q covers p's high-probability region). Now drag q to ignore p's peak but cover entire range. KL(p||q) spikes (mode-seeking vs mass-covering behavior).
Reverse KL: KL(q||p)
Penalizes placing mass where p has none. q focuses on highest mode of p. Mode-seeking, picks single mode.
Forward KL consequence: Used in expectation propagation, can learn multimodal distributions, places probability mass broadly
Reverse KL consequence: Used in variational inference, learns sharp concentrated distribution, misses modes
When to Use
Forward KL (Mode-Covering)
When you need to cover all modes of the true distribution, even at the cost of placing mass where p is low. Produces broader, more conservative approximations.
Examples: Expectation propagation, supervised learning (cross-entropy loss is forward KL), when missing a mode is dangerous (safety-critical systems).
Reverse KL (Mode-Seeking)
When you want a sharp, focused approximation that locks onto the strongest mode. Avoids placing mass where p is near zero.
Examples: Variational inference (ELBO optimization), VAE training, reinforcement learning policy optimization, when precision matters more than recall.
Reverse KL in VAEs leads to posterior collapse if not careful. Forward KL more expensive computationally but covers all modes.
Forward KL covers all modes (broader, safer). Reverse KL picks best mode (faster, sharper). Remember: asymmetry matters for distribution matching!
from scipy.special import kl_div
import numpy as np

p = np.array([0.9, 0.1])  # peaked distribution
q = np.array([0.5, 0.5])  # uniform distribution

# KL is asymmetric!
kl_pq = np.sum(p * np.log(p / q))  # forward KL
kl_qp = np.sum(q * np.log(q / p))  # reverse KL

print(f"KL(p||q) = {kl_pq:.4f}")  # forward: mean-seeking
print(f"KL(q||p) = {kl_qp:.4f}")  # reverse: mode-seeking

Forward KL (used in supervised learning) covers all modes. Reverse KL (used in VAEs) locks onto one mode.

Key Insight: Momentum accelerates convergence by accumulating velocity over time.
$$\text{Standard SGD: } \theta \leftarrow \theta - \color{#c9a96e}{\eta \cdot \nabla L(\theta)}$$
\(\theta\):Model parameters (weights, biases). \(\leftarrow\) means "update to"
\(\eta\):Learning rate—step size for parameter update. Typical range: 0.001 to 0.1 depending on optimizer. Too large = divergence, too small = slow convergence.
\(\nabla L(\theta)\):Gradient of loss w.r.t. parameters. Points in direction of steepest increase. We move opposite (negative sign) to decrease loss.
$$\text{Momentum: } v_t = \color{#c9a96e}{\beta \cdot v_{t-1}} + \nabla L(\theta), \quad \theta \leftarrow \theta - \eta \cdot v_t$$
\(v_t\):Velocity vector at iteration t. Accumulates gradients over time with decay
\(\beta\):Momentum coefficient, typically 0.9. Controls how much history to retain. \(\beta \in (0,1)\)
\(\beta \cdot v_{t-1} + \nabla L(\theta)\):Weighted sum: retain 90% of previous velocity (\(\beta=0.9\)), add current gradient. Smooths oscillations, accelerates consistent directions.
What momentum does: Accumulates velocity in consistent directions. Dampens oscillations in directions that change sign. Accelerates in plateau regions.
\(\beta\) parameter: Typically 0.9. Controls how much history to retain. Higher \(\beta\) = more history, smoother but slower to change direction.
Typical choices: \(\beta=0.9\) for standard SGD. Modern optimizers (Adam) incorporate momentum automatically.
When to Use
Vanilla SGD
When you want simple, well-understood optimization with manual tuning. Often generalizes better than adaptive methods on well-tuned problems.
Examples: Training ResNets (SGD+momentum often beats Adam), convex problems, when you have compute budget for hyperparameter search.
SGD + Momentum
Default for computer vision. Smooths noisy gradients, accelerates through plateaus. Requires learning rate scheduling.
Examples: ImageNet training, any CNN-based task, problems with ravine-shaped loss landscapes, when generalization is paramount.
Adam / AdamW
Default for NLP and transformers. Adapts per-parameter learning rates. Less tuning needed. AdamW decouples weight decay.
Examples: Transformer training (BERT, GPT), GANs, any new project as a strong default, sparse gradient problems (embeddings).
Momentum helps escape local minima and plateaus. It's like a ball rolling downhill—accumulates speed in good directions.

import numpy as np

# Simple gradient descent comparison
def sgd_step(w, grad, lr=0.01):
    return w - lr * grad

# Adam maintains running averages of gradient and squared gradient
class Adam:
    def __init__(self, lr=0.001, beta1=0.9, beta2=0.999):
        self.lr, self.beta1, self.beta2 = lr, beta1, beta2
        self.m, self.v, self.t = 0, 0, 0
    def step(self, w, grad):
        self.t += 1
        self.m = self.beta1 * self.m + (1 - self.beta1) * grad
        self.v = self.beta2 * self.v + (1 - self.beta2) * grad**2
        m_hat = self.m / (1 - self.beta1**self.t)
        v_hat = self.v / (1 - self.beta2**self.t)
        return w - self.lr * m_hat / (np.sqrt(v_hat) + 1e-8)

Adam adapts learning rate per-parameter using gradient moments. Default choice for most deep learning.

Key Insight: Learning rate controls gradient step magnitude—most critical hyperparameter.
$$\theta \leftarrow \theta - \color{#c9a96e}{\alpha \cdot \nabla L(\theta)} \quad \text{where } \alpha \text{ is learning rate}$$
Too small \(\alpha\): Training prohibitively slow. May get stuck in local minima. High variance from noise.
Too large \(\alpha\): Parameters diverge, loss oscillates wildly, training explodes. May diverge immediately.
Finding the right \(\alpha\): Learning rate schedules (decay, warm restart). Rule of thumb: start with 0.001-0.01, adjust based on loss curve.
Adaptive methods: Adam, RMSprop automatically adjust per-parameter learning rates. Reduces tuning burden.
When to Use
Fixed Learning Rate
Simplest approach. Works for quick prototyping and when using adaptive optimizers (Adam) that adjust rates internally.
Examples: Initial model exploration, Adam with default lr=0.001, simple linear models, small-scale experiments.
LR Scheduling (Step / Cosine)
Decay learning rate over training. Start high for fast progress, reduce for fine convergence. Cosine annealing is popular for vision.
Examples: ImageNet training (step decay at epochs 30, 60, 90), cosine annealing for CIFAR, any long training run with SGD.
Warmup + Decay
Start with tiny LR, ramp up linearly, then decay. Prevents early instability in large models. Standard for transformers.
Examples: BERT/GPT pretraining, any transformer model, large batch training, fine-tuning pretrained models with small warmup.
Plot loss vs. iterations! Smooth curve = good α. Diverging = too large. Flat then drop = too small. Oscillating = right size but noisy.
Learning rate is the gas pedal: too small = crawl, too large = crash. Sweet spot: smooth consistent decrease.
👉 Try this: Set \(\alpha=0.001\) (crawl), \(\alpha=0.1\) (confident), \(\alpha=0.5\) (wild bounce). Find the critical \(\alpha\) where the ball barely oscillates without bouncing off. This is the intuition behind learning rate scheduling.
import numpy as np

# Cosine annealing schedule (most popular)
def cosine_schedule(epoch, max_epochs, lr_init=0.1, lr_min=1e-6):
    return lr_min + 0.5 * (lr_init - lr_min) * (1 + np.cos(np.pi * epoch / max_epochs))

# Warmup + cosine decay
epochs = 100
lrs = [cosine_schedule(e, epochs) for e in range(epochs)]
print(f"LR at epoch 0: {lrs[0]:.4f}")   # 0.1000
print(f"LR at epoch 50: {lrs[50]:.4f}") # ~0.05
print(f"LR at epoch 99: {lrs[99]:.6f}") # ~0.000001

Cosine annealing is the default LR schedule. Start high, decay smoothly to near-zero.

SVD (Singular Value Decomposition) & Rank-k Approximation

Start with "Rank k" at 1 — the image is a blur. Increase to 5, 10, 20 — details progressively emerge. Watch how much information the top few components capture. This is why PCA and low-rank methods work in high-dimensional data.

5
Rank 5 / 50: 10% parameters, 85% variance
Key Insight: SVD factors any matrix into orthogonal matrices and singular values.
$$A_{(m \times n)} = \color{#c9a96e}{U \cdot \Sigma \cdot V^T}$$
U (m×m): Orthogonal matrix. Columns are principal directions of A.
\(\Sigma\) (m×n): Diagonal with singular values \(\sigma_1 \geq \sigma_2 \geq \ldots\) decreasing order. Represents importance of each direction.
\(V^T\) (n×n): Orthogonal matrix. Rows are principal directions in original space.
Interpretation: First r singular values capture ~95% of information. Truncate for dimensionality reduction, denoising.
When to Use
Dimensionality Reduction
Truncated SVD (keeping top-k singular values) is the mathematical backbone of PCA. Use when you need to reduce feature dimensions while preserving maximum variance.
Examples: Reducing 10,000 gene features to 50 components, image compression, preprocessing before clustering.
Recommender Systems
Matrix factorization via SVD decomposes user-item matrices into latent factors. The foundation of collaborative filtering methods.
Examples: Netflix prize (SVD++), product recommendations, music playlist suggestions, any sparse user-item matrix.
NLP / Embeddings
SVD on co-occurrence matrices produces word embeddings (LSA/LSI). Also used for model compression by low-rank approximation of weight matrices.
Examples: Latent Semantic Analysis for document similarity, LoRA (low-rank adaptation) for LLM fine-tuning, denoising signals.
SVD is numerically stable, always exists. For tall matrices, use reduced SVD (m×n factors) for efficiency.
SVD = best-fit orthogonal decomposition. U and V are rotations, Σ scales. Smaller singular values = noise.
👉 Try this: Set rank=1 — image is monochrome blur. Rank=5 — edges start appearing. Rank=20 — objects recognizable. Rank=100 — nearly perfect. The first k components capture 95% of information in most images. This is why k~20-50 works well in practice.
import numpy as np

X = np.random.randn(100, 50)  # data matrix
U, s, Vt = np.linalg.svd(X, full_matrices=False)

# Rank-k approximation
k = 10
X_approx = U[:, :k] @ np.diag(s[:k]) @ Vt[:k, :]
error = np.linalg.norm(X - X_approx) / np.linalg.norm(X)

explained = np.sum(s[:k]**2) / np.sum(s**2)
print(f"Rank-{k} keeps {explained:.1%} of variance, error={error:.3f}")

SVD truncation is optimal low-rank approximation (Eckart-Young theorem). Used in PCA, recommender systems.

Key Insight: Test error decomposes into three parts: bias, variance, and irreducible error.
$$E[(y - \hat{f})^2] = \color{#c9a96e}{\text{Bias}(\hat{f})^2} + \color{#c9a96e}{\text{Var}(\hat{f})} + \sigma^2$$
Bias: Error from wrong modeling assumptions. E.g., fitting a line to curved data. High bias = underfitting.

Irreducible error \(\sigma^2\): Noise in labels. Can't reduce below this. Sets performance ceiling.
When to Use
High Bias (Underfitting)
Your model is too simple. Fix by: increasing model complexity, adding features, reducing regularization, or training longer.

High Variance (Overfitting)
Your model is too complex. Fix by: adding regularization (L1/L2/dropout), getting more data, reducing features, or early stopping.
Diagnosis: Train error is low but test error is high (large gap). Adding more training data helps. Cross-validation shows high variance across folds.

Balanced model complexity where test error is minimized. Use cross-validation to find it. Ensemble methods (bagging, boosting) can help balance bias-variance.
Strategy: Start simple, increase complexity until validation error stops improving. Use learning curves to diagnose. Regularization strength is your main knob.
Model complexity trade-off: Simple models: high bias, low variance. Complex models: low bias, high variance.
How to diagnose underfitting vs overfitting? High training and test error = underfitting (high bias). Low training, high test error = overfitting (high variance).

L1 vs L2 Regularization

The bowl is the training error — the bottom (w*) is where the model fits training data best. The colored fence is the regularization budget — the model must stay inside. Click and drag to rotate. Scroll to zoom. Use the slider to shrink the fence.

0.30
Key Insight: Regularization limits how large the model's weights can be, preventing overfitting.
$$\text{Loss} + \color{#c9a96e}{\lambda \cdot \sum |w_i|} \quad \text{(add L1 penalty)}$$
The bowl: The 3D surface is training error. The bottom of the bowl (w*) is where the model fits the training data best. Without regularization, the model goes straight to the bottom — but that often means extreme weight values that memorize noise (overfitting).
The fence: Regularization builds a fence around the origin (w=0). The model must stay inside. Increasing \(\lambda\) shrinks the fence, forcing the model further from the bowl's bottom but with smaller, more reasonable weights. The "answer" is the lowest point on the bowl that's still inside the fence.
Why L1 is special: L1's fence is a diamond with pointy corners that sit on the axes (where one weight = 0). The lowest point on the bowl inside the diamond often lands on a corner — so one weight gets forced to exactly zero. The model drops that feature entirely. That's automatic feature selection.
Why L2 is different: L2's fence is a circle with no corners. The lowest point inside the circle lands on the smooth edge — both weights shrink toward zero but neither actually reaches zero. All features stay, just dampened.
L1 (Lasso)
Sparsity, feature selection, interpretability. Unstable if correlated features.
👉 Try this: Toggle L1/L2. Set Regularization=0.5. With L1, weights snap to exactly zero at the diamond corners. With L2, weights shrink but rarely hit zero. For feature selection (want zeros), use L1. For shrinkage (want smaller), use L2.
L2 (Ridge)
Smooth shrinkage, stable with correlated features. Interpretable weights.
When to Use
L1 (Lasso)
When you suspect many features are irrelevant and want automatic feature selection. Produces sparse models where most weights are exactly zero.
Examples: Genomics (10,000 genes, few relevant), text classification with large vocabularies, any high-dimensional sparse problem.
L2 (Ridge)
When all features are likely relevant but you want to prevent large weights. Handles correlated features better than L1 by distributing weight among them.
Examples: Predicting house prices (all features matter), multicollinear data, neural network weight decay, any regression with many correlated predictors.
Elastic Net (L1 + L2)
Combines both: L1 for sparsity and L2 for stability with correlated features. Best of both worlds when you have groups of correlated features.
Examples: Gene selection with correlated gene groups, financial modeling with correlated factors, any scenario where pure L1 is unstable.
Use L1 for feature selection when you want sparse solutions. L2 when you want to keep all features but penalize large weights.
L1 = diamonds and sparsity. L2 = smooth shrinkage. Both prevent overfitting, but L1 also does feature selection.
from sklearn.linear_model import Lasso, Ridge
import numpy as np

X = np.random.randn(100, 20)  # 20 features, only 3 matter
true_w = np.zeros(20); true_w[:3] = [3, -2, 1]
y = X @ true_w + 0.1 * np.random.randn(100)

lasso = Lasso(alpha=0.1).fit(X, y)  # L1
ridge = Ridge(alpha=1.0).fit(X, y)  # L2

print(f"Lasso zeros: {np.sum(np.abs(lasso.coef_) < 0.01)}/20")  # ~17 zeros
print(f"Ridge zeros: {np.sum(np.abs(ridge.coef_) < 0.01)}/20")  # ~0 zeros

L1 (Lasso) produces sparse weights (feature selection). L2 (Ridge) shrinks all weights evenly.

Decision Boundaries: Linear vs Non-Linear

Toggle the classifier buttons to see different boundary shapes. For a crescent-shaped dataset, notice how Linear fails, Polynomial/RBF wrap around, and Trees create blocky boundaries. Which fits the data without overfitting?

Key Insight: Kernel trick enables high/infinite-dimensional computations without explicit feature vectors.
$$\text{If } K(x, x') = \varphi(x) \cdot \varphi(x'), \text{ then can work in feature space using } K$$
Core idea: Compute dot products in high-dimensional feature space \(\varphi(x) \cdot \varphi(x')\) without computing \(\varphi\) explicitly. Only need kernel K.
Examples: Linear: \(K(x,x')=x \cdot x'\). Polynomial: \((x \cdot x'+1)^d\). RBF: \(\exp(-\gamma\|x-x'\|^2)\). Each corresponds to implicit feature space.
Advantage: SVMs can handle infinite dimensions. Polynomial kernel = implicitly polynomial features without computing them.
Limitation: Kernel matrix is \(O(n^2)\) in memory. Only for kernelized algorithms (SVM, kernel regression, GP). Not all algorithms support kernels.
When to Use
Linear Classifiers (Logistic Reg / Linear SVM)
When data is linearly separable or high-dimensional (text, genomics). Fast to train, scales well to millions of samples. Good baseline before trying complex models.
Examples: Spam detection (bag-of-words features), sentiment analysis, medical diagnosis with many features, any problem where n_features >> n_samples.
Kernel SVM (RBF / Polynomial)
When decision boundary is non-linear and dataset is small-to-medium (under ~50K samples). RBF kernel is a strong default. Expensive for large datasets due to O(n^2) kernel matrix.
Examples: Handwritten digit classification, protein structure prediction, small tabular datasets with complex boundaries, any problem where data is clearly not linearly separable.
Decision Trees
When interpretability matters. Axis-aligned splits create blocky boundaries. Prone to overfitting alone, but powerful in ensembles (Random Forest, XGBoost).
Examples: Credit scoring (need to explain decisions), medical triage rules, any tabular data where you need human-readable logic.
Neural Networks
When you have large datasets and complex patterns. Learn arbitrary decision boundaries. Overkill for small tabular data but essential for images, text, audio.
Examples: Image classification, NLP tasks, speech recognition, any problem with >100K samples and complex non-linear patterns.
Kernel trick = avoid curse of dimensionality. RBF kernel in high-dim is powerful but computationally expensive. Always check kernel matrix size.
Kernel trick = compute dot products in feature space without computing features. Magic of SVMs and kernel methods.
👉 Try this: Select a non-linearly separable dataset (crescent shape). Linear fails (straight line can't wrap). Polynomial works well (curve wraps). RBF also works. Trees create blocky regions. For complex shapes, non-linear classifiers win.

Key Insight: K-fold CV splits data into K disjoint folds for robust generalization estimates.
$$\text{Repeat K times: } \color{#c9a96e}{\text{Train on K-1 folds}}\text{, test on held-out fold}$$
Process: 1. Split data into K disjoint folds. 2. Train on K-1, test on 1 fold. 3. Repeat K times. 4. Average K test errors.
Advantage: Uses all data for both training and evaluation. Robust to train/test split randomness. Reduces variance of estimate.

Stratified K-fold: For classification, maintain class distribution in each fold. Important for imbalanced datasets.
When to Use
K-Fold (K=5 or K=10)
Standard for model selection and hyperparameter tuning on tabular data. Good balance of computation and estimate quality. Use stratified K-fold for classification.
Examples: Comparing Random Forest vs. XGBoost, tuning regularization strength, Kaggle competitions, any model selection task.
Leave-One-Out (LOO)
When data is very scarce (n < 50). Each sample is a fold. Unbiased but high variance and computationally expensive (n model fits).
Examples: Rare disease studies with 20 patients, materials science with few expensive experiments, any tiny dataset where every sample matters.
Time-Series Split
For temporal data where standard K-fold leaks future information. Use expanding or sliding window: always train on past, test on future.
Examples: Stock price prediction, demand forecasting, weather modeling, any sequential data where temporal order matters.
Don't use K-fold for time series! Data leakage. Use expanding window or time-series split instead.
K-fold = multiple train/test splits averaged. Robust estimate of true generalization error. Standard in ML competitions.

Interview Tips: Understanding p-values and statistical significance is crucial.
p-value definition: Probability of observing test statistic as extreme as observed, under null hypothesis H₀ (no effect).
What p-value is NOT: NOT probability that H₀ is true. NOT probability your result is reproducible. NOT effect size or practical significance.
$$\text{Decision rule: p-value } \color{#c9a96e}{< \alpha} \text{ (typically 0.05)} \to \text{reject } H_0$$
Type I error (false positive): Reject \(H_0\) when it's true. Probability = \(\alpha\). Setting \(\alpha=0.05\) means 5% false positive rate.
Type II error (false negative): Fail to reject \(H_0\) when it's false. Probability = \(\beta\). Power = \(1 - \beta\).
When to Use
A/B Testing
Compare two variants (control vs. treatment) using a statistical test. Use t-test for continuous metrics, chi-squared for proportions. Always compute power before running the test.
Examples: Testing a new recommendation algorithm, comparing UI button colors on conversion rate, evaluating a new pricing strategy.
Model Comparison
Test whether Model A is significantly better than Model B. Use paired tests (paired t-test, Wilcoxon) on cross-validation scores to account for dataset correlation.
Examples: Is XGBoost significantly better than Random Forest? Does the new feature improve accuracy? Comparing AUC scores across K folds.
Multiple Testing Correction
When testing many hypotheses simultaneously, correct for false discovery rate (Benjamini-Hochberg) or family-wise error (Bonferroni). Critical in genomics and feature selection.
Examples: Testing 20,000 genes for significance, running multiple A/B tests simultaneously, feature selection with univariate tests.
Multiple comparisons: if you test 20 hypotheses at \(\alpha=0.05\), expect ~1 false positive. Use Bonferroni correction or false discovery rate control.
p-value = evidence against H₀ under assumption H₀ is true. Small p = unlikely data under H₀. But doesn't mean H₀ is false!
👉 Try this: Effect Size=0.5 (moderate effect). Power ≈ 0.6 (60% chance to detect). Increase to Effect Size=2 (huge effect). Power → 1 (almost certain to detect). Increase Sample Size (same effect size) → power increases. Larger effects and bigger samples = higher power.
from scipy import stats
import numpy as np

# Two-sample t-test: are these groups different?
group_a = np.random.normal(loc=100, scale=15, size=50)
group_b = np.random.normal(loc=108, scale=15, size=50)

t_stat, p_value = stats.ttest_ind(group_a, group_b)
print(f"t-statistic: {t_stat:.3f}, p-value: {p_value:.4f}")
print(f"Significant at alpha=0.05? {p_value < 0.05}")

# Effect size (Cohen's d)
cohens_d = (group_b.mean() - group_a.mean()) / np.sqrt((group_a.std()**2 + group_b.std()**2) / 2)
print(f"Cohen's d: {cohens_d:.2f}")

Always report effect size alongside p-value. Statistical significance != practical significance.

Interview Tips: Monte Carlo sampling is fundamental for intractable expectations.
$$E[f(X)] = \int f(x)p(x)dx \approx \color{#c9a96e}{\frac{1}{n} \cdot \sum f(x_i)} \quad \text{where } x_i \sim p(x)$$
Algorithm: 1. Sample \(x_1,\ldots,x_n\) from \(p(x)\). 2. Compute average \(\hat{f} = \frac{1}{n} \cdot \sum f(x_i)\). 3. \(\hat{f} \to E[f(X)]\) as \(n \to \infty\) (Law of Large Numbers).
Convergence rate: \(O(1/\sqrt{n})\). Very slow! Need \(n^2\) samples to halve error. Curse of dimensionality.
Variance reduction: Importance sampling, control variates, stratified sampling reduce variance per sample.
Application: Computing partition functions in graphical models, expectations under intractable posteriors in Bayesian inference.
When to Use
Bayesian Inference
When posterior distributions are intractable analytically. MCMC (Markov Chain Monte Carlo) samples from complex posteriors. Gives full uncertainty estimates, not just point estimates.
Examples: Bayesian neural networks, hierarchical models, Gaussian processes with non-conjugate likelihoods, probabilistic programming (Stan, PyMC).
Simulation & Integration
When you need to estimate expectations or integrals that have no closed form. Embarrassingly parallel -- scales well with compute.
Examples: Estimating pi (as shown), option pricing in finance, particle physics simulations, computing normalizing constants.
Reinforcement Learning
Monte Carlo methods estimate value functions by averaging returns from sampled episodes. No model of the environment needed (model-free).
Examples: Policy evaluation in game playing, Monte Carlo Tree Search (AlphaGo), estimating Q-values from rollouts, exploration strategies.
Slow convergence but embarrassingly parallel. Useful when analytical integral is intractable. Always consider variance reduction.
Monte Carlo = approximate integral by sampling. Simple, general, slow (\(O(1/\sqrt{n})\)). Variance reduction is key.
👉 Try this: Generate 100 samples — estimate bounces around π. Generate 1000 samples (click 10 times) — estimate settles. Generate 10,000 samples — estimate ≈ 3.14159. Sampling error shrinks as \(O(1/\sqrt{n})\). Quadruple samples to halve error.
import numpy as np

def estimate_pi(n_samples):
    x = np.random.uniform(-1, 1, n_samples)
    y = np.random.uniform(-1, 1, n_samples)
    inside_circle = (x**2 + y**2) <= 1
    return 4 * inside_circle.mean()

for n in [100, 1000, 10000, 100000]:
    pi_est = estimate_pi(n)
    print(f"n={n:>6d}: pi ≈ {pi_est:.5f}, error={abs(pi_est - np.pi):.5f}")

Monte Carlo error scales as O(1/sqrt(n)). Quadruple samples to halve the error.

Numerical Stability: Log-Sum-Exp Trick

Set "Temperature" to 1 and increase the logit values — the blue (naive) softmax overflows to NaN, but red (stable) softmax remains valid. The trick: subtract the max before exponentiating. Why? To keep exponents in a safe range.

1.0
100
Interview Tips: Log-sum-exp trick is essential for numerical stability in machine learning.
$$\log\left(\sum \exp(x_i)\right) = \color{#c9a96e}{m + \log\left(\sum \exp(x_i - m)\right)} \quad \text{where } m = \max(x_i)$$
The problem: Direct computation exp(x) overflows when x is large. E.g., exp(1000) exceeds float64 range.
The trick: Subtract max before exponentiating. exp(xᵢ - m) stays bounded in [0,1]. Then add log(m) back.
Application: Softmax: log softmax(xᵢ) = xᵢ - logsumexp(x). Log-likelihood in classification. Gradient computation.
Equivalent form: \(\text{logsumexp}(x) = m + \log(\sum \exp(x - m))\). PyTorch: torch.logsumexp(). NumPy: scipy.special.logsumexp().
When to Use
Log-Sum-Exp for Softmax
Always use when computing softmax or log-softmax. Without this trick, large logits cause overflow (exp(1000) = Inf) and small ones cause underflow (exp(-1000) = 0).
Examples: Classification output layers, attention score computation in transformers, mixture model responsibilities, any softmax computation.
Log-Space Probability
Always work in log-space when multiplying many probabilities. Products of small numbers underflow to zero; sums of logs stay stable.
Examples: HMM forward/backward algorithms, language model perplexity, Bayesian model evidence computation, sequence probability in NLP.
Gradient Clipping
Another numerical stability technique: cap gradient magnitude to prevent exploding gradients. Essential for RNNs and deep networks.
Examples: LSTM/GRU training, transformer training, any deep network with >20 layers, reinforcement learning with unstable rewards.
Always use log-space computation for probabilities and likelihoods. Never compute softmax as \(\exp(x)/\sum\exp(x)\) directly.
Log-sum-exp = prevent overflow/underflow by shifting to max. Subtract max, exponentiate safely, add max back in log space.
👉 Try this: Set logits to [0, 1, 2] with Temperature=1. Both softmax versions work. Increase logits to [0, 100, 200]. Blue softmax overflows to NaN (exp(200) is huge!). Red (stable) softmax gives [~0, ~0, ~1]. Subtract max first, always.
import numpy as np

def stable_softmax(logits):
    shifted = logits - np.max(logits)  # subtract max for stability
    exp_shifted = np.exp(shifted)
    return exp_shifted / exp_shifted.sum()

logits = np.array([0, 100, 200])  # would overflow without trick!
# np.exp(200) = inf, but stable_softmax handles it
probs = stable_softmax(logits)
print(f"Softmax: {probs}")  # [0, 0, 1] approximately
print(f"Log-sum-exp: {np.max(logits) + np.log(np.sum(np.exp(logits - np.max(logits)))):.2f}")

Always subtract max before exp(). This is built into torch.nn.functional.log_softmax.
