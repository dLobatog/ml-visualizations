Collaborative Filtering: Matrix Factorization

Slide k (latent dimensions) from 1 to 50. Watch the reconstruction of the rating matrix improve—at k=1, barely captures patterns; at k=50, reconstructs almost perfectly. Notice the rmse metric: higher k = better reconstruction. But too high k risks overfitting. Find the sweet spot around k=10–20.

5
Reconstruction Error: --
Key Insight: Singular Value Decomposition decomposes user-item matrix R into low-rank factors U, Σ, V, enabling scalable collaborative filtering despite sparse data.
$$R \approx U \cdot \Sigma \cdot V^T, \text{ where } \|R - \color{#ff9ff3}{U_k \cdot \Sigma_k \cdot V_k^T}\|^2 \text{ is minimized for rank-k approximation}$$
\(R\):User-item interaction matrix (m users × n items), sparse with mostly missing entries (0.01-1% observed)
\(U\):User embedding matrix (m × k), each row is a user's latent preference vector in k-dimensional space
\(\Sigma\):Singular values diagonal matrix (k × k), ordered by magnitude; higher values indicate more important latent factors
\(V^T\):Item embedding matrix transposed (k × n), each column is an item's latent factor vector
k:Latent dimension (rank of approximation), typically 64-512 in production; controls model capacity and latency trade-off
Data property:Netflix-scale systems have only ~0.01-1% observed entries; explicit SVD cannot handle missing data
Parameter reduction:From m·n (dense) to (m+n)·k (low-rank); k controls model capacity and latency trade-off
Algorithms:SGD or Alternating Least Squares (ALS) instead of explicit eigendecomposition
Production detail:L2 regularization \(\lambda\|U\|^2 + \lambda\|V\|^2\) prevents overfitting; normalize random initialization
👉 Try this: Slide k from 1 to 50. At k=1, reconstruction is poor (RMSE high). At k=10–15, quality plateaus—diminishing returns. At k=50, you're overfitting (perfect train but bad test). The sweet spot is k=10–20. Also watch rank: smaller k = simpler model = generalizes better. This is the bias-variance tradeoff in dimensionality reduction.
Why does sparse data require different algorithms? Explicit SVD solvers (eigendecomposition) cannot handle missing values; SGD/ALS gracefully skip unobserved entries by computing only on observed pairs.
How do you choose k? Via cross-validation: lower k is faster at serving (fewer parameters) but less accurate; higher k improves NDCG but increases latency—typical production range is k=64-512.
Think of SVD as embedding users and items in a shared k-dimensional space; similar users have similar embedding vectors, so their preferences are correlated.

Top-3 Recommendations: --
Key Insight: Two-Tower (TT) decouples user and item feature processing, enabling pre-computed item embeddings and fast ANN-based retrieval at serving time.
$$\text{Score}(u, i) = u \cdot i, \text{ where } u \in \mathbb{R}^d \text{ (user embedding) and } i \in \mathbb{R}^d \text{ (item embedding) via independent towers}$$
Score(u, i):Relevance score for user u and item i; higher scores indicate better recommendations; used for ranking candidates
u:User embedding vector (d-dimensional), output of user tower; learned from user features (ID, demographics, history)
i:Item embedding vector (d-dimensional), output of item tower; learned from item features (metadata, content, category)
d:Embedding dimension, typically 32-256 in production; controls model capacity and serving latency
· (dot product):Inner product measures alignment between user and item preferences in shared d-dimensional embedding space
Architecture:Separate neural networks for user/item features (ID, demographics, metadata); each tower learns independent dense embeddings
Training loss:Softmax triplet loss or in-batch negatives to push relevant items closer and irrelevant items farther
Serving advantage:Pre-compute all item embeddings offline, index in HNSW/LSH/IVF for sub-millisecond retrieval over millions of items
Limitation:Cannot capture fine-grained user-item interactions (specific feature combinations); requires ranking model for precision
👉 Try this: Toggle "Exact vs Approximate Search". Exact computes dot product for all items (slow, accurate). Approximate uses HNSW indexing (fast, slight accuracy loss). Watch latency: exact scales with catalog size (100K items = 100ms), approximate stays 1–5ms. Then increase Embedding Dimension to 256. Exact gets slower, approximate barely changes. This is why two-tower dominates.
Why pre-compute item embeddings? User tower requires online inference (live user context), but item embeddings are static—pre-computing and indexing eliminates real-time item inference cost.
Why dot product over cosine? Cosine similarity requires L2 normalization at serving time; dot product is faster in FAISS/Annoy libraries and achieves similar ranking when embeddings are approximately normalized.
Two-tower is like two separate coordinate systems: users and items map independently into shared embedding space, then you find nearest neighbors via simple dot product.

Key Insight: Wide & Deep hybrid combines memorization (linear wide path) and generalization (deep neural path) to handle both frequent interactions and unseen feature combinations.
Wide (Linear)
Hand-crafted cross-product features (user_id × item_id, category × region); memorizes low-order interactions perfectly but poor generalization to unseen combos
Deep (Neural)
Embeddings + ReLU layers learn implicit nonlinear patterns; generalizes well but may miss important low-order feature interactions
$$L = \alpha \cdot L_{\text{wide}} + (1-\alpha) \cdot L_{\text{deep}}, \text{ balancing memorization and generalization } (\alpha \text{ typically } 0.5)$$
\(L\):Total training loss to optimize; weighted sum of wide and deep path losses
\(L_{\text{wide}}\):Cross-entropy or logistic loss from wide path (linear model); penalizes incorrect memorized patterns
\(L_{\text{deep}}\):Cross-entropy or logistic loss from deep path (neural network); penalizes incorrect generalized patterns
\(\alpha\):Weight balancing memorization vs generalization; \(\alpha \in [0, 1]\); \(\alpha=0.5\) gives equal importance; tune via validation
Serving latency:Wide path (linear) adds minimal cost; deep path dominates computation
Regularization:L2 on wide weights (prevent overfit to specific user-item pairs); dropout on deep layers (prevent co-adaptation)
Shared embeddings:Both paths share embedding tables; deep path learns better representations, improving wide path

Think of it as: Wide handles special cases (memorize important user-item patterns), Deep handles the general case (learn reusable patterns from raw features).

DCN (Deep & Cross Network): Automatic Feature Crossing

Click 'Expand Cross Layers' to see how each layer computes feature crosses (feature × feature interactions). Watch the cross pattern at layer 1 (pairwise features) vs layer 3 (higher-order). Drag 'Num Cross Layers' to balance expressiveness vs compute. More layers = higher-order interactions but slower training.

3
Cross Layers: -- | Deep Layers: 4
Key Insight: Deep & Cross Network (DCN) automates feature crossing without exponential explosion: each cross layer generates polynomial order 2^l interactions with only O(d) parameters per layer.
$$x_{l+1} = x_0 \odot \color{#ff9ff3}{(w_l^T x_l + b_l)} + x_l, \text{ where } \odot \text{ is element-wise multiplication and } x_0 \text{ is original input}$$
\(x_{l+1}\):Output of l-th cross layer; contains polynomial feature interactions up to order \(2^l\)
\(x_0\):Original input feature vector to first cross layer; preserved and used in residual connection throughout network
\(x_l\):Input to l-th cross layer; output of previous cross layer (or \(x_0\) if l=1)
\(w_l\):Weight vector for l-th cross layer (d-dimensional); learns linear projection of input \(x_l\) to scalar
\(b_l\):Bias term for l-th cross layer (scalar); adds offset to weighted sum \(w_l^T x_l\)
\(\odot\):Element-wise (Hadamard) multiplication; gates original input \(x_0\) by learned projection \((w_l^T x_l + b_l)\)
l:Cross layer index starting from 0; l-th layer produces 2^l-order feature interactions
\(d\):Feature dimension; same throughout network; parameter count per layer is \(O(d)\), not \(O(d^2)\)
vs Wide & Deep:Wide requires manual feature engineering (expensive); DCN learns feature crosses automatically
Cross layer output:l-th layer produces polynomial order \(2^l\) (Layer 1: 2nd order, Layer 2: 4th order, etc.); exponential expressivity with linear parameters
Architecture:Cross network (automatic feature crosses) + Deep network (embeddings + MLPs) in parallel; concatenate outputs for final classification layer
Serving cost:Minimal latency overhead (simple dot products and element-wise ops); ideal for production constraints
👉 Try this: Increase "Cross Layers" from 1 to 5. Watch Cross Network learn feature interactions. Layer 1 learns pairwise crosses (age × gender). Layer 2 learns order-3 crosses. Layer 4+ very high-order. Increase too much and you overfit. Slide "DCN Strength" to see regularization effect. The cross network is remarkably sample-efficient—beats traditional feature engineering with 10x fewer parameters.
Why more efficient than hand-crafted crosses? Exponential feature explosion: to get 4th-order interactions manually, you need \(O(d^4)\) features. DCN achieves it with \(O(4d)\) parameters via automatic layer-wise composition.
Interpretability trade-off? Unlike Wide & Deep (you can inspect wide weights to see which crosses matter), DCN cross layers are learned implicitly—harder to debug but more automated.
Think of cross layers as a learnable feature interaction funnel: each layer multiplies original input by learned gates to extract higher-order patterns progressively.

DIN: Deep Interest Network (Attention Mechanism)

Click 'Show User History' to see a user's past items. Now select a query item and watch the attention weights appear—recent items and similar items get higher weight. Toggle 'Temporal Weight' to see how recency affects attention. Items from 1 week ago might weight 10x more than items from 1 year ago.

10
Attention Weights: --
Key Insight: Attention computes dynamic relevance weights for historical items, capturing user's multi-faceted interests as weighted aggregation.
$$I = \sum_i \alpha_i \cdot \text{embed}(\text{item}_i), \text{ where } \alpha_i = \text{softmax}(\text{score}(\text{item}_i, \text{candidate}))$$
\(I\):Final user interest representation (d-dimensional); weighted average of historical item embeddings relevant to candidate
\(\sum_i\):Sum over all i historical items in user's sequence (i = 1..sequence_length)
\(\alpha_i\):Attention weight for item i; \(\alpha_i \in [0,1]\) and \(\sum \alpha_i = 1\); higher if item i is relevant to candidate
\(\text{embed}(\text{item}_i)\):Embedding vector (d-dimensional) of historical item i; learned during training
\(\text{score}(\text{item}_i, \text{candidate})\):Raw relevance score (before softmax); typically dot product or learned bilinear similarity between historical and candidate embeddings
\(\text{softmax}(\cdot)\):Converts scores to normalized weights; ensures \(\alpha_i \geq 0\) and \(\sum \alpha_i = 1\); emphasizes high-scoring items exponentially
candidate:Query item being ranked; all attention weights computed relative to this single candidate
Scoring function:Typically dot product or learned bilinear similarity between historical item and candidate embeddings
Dynamic weighting:Candidate item "pulls" relevant historical items to the forefront; users with diverse interests (action + documentaries) are naturally handled
vs Average pooling:Average treats all history equally (loses context); attention weights history adaptively based on candidate
Computational cost:O(sequence_length) per candidate; can be prohibitive if sequences are long (100+ items)
Extensions:Multi-head attention (k separate relevance functions); Transformers (parallel all-pairs attention); attention-over-attention (hierarchical)
👉 Try this: Click "Show User History" to see past items. Select a query item. Watch attention weights light up—which past items influence this prediction? Toggle "Temporal Weighting". With temporal weighting ON, recent items dominate (1 week ago = 10x weight of 1 month ago). Toggle OFF, uniform weighting. Notice predictions change: temporal is more accurate for short-term trends.
Why better than average? Example: User watched [Romance, Romance, Romance, Action]. Average gives little weight to Action preference. Attention on Action candidate upweights the Action history item.
Recency bias emerges naturally: if items are ordered temporally, softmax attention over recent items (higher scores) gives more weight to recent history without explicit time decay.
Attention = Adaptive history summarization: different candidates pull different "summaries" from the same user history based on relevance.

Key Insight: Transformers via self-attention capture long-range item dependencies without sequential bottleneck; each item attends to all prior items in sequence.
$$\text{Attention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^T}{\color{#ff9ff3}{\sqrt{d}}}\right)V$$
\(Q\) (Query):Query matrix (sequence_length × d); projections of input embeddings; represents "what to attend to"
\(K\) (Key):Key matrix (sequence_length × d); projections of input embeddings; represents "what can be attended"
\(V\) (Value):Value matrix (sequence_length × d); projections of input embeddings; represents "information to aggregate"
\(QK^T\):Matrix multiplication (sequence_length × sequence_length); each element [i,j] = dot product of query i and key j
\(d\):Embedding dimension; typically 64 per head in multi-head attention; impacts gradient scale
\(\sqrt{d}\) (scaling factor):Divides attention scores by \(\sqrt{d}\); stabilizes gradients (prevents very large/small dot products); ensures softmax doesn't saturate
softmax(·):Converts scaled scores to normalized attention weights; each row sums to 1; emphasizes high-scoring positions
softmax(QK^T/√d)V:Final output (sequence_length × d); weighted sum of values by attention weights; each position attends to all previous positions
Mechanism:Each position i attends to all previous positions [1..i-1] (causal masking prevents future leakage); item representation = weighted sum of all prior items
vs RNNs/LSTMs:Parallel training (no sequential bottleneck), better gradient flow (no vanishing gradients), scales to longer sequences (100+ items)
Learned patterns:Item transitions (shoe → sock → laces); recency bias (softmax places higher weight on recent items) emerges naturally
Engineering details:Position embeddings (absolute or relative) encode temporal order; layer-normalization stabilizes; multi-head attention learns k separate relevance functions

Key Insight: Multi-Task Learning optimizes correlated objectives (CTR, CVR, watch-time) jointly; Multi-gate Mixture of Experts (MMOE) allows task-specific expert selection via gating.
$$\text{Task}_k\text{ output} = \text{gate}_k \otimes \sum_i \color{#ff9ff3}{\text{expert}_i}, \text{ where gate}_k = \text{softmax}(W_k \cdot \text{features})$$
\(\text{Task}_k\) output:Final prediction for task k (CTR, CVR, watch-time, etc.); vector of shape (batch_size, output_dim)
\(\text{gate}_k\):Gating vector for task k (m-dimensional); softmax weights over m experts; emphasizes task-relevant expert representations
\(\sum_i \text{expert}_i\):Sum of all expert outputs; each expert i is a small MLP producing (batch_size, output_dim) representation
m:Number of shared experts; typical range 4-10 in production; each expert is a small MLP (2-3 hidden layers)
\(\otimes\):Broadcasting multiplication; \(\text{gate}_k\) (m-dimensional) multiplies across experts dimension of \(\sum_i \text{expert}_i\)
W_k:Weight matrix for task k's gating network; shapes (feature_dim, m); learns task-specific expert preferences
features:Input feature vector (user context, item features, etc.); shared across all experts and gating networks
softmax(W_k · features):Normalized weights over m experts; ensures gate_k sums to 1 and all weights in [0,1]
k:Task index; separate gating networks for each task (CTR, CVR, etc.); experts shared across all tasks
Objectives:CTR (click), CVR (conversion), watch time, engagement—all correlated; joint training improves data efficiency and calibration
MoE architecture:m shared experts (small MLPs, e.g., m=6); per-task gating networks learn softmax weights over experts; task output = weighted sum
Benefits:Parameter efficiency (shared experts << task-specific towers); mutual information between tasks improves generalization; flexible task importance weighting
Joint loss:\(L = \alpha \cdot L_{\text{ctr}} + \beta \cdot L_{\text{cvr}} + \gamma \cdot L_{\text{watch}}\), where \(\alpha, \beta, \gamma\) reflect business priorities (tune on validation set)
Production gotchas:Task calibration (different loss scales → gradient dominance); label delay (CVR takes days, CTR immediate); negative transfer (tasks conflict)

Learning to Rank (LTR): Loss Function Comparison

Click the three tabs to see each loss function. Pointwise (simplest): predict score for each doc independently. Pairwise: learns ranking ('doc A > doc B'), more complex. Listwise (most complex): optimizes entire ranking order. Compare loss curves—listwise is harder to optimize but produces better rankings when it converges.

10
Mode: Pairwise | Loss: --
Key Insight: Learning to Rank (LTR) optimizes ranking metrics (NDCG, MRR) directly; pointwise ignores ranking structure; pairwise captures pairs; listwise optimizes full ranking.
Pointwise
Treat each document independently; loss = classification_loss. Simple but ignores ranking structure and position bias.
Pairwise
Minimize inversion loss for pairs: \(\log(1 + \exp(-\sigma(s_i - s_j)))\). Captures ranking structure; \(O(n^2)\) pairs per example.
Listwise:Direct NDCG optimization: loss = -NDCG(ranked_list). Most principled but requires sorting (expensive); LambdaMART combines with pairwise for efficiency
LambdaMART:Computes pairwise gradients but weights by NDCG improvement (lambda gradient weighting); efficient listwise optimization
Accuracy vs Computation:Pointwise (fast) << Pairwise (medium) << Listwise (slow but best); production typically uses LambdaMART via XGBoost/LightGBM

Why does pointwise fail on ranked outputs? Position bias: top-ranked docs receive more clicks regardless of relevance. Evaluation metric mismatch: classification loss doesn't align with NDCG/MRR metrics.
Variable-length lists: Pad with negative labels (rank 0) to uniform length. LambdaMART handles missing labels gracefully (skips pairs with missing labels).
Pointwise = per-item scores ignoring context; Pairwise = respecting relative order; Listwise = optimizing the entire ranked order for metric.

Multi-touch attribution: Why harder? Multiple touchpoints in user journey; isolating which ad caused conversion is confounded by user intent. Shapley value fair but computationally expensive.
Think of auction equation as: (Advertiser value) × (User relevance) × (Platform quality); all three terms must be non-zero for ads to rank well.

Click 'Bi-Encoder': embeds query and doc independently, scores by dot product—fast. Click 'Cross-Encoder': scores query+doc jointly—slow but more accurate. Click 'ColBERT': embeds independently but scores via late interaction (token-level matching)—best of both. See the accuracy vs speed tradeoff: Bi-Encoder fast, Cross-Encoder accurate, ColBERT balanced.

Key Insight: Dense retrieval encodes queries/documents as embeddings; bi-encoder is fast but coarse; cross-encoder is accurate but slow; ColBERT bridges both.
Bi-encoder
query_emb = BERT(query), doc_emb = BERT(doc); score = cosine(q, d). Pre-index all docs in FAISS/Annoy; O(1) serving cost.
Cross-encoder
score = BERT([query, doc]); direct relevance. O(n) serving cost (re-encode each candidate). Accurate but expensive.
ColBERT (late interaction):Per-token embeddings; score = max_{i,j} cos(q_i, d_j); defers fine-grained matching to serving without full re-encoding
vs BM25:BM25 lexical matching cheap and handles rare terms; dense retrieval semantic matching better for paraphrases and out-of-vocabulary
Production hybrid:BM25 (cheap retrieval) → bi-encoder re-rank (efficient) → cross-encoder/ColBERT refine top-k (accurate); multi-signal fusion
Cold-start new docs:Compute embeddings from content (metadata, title, body); no user interaction history required; fine-tune as signals accumulate

For candidate gen, use Bi-Encoder. For reranking top-100, use Cross-Encoder. ColBERT is sweet spot.
Why does ColBERT avoid early-interaction bottleneck? Standard dense matching locks in interaction decision at embedding time; ColBERT defers to serving (per-token flexibility).
Measure retrieval quality: recall@k (fraction of relevant docs in top-k), MRR (mean reciprocal rank of first relevant doc), NDCG. Validate on holdout relevance judgments.
