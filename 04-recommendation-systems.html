Key Insight: Singular Value Decomposition decomposes user-item matrix R into low-rank factors U, Σ, V, enabling scalable collaborative filtering despite sparse data.
$$R \approx U \cdot \Sigma \cdot V^T, \text{ where } \|R - \color{#ff9ff3}{U_k \cdot \Sigma_k \cdot V_k^T}\|^2 \text{ is minimized for rank-k approximation}$$
\(R\):User-item interaction matrix (m users × n items), sparse with mostly missing entries (0.01-1% observed)
\(U\):User embedding matrix (m × k), each row is a user's latent preference vector in k-dimensional space
\(\Sigma\):Singular values diagonal matrix (k × k), ordered by magnitude; higher values indicate more important latent factors
\(V^T\):Item embedding matrix transposed (k × n), each column is an item's latent factor vector
k:Latent dimension (rank of approximation), typically 64-512 in production; controls model capacity and latency trade-off
Data property:Netflix-scale systems have only ~0.01-1% observed entries; explicit SVD cannot handle missing data
Parameter reduction:From m·n (dense) to (m+n)·k (low-rank); k controls model capacity and latency trade-off
Algorithms:SGD or Alternating Least Squares (ALS) instead of explicit eigendecomposition
Production detail:L2 regularization \(\lambda\|U\|^2 + \lambda\|V\|^2\) prevents overfitting; normalize random initialization
When to Use
SVD / Matrix Factorization
Best when you have a large, sparse user-item interaction matrix with explicit ratings. Classic use: Netflix movie ratings, Amazon product ratings. Works well when item catalog is stable and you need interpretable latent factors (e.g., genre, mood).
Implicit Feedback ALS
Use when you only have implicit signals (clicks, views, purchases) without explicit ratings. Classic use: Spotify play counts, YouTube watch history. ALS scales better on distributed systems (Spark MLlib) for billion-scale interactions.
👉 Try this: Slide k from 1 to 50. At k=1, reconstruction is poor (RMSE high). At k=10–15, quality plateaus—diminishing returns. At k=50, you're overfitting (perfect train but bad test). The sweet spot is k=10–20. Also watch rank: smaller k = simpler model = generalizes better. This is the bias-variance tradeoff in dimensionality reduction.
Why does sparse data require different algorithms? Explicit SVD solvers (eigendecomposition) cannot handle missing values; SGD/ALS gracefully skip unobserved entries by computing only on observed pairs.
How do you choose k? Via cross-validation: lower k is faster at serving (fewer parameters) but less accurate; higher k improves NDCG but increases latency—typical production range is k=64-512.
Think of SVD as embedding users and items in a shared k-dimensional space; similar users have similar embedding vectors, so their preferences are correlated.
import torch.nn as nn

class MF(nn.Module):
    def __init__(self, n_users, n_items, dim=32):
        super().__init__()
        self.user_emb = nn.Embedding(n_users, dim)
        self.item_emb = nn.Embedding(n_items, dim)
        self.user_bias = nn.Embedding(n_users, 1)
        self.item_bias = nn.Embedding(n_items, 1)

    def forward(self, user, item):
        dot = (self.user_emb(user) * self.item_emb(item)).sum(-1)
        return dot + self.user_bias(user).squeeze() + self.item_bias(item).squeeze()

Key Insight: Two-Tower (TT) decouples user and item feature processing, enabling pre-computed item embeddings and fast ANN-based retrieval at serving time.
$$\text{Score}(u, i) = u \cdot i, \text{ where } u \in \mathbb{R}^d \text{ (user embedding) and } i \in \mathbb{R}^d \text{ (item embedding) via independent towers}$$
Score(u, i):Relevance score for user u and item i; higher scores indicate better recommendations; used for ranking candidates
u:User embedding vector (d-dimensional), output of user tower; learned from user features (ID, demographics, history)
i:Item embedding vector (d-dimensional), output of item tower; learned from item features (metadata, content, category)
d:Embedding dimension, typically 32-256 in production; controls model capacity and serving latency
· (dot product):Inner product measures alignment between user and item preferences in shared d-dimensional embedding space
Architecture:Separate neural networks for user/item features (ID, demographics, metadata); each tower learns independent dense embeddings
Training loss:Softmax triplet loss or in-batch negatives to push relevant items closer and irrelevant items farther
Serving advantage:Pre-compute all item embeddings offline, index in HNSW/LSH/IVF for sub-millisecond retrieval over millions of items
Limitation:Cannot capture fine-grained user-item interactions (specific feature combinations); requires ranking model for precision
When to Use
Two-Tower for Candidate Generation
Use as Stage 1 retrieval when catalog is massive (millions of items). Classic use: YouTube video candidate generation, Pinterest pin retrieval, LinkedIn job recommendations. Pre-compute item embeddings offline and serve via ANN index (FAISS/ScaNN).
Single-Tower / Cross-Attention for Ranking
Use when you need fine-grained feature interactions between user and item (Stage 2 ranking). Classic use: Netflix final ranking of 100 candidates, Amazon product detail page "similar items". Slower but more precise than two-tower.
👉 Try this: Toggle "Exact vs Approximate Search". Exact computes dot product for all items (slow, accurate). Approximate uses HNSW indexing (fast, slight accuracy loss). Watch latency: exact scales with catalog size (100K items = 100ms), approximate stays 1–5ms. Then increase Embedding Dimension to 256. Exact gets slower, approximate barely changes. This is why two-tower dominates.
Why pre-compute item embeddings? User tower requires online inference (live user context), but item embeddings are static—pre-computing and indexing eliminates real-time item inference cost.
Why dot product over cosine? Cosine similarity requires L2 normalization at serving time; dot product is faster in FAISS/Annoy libraries and achieves similar ranking when embeddings are approximately normalized.
Two-tower is like two separate coordinate systems: users and items map independently into shared embedding space, then you find nearest neighbors via simple dot product.
import torch.nn as nn

class TwoTower(nn.Module):
    def __init__(self, user_feat_dim, item_feat_dim, emb_dim=64):
        super().__init__()
        self.user_tower = nn.Sequential(
            nn.Linear(user_feat_dim, 128), nn.ReLU(), nn.Linear(128, emb_dim))
        self.item_tower = nn.Sequential(
            nn.Linear(item_feat_dim, 128), nn.ReLU(), nn.Linear(128, emb_dim))

    def forward(self, user_feat, item_feat):
        u = nn.functional.normalize(self.user_tower(user_feat), dim=-1)
        i = nn.functional.normalize(self.item_tower(item_feat), dim=-1)
        return (u * i).sum(-1)  # cosine similarity

Separate user and item towers produce normalized embeddings. Item embeddings are precomputed for fast ANN retrieval.

import torch.nn as nn

class HybridRecommender(nn.Module):
    def __init__(self, n_users, n_items, content_dim, emb_dim=32):
        super().__init__()
        self.user_emb = nn.Embedding(n_users, emb_dim)
        self.item_emb = nn.Embedding(n_items, emb_dim)
        self.content_proj = nn.Linear(content_dim, emb_dim)  # for cold items

    def forward(self, user, item, content_feat, is_cold):
        u = self.user_emb(user)
        i = torch.where(is_cold.unsqueeze(-1),
                        self.content_proj(content_feat), self.item_emb(item))
        return (u * i).sum(-1)

Falls back to content features for new items with no interactions. Warm items use learned collaborative embeddings.

Key Insight: Wide & Deep hybrid combines memorization (linear wide path) and generalization (deep neural path) to handle both frequent interactions and unseen feature combinations.
Wide (Linear)
Hand-crafted cross-product features (user_id × item_id, category × region); memorizes low-order interactions perfectly but poor generalization to unseen combos
Deep (Neural)
Embeddings + ReLU layers learn implicit nonlinear patterns; generalizes well but may miss important low-order feature interactions
$$L = \alpha \cdot L_{\text{wide}} + (1-\alpha) \cdot L_{\text{deep}}, \text{ balancing memorization and generalization } (\alpha \text{ typically } 0.5)$$
\(L\):Total training loss to optimize; weighted sum of wide and deep path losses
\(L_{\text{wide}}\):Cross-entropy or logistic loss from wide path (linear model); penalizes incorrect memorized patterns
\(L_{\text{deep}}\):Cross-entropy or logistic loss from deep path (neural network); penalizes incorrect generalized patterns
\(\alpha\):Weight balancing memorization vs generalization; \(\alpha \in [0, 1]\); \(\alpha=0.5\) gives equal importance; tune via validation
Serving latency:Wide path (linear) adds minimal cost; deep path dominates computation
Regularization:L2 on wide weights (prevent overfit to specific user-item pairs); dropout on deep layers (prevent co-adaptation)
Shared embeddings:Both paths share embedding tables; deep path learns better representations, improving wide path
When to Use
Wide & Deep
Use when you have known important feature crosses and want both memorization and generalization. Classic use: Google Play app recommendations (memorize "user installed game X, show game Y" + generalize from app category embeddings). Best when domain experts can identify key feature crosses.
Deep-Only Models
Use when feature engineering is too expensive or catalog changes rapidly. Classic use: YouTube video recommendations (massive feature space, too many crosses to enumerate). Simpler architecture, easier to maintain, but may miss specific memorization patterns.
👉 Try this: Drag features into model. Put "user_id" and "item_id" in Wide path (memorization—remember exact user-item pairs). Put "user_age" and "item_category" in Deep path (learn embeddings). Watch prediction change. Remove features from Wide, add to Deep—prediction becomes more generalized but misses specific user patterns. 60/40 split (by param count) usually works well.
How do you choose wide-path features? High-dimensional, expensive to hand-engineer. Start with top-k frequent feature crosses (identified via statistical tests), then iteratively add new crosses based on model analysis.
Compare to baselines: Deep-only struggles with memorization (e.g., specific promo combos); Wide-only cannot generalize. Wide&Deep achieves both—validate via A/B test: Deep-only vs Wide-only vs Wide&Deep.
Think of it as: Wide handles special cases (memorize important user-item patterns), Deep handles the general case (learn reusable patterns from raw features).
import torch.nn as nn

class WideAndDeep(nn.Module):
    def __init__(self, wide_dim, deep_dims, hidden=128):
        super().__init__()
        self.wide = nn.Linear(wide_dim, 1)  # memorization
        self.deep = nn.Sequential(
            nn.Linear(deep_dims, hidden), nn.ReLU(),
            nn.Linear(hidden, hidden), nn.ReLU(),
            nn.Linear(hidden, 1))  # generalization

    def forward(self, wide_x, deep_x):
        return torch.sigmoid(self.wide(wide_x) + self.deep(deep_x))

Wide component memorizes feature crosses; deep component generalizes via embeddings. Used at Google Play.

DCN (Deep & Cross Network): Automatic Feature Crossing

Key Insight: Deep & Cross Network (DCN) automates feature crossing without exponential explosion: each cross layer generates polynomial order 2^l interactions with only O(d) parameters per layer.
$$x_{l+1} = x_0 \odot \color{#ff9ff3}{(w_l^T x_l + b_l)} + x_l, \text{ where } \odot \text{ is element-wise multiplication and } x_0 \text{ is original input}$$
\(x_{l+1}\):Output of l-th cross layer; contains polynomial feature interactions up to order \(2^l\)
\(x_0\):Original input feature vector to first cross layer; preserved and used in residual connection throughout network
\(x_l\):Input to l-th cross layer; output of previous cross layer (or \(x_0\) if l=1)
\(w_l\):Weight vector for l-th cross layer (d-dimensional); learns linear projection of input \(x_l\) to scalar
\(b_l\):Bias term for l-th cross layer (scalar); adds offset to weighted sum \(w_l^T x_l\)
\(\odot\):Element-wise (Hadamard) multiplication; gates original input \(x_0\) by learned projection \((w_l^T x_l + b_l)\)
l:Cross layer index starting from 0; l-th layer produces 2^l-order feature interactions
\(d\):Feature dimension; same throughout network; parameter count per layer is \(O(d)\), not \(O(d^2)\)
vs Wide & Deep:Wide requires manual feature engineering (expensive); DCN learns feature crosses automatically
Cross layer output:l-th layer produces polynomial order \(2^l\) (Layer 1: 2nd order, Layer 2: 4th order, etc.); exponential expressivity with linear parameters
Architecture:Cross network (automatic feature crosses) + Deep network (embeddings + MLPs) in parallel; concatenate outputs for final classification layer
Serving cost:Minimal latency overhead (simple dot products and element-wise ops); ideal for production constraints
When to Use
DCN (Deep & Cross Network)
Use when you want automatic feature crossing without manual engineering. Classic use: Google Ads CTR prediction (hundreds of sparse features, too many crosses to hand-craft), Amazon product ranking with rich categorical features. Replaces Wide & Deep's manual wide path with learned crosses.
Manual Feature Crosses (Wide & Deep)
Use when you have strong domain knowledge about which feature interactions matter. Classic use: e-commerce where "brand x price_range" is a known strong signal. More interpretable but requires ongoing feature engineering as catalog evolves.
👉 Try this: Increase "Cross Layers" from 1 to 5. Watch Cross Network learn feature interactions. Layer 1 learns pairwise crosses (age × gender). Layer 2 learns order-3 crosses. Layer 4+ very high-order. Increase too much and you overfit. Slide "DCN Strength" to see regularization effect. The cross network is remarkably sample-efficient—beats traditional feature engineering with 10x fewer parameters.
Why more efficient than hand-crafted crosses? Exponential feature explosion: to get 4th-order interactions manually, you need \(O(d^4)\) features. DCN achieves it with \(O(4d)\) parameters via automatic layer-wise composition.
Interpretability trade-off? Unlike Wide & Deep (you can inspect wide weights to see which crosses matter), DCN cross layers are learned implicitly—harder to debug but more automated.
Think of cross layers as a learnable feature interaction funnel: each layer multiplies original input by learned gates to extract higher-order patterns progressively.
import torch.nn as nn

class CrossLayer(nn.Module):
    def __init__(self, dim):
        super().__init__()
        self.w = nn.Linear(dim, dim, bias=True)

    def forward(self, x0, x):
        return x0 * self.w(x) + x  # element-wise cross

class DCN(nn.Module):
    def __init__(self, input_dim, n_cross=3, hidden=128):
        super().__init__()
        self.cross_layers = nn.ModuleList([CrossLayer(input_dim) for _ in range(n_cross)])
        self.deep = nn.Sequential(nn.Linear(input_dim, hidden), nn.ReLU(),
                                  nn.Linear(hidden, hidden), nn.ReLU())
        self.out = nn.Linear(input_dim + hidden, 1)

    def forward(self, x):
        xc = x
        for cl in self.cross_layers:
            xc = cl(x, xc)
        return torch.sigmoid(self.out(torch.cat([xc, self.deep(x)], dim=-1)))

Cross layers learn explicit feature interactions without manual engineering. Stacked cross layers capture higher-order crosses.

Attention Weights: --
Key Insight: Attention computes dynamic relevance weights for historical items, capturing user's multi-faceted interests as weighted aggregation.
$$I = \sum_i \alpha_i \cdot \text{embed}(\text{item}_i), \text{ where } \alpha_i = \text{softmax}(\text{score}(\text{item}_i, \text{candidate}))$$
\(I\):Final user interest representation (d-dimensional); weighted average of historical item embeddings relevant to candidate
\(\sum_i\):Sum over all i historical items in user's sequence (i = 1..sequence_length)
\(\alpha_i\):Attention weight for item i; \(\alpha_i \in [0,1]\) and \(\sum \alpha_i = 1\); higher if item i is relevant to candidate
\(\text{embed}(\text{item}_i)\):Embedding vector (d-dimensional) of historical item i; learned during training
\(\text{score}(\text{item}_i, \text{candidate})\):Raw relevance score (before softmax); typically dot product or learned bilinear similarity between historical and candidate embeddings
\(\text{softmax}(\cdot)\):Converts scores to normalized weights; ensures \(\alpha_i \geq 0\) and \(\sum \alpha_i = 1\); emphasizes high-scoring items exponentially
candidate:Query item being ranked; all attention weights computed relative to this single candidate
Scoring function:Typically dot product or learned bilinear similarity between historical item and candidate embeddings
Dynamic weighting:Candidate item "pulls" relevant historical items to the forefront; users with diverse interests (action + documentaries) are naturally handled
vs Average pooling:Average treats all history equally (loses context); attention weights history adaptively based on candidate
Computational cost:O(sequence_length) per candidate; can be prohibitive if sequences are long (100+ items)
Extensions:Multi-head attention (k separate relevance functions); Transformers (parallel all-pairs attention); attention-over-attention (hierarchical)
When to Use
DIN (Attention-Based)
Use when users have diverse, multi-interest behavior histories and you need candidate-aware user representations. Classic use: Alibaba product recommendations (user browses electronics AND clothing -- attention activates relevant history per candidate), Amazon "customers who viewed this also viewed".
Average Pooling (No Attention)
Use when user histories are short (under 10 items) or interests are homogeneous. Classic use: niche e-commerce (single-category store), early-stage startups with limited history data. Simpler, faster, and sufficient when attention overhead is not justified.
👉 Try this: Click "Show User History" to see past items. Select a query item. Watch attention weights light up—which past items influence this prediction? Toggle "Temporal Weighting". With temporal weighting ON, recent items dominate (1 week ago = 10x weight of 1 month ago). Toggle OFF, uniform weighting. Notice predictions change: temporal is more accurate for short-term trends.
Why better than average? Example: User watched [Romance, Romance, Romance, Action]. Average gives little weight to Action preference. Attention on Action candidate upweights the Action history item.
Recency bias emerges naturally: if items are ordered temporally, softmax attention over recent items (higher scores) gives more weight to recent history without explicit time decay.
Attention = Adaptive history summarization: different candidates pull different "summaries" from the same user history based on relevance.
import torch, torch.nn as nn

class DINAttention(nn.Module):
    def __init__(self, emb_dim):
        super().__init__()
        self.mlp = nn.Sequential(
            nn.Linear(emb_dim * 4, 64), nn.ReLU(),
            nn.Linear(64, 1))

    def forward(self, query, keys):  # query: (B, D), keys: (B, T, D)
        q = query.unsqueeze(1).expand_as(keys)
        cross = torch.cat([q, keys, q - keys, q * keys], dim=-1)
        weights = self.mlp(cross).squeeze(-1)  # (B, T)
        weights = torch.softmax(weights, dim=-1)
        return (weights.unsqueeze(-1) * keys).sum(1)  # (B, D)

Computes attention between candidate item and user history. Captures which past behaviors are relevant to the current item.

Key Insight: Multi-stage funnel trades off latency (500ms end-to-end) against accuracy via recall→precision→business-logic progression.
Stage 1: Candidate Gen (50-200ms)
Fast, approximate methods (ANN, collaborative filtering, popularity); optimize recall—do not miss relevant items
Stage 2: Ranking (100-300ms)
Expensive models (DIN, Wide&Deep, DCN) score candidates; optimize precision—fine-grained relevance
Stage 3: Re-ranking (50-100ms):Business logic—diversity filters, freshness boost, ad slots, CTR thresholds; final ranking before returning
Latency budget:Each stage has fixed time slice; if Stage 1 uses 150ms, Stages 2-3 share 350ms—strict SLA enforcement
Fundamental trade-off:More stages enable accuracy improvements but consume latency; fewer stages are faster but may discard relevant items too early
Variations:Batch pre-compute scores offline; use LambdaMART to optimize ranking metrics directly; different models for head vs tail items
When to Use
Multi-Stage Pipeline
Use for any production system with large catalog (100K+ items) and strict latency SLA (under 500ms). Classic use: YouTube (100M videos, 3-stage funnel), Netflix homepage (10K titles, 2-stage), Spotify Discover Weekly (70M tracks). Essential when single-model scoring of full catalog is infeasible.
Single-Stage Ranking
Use when catalog is small (under 10K items) and latency budget is generous. Classic use: small e-commerce stores, curated content feeds, enterprise internal search. Simpler to debug and maintain; no recall loss from candidate generation filtering.

Sequential Recommendations: SASRec (Self-Attentive Sequential Recommendation)

Click 'Play Sequence' to step through a user's actions. Watch self-attention weights highlight which past actions most influence the next prediction. Notice: recent items weight heavily, but the model also learns to weight semantically related items from anywhere in history. Toggle 'Attention Mask' to show attention pattern as a heatmap.

10
Predicted Next Item: -- | Confidence: --
Key Insight: Transformers via self-attention capture long-range item dependencies without sequential bottleneck; each item attends to all prior items in sequence.
$$\text{Attention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^T}{\color{#ff9ff3}{\sqrt{d}}}\right)V$$
\(Q\) (Query):Query matrix (sequence_length × d); projections of input embeddings; represents "what to attend to"
\(K\) (Key):Key matrix (sequence_length × d); projections of input embeddings; represents "what can be attended"
\(V\) (Value):Value matrix (sequence_length × d); projections of input embeddings; represents "information to aggregate"
\(QK^T\):Matrix multiplication (sequence_length × sequence_length); each element [i,j] = dot product of query i and key j
\(d\):Embedding dimension; typically 64 per head in multi-head attention; impacts gradient scale
\(\sqrt{d}\) (scaling factor):Divides attention scores by \(\sqrt{d}\); stabilizes gradients (prevents very large/small dot products); ensures softmax doesn't saturate
softmax(·):Converts scaled scores to normalized attention weights; each row sums to 1; emphasizes high-scoring positions
softmax(QK^T/√d)V:Final output (sequence_length × d); weighted sum of values by attention weights; each position attends to all previous positions
Mechanism:Each position i attends to all previous positions [1..i-1] (causal masking prevents future leakage); item representation = weighted sum of all prior items
vs RNNs/LSTMs:Parallel training (no sequential bottleneck), better gradient flow (no vanishing gradients), scales to longer sequences (100+ items)
Learned patterns:Item transitions (shoe → sock → laces); recency bias (softmax places higher weight on recent items) emerges naturally
Engineering details:Position embeddings (absolute or relative) encode temporal order; layer-normalization stabilizes; multi-head attention learns k separate relevance functions
When to Use
SASRec / Transformer-Based Sequential
Use when temporal order of user actions matters and histories are long (50+ items). Classic use: Spotify playlist continuation (next song depends on sequence context), Amazon "frequently bought together" sequences, TikTok next-video prediction. Captures long-range dependencies that RNNs miss.
GRU4Rec / RNN-Based Sequential
Use when sequences are short (under 20 items) and training resources are limited. Classic use: session-based recommendations on e-commerce (anonymous users with short browsing sessions), news article recommendations. Lighter weight, faster training, sufficient for short sequences.
👉 Try this: Click "Play Sequence" and step through user actions. Watch self-attention heatmap highlight which past actions most influence the next prediction. Toggle "Attention Mask": with mask, can't look at future actions (causal). Without mask, can (violates time order). Increase sequence length to 50 items. Transformers scale gracefully; RNNs would forget early items. This is why Transformers dominate sequence modeling.
Variable-length sequences: Pad to max length with special tokens, use attention mask to ignore padding. Alternatively, truncate to recent k items (e.g., last 50 items).
Recency bias: Emerges naturally from attention weights (recent items higher QK scores); no explicit time decay needed. Validate via attention weight visualization.
Self-attention = parallel RNN: each position "reads" previous positions simultaneously via learned attention weights, avoiding sequential computation.
import torch.nn as nn

class SASRec(nn.Module):
    def __init__(self, n_items, d_model=64, max_len=50, n_heads=2, n_layers=2):
        super().__init__()
        self.item_emb = nn.Embedding(n_items + 1, d_model, padding_idx=0)
        self.pos_emb = nn.Embedding(max_len, d_model)
        encoder_layer = nn.TransformerEncoderLayer(d_model, n_heads, d_model * 4,
                                                    dropout=0.1, batch_first=True)
        self.encoder = nn.TransformerEncoder(encoder_layer, n_layers)

    def forward(self, seq):  # seq: (B, T) item IDs
        mask = torch.triu(torch.ones(seq.size(1), seq.size(1)), diagonal=1).bool()
        x = self.item_emb(seq) + self.pos_emb(torch.arange(seq.size(1)))
        return self.encoder(x, mask=mask)  # (B, T, D)

Applies causal self-attention over item sequences. The last hidden state predicts the next item.

Key Insight: Multi-Task Learning optimizes correlated objectives (CTR, CVR, watch-time) jointly; Multi-gate Mixture of Experts (MMOE) allows task-specific expert selection via gating.
$$\text{Task}_k\text{ output} = \text{gate}_k \otimes \sum_i \color{#ff9ff3}{\text{expert}_i}, \text{ where gate}_k = \text{softmax}(W_k \cdot \text{features})$$
\(\text{Task}_k\) output:Final prediction for task k (CTR, CVR, watch-time, etc.); vector of shape (batch_size, output_dim)
\(\text{gate}_k\):Gating vector for task k (m-dimensional); softmax weights over m experts; emphasizes task-relevant expert representations
\(\sum_i \text{expert}_i\):Sum of all expert outputs; each expert i is a small MLP producing (batch_size, output_dim) representation
m:Number of shared experts; typical range 4-10 in production; each expert is a small MLP (2-3 hidden layers)
\(\otimes\):Broadcasting multiplication; \(\text{gate}_k\) (m-dimensional) multiplies across experts dimension of \(\sum_i \text{expert}_i\)
W_k:Weight matrix for task k's gating network; shapes (feature_dim, m); learns task-specific expert preferences
features:Input feature vector (user context, item features, etc.); shared across all experts and gating networks
softmax(W_k · features):Normalized weights over m experts; ensures gate_k sums to 1 and all weights in [0,1]
k:Task index; separate gating networks for each task (CTR, CVR, etc.); experts shared across all tasks
Objectives:CTR (click), CVR (conversion), watch time, engagement—all correlated; joint training improves data efficiency and calibration
MoE architecture:m shared experts (small MLPs, e.g., m=6); per-task gating networks learn softmax weights over experts; task output = weighted sum
Benefits:Parameter efficiency (shared experts << task-specific towers); mutual information between tasks improves generalization; flexible task importance weighting
Joint loss:\(L = \alpha \cdot L_{\text{ctr}} + \beta \cdot L_{\text{cvr}} + \gamma \cdot L_{\text{watch}}\), where \(\alpha, \beta, \gamma\) reflect business priorities (tune on validation set)
Production gotchas:Task calibration (different loss scales → gradient dominance); label delay (CVR takes days, CTR immediate); negative transfer (tasks conflict)
When to Use
MMOE Multi-Task
Use when optimizing multiple correlated but distinct objectives simultaneously. Classic use: YouTube ranking (optimize CTR + watch time + satisfaction jointly), Uber Eats (optimize order rate + delivery time + restaurant quality). Shared experts improve data efficiency when tasks share underlying patterns.
Separate Single-Task Models
Use when tasks are weakly correlated or conflicting (negative transfer risk). Classic use: when CTR optimization hurts long-term retention, or when ad revenue goals conflict with user satisfaction. Easier to debug, tune, and deploy independently per objective.

Handle label imbalance: CVR is ~1% (rarer than CTR ~10%). Use weighted sampling or loss weighting to balance task gradients; monitor which task dominates training.
Validate improvements: Compare multi-task vs CTR-only baseline. Measure both task accuracy (does each task improve?) and correlation (do tasks help each other?).
MoE gating = skill specialization: CTR gate prefers experts good at short-term clicks; CVR gate prefers experts predicting long-term conversions.
import torch.nn as nn

class MMOE(nn.Module):
    def __init__(self, input_dim, n_experts=4, expert_dim=64, n_tasks=2):
        super().__init__()
        self.experts = nn.ModuleList(
            [nn.Sequential(nn.Linear(input_dim, expert_dim), nn.ReLU()) for _ in range(n_experts)])
        self.gates = nn.ModuleList(
            [nn.Linear(input_dim, n_experts) for _ in range(n_tasks)])
        self.towers = nn.ModuleList(
            [nn.Linear(expert_dim, 1) for _ in range(n_tasks)])

    def forward(self, x):
        expert_outs = torch.stack([e(x) for e in self.experts], dim=1)  # (B, E, D)
        outputs = []
        for gate, tower in zip(self.gates, self.towers):
            g = torch.softmax(gate(x), dim=-1).unsqueeze(-1)  # (B, E, 1)
            mixed = (expert_outs * g).sum(1)  # (B, D)
            outputs.append(tower(mixed))
        return outputs

Each task has its own gating network to softly select from shared experts. Balances task-specific vs shared learning.

Learning to Rank (LTR): Loss Function Comparison

Click the three tabs to see each loss function. Pointwise (simplest): predict score for each doc independently. Pairwise: learns ranking ('doc A > doc B'), more complex. Listwise (most complex): optimizes entire ranking order. Compare loss curves—listwise is harder to optimize but produces better rankings when it converges.

10
Mode: Pairwise | Loss: --
Key Insight: Learning to Rank (LTR) optimizes ranking metrics (NDCG, MRR) directly; pointwise ignores ranking structure; pairwise captures pairs; listwise optimizes full ranking.
Pointwise
Treat each document independently; loss = classification_loss. Simple but ignores ranking structure and position bias.
Pairwise
Minimize inversion loss for pairs: \(\log(1 + \exp(-\sigma(s_i - s_j)))\). Captures ranking structure; \(O(n^2)\) pairs per example.
Listwise:Direct NDCG optimization: loss = -NDCG(ranked_list). Most principled but requires sorting (expensive); LambdaMART combines with pairwise for efficiency
LambdaMART:Computes pairwise gradients but weights by NDCG improvement (lambda gradient weighting); efficient listwise optimization
Accuracy vs Computation:Pointwise (fast) << Pairwise (medium) << Listwise (slow but best); production typically uses LambdaMART via XGBoost/LightGBM
When to Use
Pairwise / LambdaMART
Use for production ranking where NDCG optimization matters and you have graded relevance labels. Classic use: Google Search ranking, Bing web search, Airbnb search ranking. LambdaMART (via XGBoost/LightGBM) is the industry standard -- fast training, handles missing features, excellent NDCG.
Pointwise (Classification/Regression)
Use as a simple baseline or when you only have binary labels (relevant/not). Classic use: email spam ranking, simple content feeds with thumbs up/down. Fast to train and debug, but ignores position bias and ranking structure -- upgrade to pairwise when quality matters.
👉 Try this: Toggle "Pointwise", "Pairwise", "Listwise" loss functions. Pointwise (simplest): treats each doc independently, ignores ranking. Pairwise: learns 'doc A > doc B'. Listwise: directly optimizes NDCG (ranking metric). Compare convergence: Pointwise converges fast but gives mediocre ranking. Listwise is harder to optimize but produces best final ranking. Click "Show Metrics" to see NDCG, MRR, MAP differences.
Why does pointwise fail on ranked outputs? Position bias: top-ranked docs receive more clicks regardless of relevance. Evaluation metric mismatch: classification loss doesn't align with NDCG/MRR metrics.
Variable-length lists: Pad with negative labels (rank 0) to uniform length. LambdaMART handles missing labels gracefully (skips pairs with missing labels).
Pointwise = per-item scores ignoring context; Pairwise = respecting relative order; Listwise = optimizing the entire ranked order for metric.
import torch
import torch.nn.functional as F

def listwise_softmax_loss(scores, relevance):
    """ListNet-style listwise ranking loss."""
    pred_dist = F.log_softmax(scores, dim=-1)
    true_dist = F.softmax(relevance.float(), dim=-1)
    return F.kl_div(pred_dist, true_dist, reduction='batchmean')

import torch.nn as nn

class CTRModel(nn.Module):
    """Click-Through Rate prediction for ad ranking."""
    def __init__(self, sparse_dims, dense_dim, emb_dim=16):
        super().__init__()
        self.embeddings = nn.ModuleList([nn.Embedding(d, emb_dim) for d in sparse_dims])
        total_dim = len(sparse_dims) * emb_dim + dense_dim
        self.mlp = nn.Sequential(nn.Linear(total_dim, 128), nn.ReLU(),
                                 nn.Linear(128, 64), nn.ReLU(), nn.Linear(64, 1))

    def forward(self, sparse_feats, dense_feats):
        embs = [e(sparse_feats[:, i]) for i, e in enumerate(self.embeddings)]
        x = torch.cat(embs + [dense_feats], dim=-1)
        return torch.sigmoid(self.mlp(x))  # P(click)

Sparse features (user/ad IDs) get embeddings; dense features are concatenated. Output is click probability for ad auction ranking.

Bi-encoder: 1ms | Cross-encoder: 500ms | ColBERT: 50ms
Key Insight: Dense retrieval encodes queries/documents as embeddings; bi-encoder is fast but coarse; cross-encoder is accurate but slow; ColBERT bridges both.
Bi-encoder
query_emb = BERT(query), doc_emb = BERT(doc); score = cosine(q, d). Pre-index all docs in FAISS/Annoy; O(1) serving cost.
Cross-encoder
score = BERT([query, doc]); direct relevance. O(n) serving cost (re-encode each candidate). Accurate but expensive.
ColBERT (late interaction):Per-token embeddings; score = max_{i,j} cos(q_i, d_j); defers fine-grained matching to serving without full re-encoding

Production hybrid:BM25 (cheap retrieval) → bi-encoder re-rank (efficient) → cross-encoder/ColBERT refine top-k (accurate); multi-signal fusion
Cold-start new docs:Compute embeddings from content (metadata, title, body); no user interaction history required; fine-tune as signals accumulate
When to Use
Bi-Encoder + ANN Index
Use for first-stage retrieval over massive corpora (millions of documents). Classic use: Google Search candidate retrieval, Spotify podcast search, LinkedIn people search. Pre-compute document embeddings in FAISS/ScaNN for sub-millisecond retrieval at scale.
Cross-Encoder / ColBERT Re-ranking
Use for re-ranking top-k candidates (50-200) where precision matters more than speed. Classic use: Google Search final ranking of top-100 results, Bing passage ranking, legal document search (accuracy critical). ColBERT offers the best speed-accuracy tradeoff for mid-size candidate sets.

Generative Recommenders: Next-Token Prediction for Recs

Traditional rec systems use a 3-stage pipeline (retrieve→rank→rerank). Generative recommenders replace this with a single LLM that generates the next item directly — just like GPT generates the next word. Click through the steps to see how items become tokens and how the model generates recommendations.

Traditional: 3 models, 10B params in embeddings, ~500ms | Generative: 1 model, 0 embedding tables, ~200ms
Key Insight: Generative recommenders reframe recommendation as sequence modeling — predicting the next "item token" just like LLMs predict the next word token.
Semantic IDs (SID):Items are compressed into short token sequences using RQ-VAE (Residual Quantized VAE) — a neural network that learns hierarchical codebooks to assign each item a "zip code." Level 1 captures broad category (genre), Level 2 captures sub-category, Level 3 identifies the specific item. Similar items share token prefixes, enabling generalization to unseen items. YouTube's SID-v2 uses codebooks of decreasing size (2048→1024 entries).
No embedding tables:Traditional models need O(catalog_size) embedding parameters. PLUM allocates 90% of parameters to neural networks vs 0.4% in traditional models. This enables better generalization.
Cold start solved:New items get Semantic IDs from their content (video/audio/text) immediately — no need for interaction history.
Scaling laws apply:Like LLMs, generative rec performance scales predictably with model size and compute (power-law).
PLUM results:+4.96% CTR on YouTube Shorts, 13x vocabulary coverage vs traditional models, first neural model without embedding tables in YouTube production.
OneRec results:Replaced entire Kuaishou pipeline, 0.54% engagement gain, 90% cost reduction, 10x FLOPs improvement.
Traditional Pipeline
3 separate models (retrieval, ranking, reranking), each optimized separately. Embedding tables scale with catalog (10M items = huge table). Slow to iterate, cold-start problem.
Generative Recommender
Single LLM generates item tokens directly. No embedding tables. Content-based Semantic IDs solve cold start. Scaling laws enable predictable improvement.
Interview tip: "Generative recommenders unify retrieve+rank into one model by treating items as token sequences. The key innovation is Semantic IDs — RQ-VAE compresses items into discrete tokens the LLM can generate. This is the 'ChatGPT moment' for RecSys."
Real-world tip: YouTube's PLUM is in production serving both Shorts and long-form video. Kuaishou's OneRec serves hundreds of millions of daily users. This is not theoretical — it's the direction the industry is moving.
Traditional = "search then score" (embedding lookup → dot product → rerank). Generative = "just say the answer" (generate item tokens directly, like completing a sentence).
